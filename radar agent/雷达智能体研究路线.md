# 雷达智能体研究路线

## 1. 雷达智能体研究目标与设计定位

雷达智能体研究的核心目标，是在现有雷达感知、信息处理、显控交互和控制执行体系之上，构建一个面向雷达任务的智能推理与协同决策能力，使雷达系统从“感知结果展示 + 人工经验配置”的工作方式，逐步升级为“任务驱动、证据支撑、智能推荐、受控执行、持续演进”的认知增强模式。该研究以雷达任务闭环为牵引，把智能体工程方法与认知雷达的感知-行动循环结合起来，形成能够服务当前环境目标研判、智能调度、波形参数设置和经验沉淀的工程闭环。

从雷达领域看，认知雷达强调基于环境反馈、目标状态和历史经验进行自适应调整，其关键在于接收端感知、状态估计、发射端动作选择和全局反馈之间的闭环。雷达智能体的研究应继承这一思想，让大模型承担任务理解、证据组织、方案生成、风险解释和复盘抽取等认知环节；参数仿真、边界校验、人工确认、控制请求适配和回滚恢复等工程动作，则通过工具和工作流进入受控执行链路。这样既能利用大模型在语义理解、知识整合、复杂任务规划方面的优势，又能保持雷达系统对实时性、安全性、可解释性和可追溯性的要求。

从智能体工程看，成熟的 Agent 落地实践已经从“让模型自主完成一切”转向“模型、上下文、工具、工作流、记忆、评估和观测共同构成系统”。雷达智能体应采用简单、可组合、可治理的智能体架构：用 Agent Harness 承载任务生命周期和运行状态，用上下文工程保证模型在正确证据范围内推理，用雷达 Skills 固化领域能力和输出格式，用工具调用连接确定性计算和专业能力，用工作流管理高风险流程，用三阶分层记忆沉淀任务上下文、任务链和长期经验，用追溯与评估机制持续发现问题并改进能力。其研究重点是把模型能力纳入雷达工程任务闭环，形成可运行、可评估、可持续优化的智能体系统。

从系统边界看，雷达智能体属于雷达内部的智能推理与协同子系统，服务对象主要是显控分系统、信息处理感知子系统、雷达状态数据、数据知识底座和控制执行链路。信息处理感知子系统负责提供目标检测、识别、跟踪、定位、置信度和异常评分等感知结果；控制执行相关分系统负责完成真实控制指令下发和执行反馈；雷达智能体重点完成任务建模、上下文组织、研判推理、候选方案生成、流程触发、结果解释、追溯记录和经验回流。该定位使智能体能够进入雷达核心业务，同时保持智能推理链路与工程控制链路的清晰衔接。

## 2. 雷达智能体关键能力体系

雷达智能体的关键能力体系应围绕“任务如何进入智能体、证据如何支撑推理、模型如何形成方案、方案如何进入工程链路、经验如何持续演进”展开。这里按照智能体落地必须具备的能力链条组织，每一类能力都对应一组研究问题、工程抓手和可验收成果。

为便于从雷达工程视角理解智能体设计，需要先明确几个基础概念。Agent Harness 可以理解为智能体的“运行底座”和“任务容器”，负责把一次用户请求或一次系统事件变成可跟踪、可恢复、可审计的任务实例，并管理任务状态、上下文、模型调用、工具调用、工作流触发和结果发布。上下文工程是指为每次推理构造合适的信息环境，决定哪些感知结果、雷达状态、历史经验、专业知识和约束条件进入模型输入，哪些内容作为证据引用，哪些内容因为过期、冲突或低置信而需要降权。Skills 是面向领域任务沉淀下来的可复用能力单元，包含任务提示、知识引用规则、工具选择规则、输出结构和质量检查要求。工具调用是智能体连接外部确定性能力的方式，负责查询、计算、仿真、检索、报告生成等动作。工作流门控则用于高风险或标准化流程，把参数设置、人工确认、控制请求适配和回滚恢复纳入可控步骤。三阶分层记忆用于把当前任务上下文、近期任务链和长期经验资产分层管理，使智能体既能完成当前任务，又能在后续任务中受控复用经验。雷达智能体研究的关键，就是把这些智能体工程概念转化为雷达任务闭环中的可落地机制。

### 2.1 任务理解与 Agent Harness 运行组织能力

任务理解与运行组织能力解决雷达智能体的入口问题，即如何把显控指令、感知事件、状态告警和控制回传统一组织为可管理的智能体任务。雷达场景中的输入通常由自然语言、目标选择、区域框选、告警卡片、参数面板状态、感知结果事件和设备状态变化共同构成。智能体需要把这些输入绑定到明确的目标、区域、时间窗口和任务状态，使后续推理始终贴合雷达业务对象。

该能力需要以 Agent Harness 作为任务运行底座。Harness 不是某个单一算法，也不是简单把大模型和工具串起来的调用脚本，而是智能体系统的工程外壳。大模型本身只负责在给定上下文中进行语言理解、推理和生成；Harness 则负责回答“这次推理属于哪个任务、使用哪一版上下文、允许调用哪些能力、当前执行到哪一步、失败后如何恢复、结果如何发布、过程如何追溯”等工程问题。没有 Harness，智能体容易停留在一次性问答；有了 Harness，智能体才能成为能够连续运行、被系统调度、被人工监督、被事后复盘的工程子系统。

在雷达智能体中，Harness 需要承载任务实例、上下文版本、能力选择、工具调用、工作流触发、状态同步、异常处理、结果发布和追溯记录。每个任务实例应包含任务类型、触发来源、目标或区域引用、时间窗口、输入证据、风险等级、上下文版本、执行状态和追溯标识。对于多轮交互，Harness 需要维护“这个目标”“沿用上次方案”“继续刚才研判”等指代关系与真实任务对象之间的映射；对于事件触发任务，Harness 需要把感知事件、状态告警和人机交互纳入同一任务状态机。

从工程实现角度看，Harness 应至少提供四类运行能力。一是任务状态机，描述任务从创建、受理、上下文装载、推理规划、工具调用、工作流执行、结果发布到归档复盘的状态变化。二是运行上下文管理，确保每次模型调用、工具调用和工作流节点都绑定到同一任务编号和上下文版本。三是能力路由，依据任务类型、风险等级、证据完整性和实时性要求，选择只读推理、工具验证、工作流门控或人工确认路径。四是运行观测与追溯，记录模型输入输出、工具结果、工作流节点、异常处理和人工反馈，使智能体行为能够被调试、评估和审计。

该能力的研究重点包括三方面。第一是任务语义建模，即把自然语言意图、显控对象和系统事件转换为统一任务表示。第二是任务状态管理，即建立从任务受理、上下文装载、推理规划、工具执行、工作流流转、结果发布到归档复盘的状态流转机制。第三是风险分级路由，即根据任务是否只读、是否推荐参数、是否触发控制请求、是否需要人工确认，选择动态推理、工具验证或受控工作流路径。阶段成果应包括任务对象规范、Harness 状态流转规则、典型任务样例、连续对话对象绑定样例、任务风险分级规则和追溯编号规则。

### 2.2 上下文工程与雷达证据组织能力

上下文工程是雷达智能体可靠性的基础。智能体的上下文需要在每次任务中明确“模型应该基于哪些事实、哪些状态、哪些历史经验和哪些约束进行推理”。雷达领域尤其需要上下文工程，因为目标研判和参数推荐依赖的证据具有强时效性、强版本性和强条件性：同一个目标在不同时间窗口、不同工作模式、不同环境条件下，其研判依据和参数策略可能完全不同。

从通用智能体实践看，上下文工程的目标不是追求输入越多越好，而是追求输入“与当前任务强相关、来源清楚、时效明确、质量可判断、约束可执行”。对于雷达智能体，这一点尤其关键。雷达感知结果可能来自不同模型版本，雷达状态可能在短时间内变化，历史案例可能只适用于相似但不完全相同的环境，参数策略也可能受设备状态和任务优先级约束。上下文工程需要把这些信息整理成模型可用、人工可查、系统可回放的任务上下文。

该能力需要建立面向雷达任务的任务上下文。感知结果侧应组织目标检测、识别、跟踪、定位、置信度、异常评分、航迹质量和感知模型版本；雷达状态侧应组织当前工作模式、波形参数、波束资源占用、设备健康状态、告警状态、参数版本和控制回传；环境与知识侧应组织气象、电磁环境、历史案例、专业知识、参数策略、操作规程和复盘结论。系统在装载上下文时，需要区分观测事实、模型判断、工程约束、历史经验、推理结论和人工反馈，避免把经验性判断包装成当前事实，也避免把低置信感知结果作为确定依据。

该能力的工程抓手包括证据项、上下文版本、数据质量标识和证据链。证据项用于标识每条输入的来源、时间、版本、质量和适用范围；上下文版本用于保证一次任务的推理、工具调用、工作流节点和输出结论能够回放；数据质量标识用于提示缺失、冲突、过期、低置信和版本不一致等情况；证据链用于把目标研判、参数推荐和风险提示与支撑依据关联起来。对于目标研判场景，应形成目标类别、行为特征、航迹变化、异常程度、威胁等级、证据缺口和处置建议的证据模板；对于参数推荐场景，应形成任务目标、目标状态、资源约束、当前参数、历史效果和风险条件的上下文模板。阶段成果应包括上下文装载策略、任务上下文组织规范、证据项规范、证据链输出样例、数据质量标识规则和目标研判/参数推荐上下文模板。

### 2.3 雷达 Skills 与智能推理/参数推荐能力

智能推理与参数推荐能力是雷达智能体的认知核心，关键在于把大模型推理约束在雷达任务、雷达知识和雷达工程边界内。主流 Agent 实践表明，复杂任务需要把推理、行动和观察结合起来；在雷达场景中，这种 ReAct 式动态推理需要由领域 Skills、结构化输出和风险门控共同约束。雷达 Skills 应沉淀目标研判、异常解释、参数推荐、任务复盘、报告生成等可复用能力，明确每类任务使用的知识来源、工具选择、输出结构、风险表达和降级策略。

这里的 Skills 不是传统意义上的软件模块，也不是简单的提示词集合，而是“领域任务能力包”。一个成熟的雷达 Skill 应当说明它面向什么任务、需要哪些输入证据、可以调用哪些工具、输出什么结构化结果、如何表达置信度和不确定性、遇到证据不足时如何降级、哪些情况必须进入工作流或人工确认。例如，目标研判 Skill 应规定如何引用感知结果、历史案例和专业知识；波形参数推荐 Skill 应规定如何组织任务目标、目标状态、资源约束、当前参数和历史效果；复盘生成 Skill 应规定如何从追溯记录中抽取任务过程、效果指标和经验候选。

ReAct 式推理可以理解为“边推理、边调用工具、再根据工具反馈继续推理”的循环。它适合开放式研判，因为目标异常解释、证据补充、历史案例查询往往不能一次完成；模型需要先判断缺什么证据，再调用查询或分析工具，拿到结果后修正判断。但雷达场景中的 ReAct 不能放任模型任意行动，而要由 Harness、Skills 和工具门控共同约束：模型可以决定需要补充哪类证据，可以提出候选参数方案，可以请求仿真评估；涉及参数设置、控制请求和回滚恢复时，则必须转入工作流。

在当前环境目标研判方面，智能体应围绕目标状态、航迹变化、感知置信度、异常评分、环境条件和历史案例进行多证据综合分析，形成“观测事实-证据解释-推理判断-不确定性说明-建议动作”的结构化结果。系统应能够识别证据不足、感知结果冲突、目标状态突变、环境条件异常和模型版本变化等情况，并通过补证请求、备用工具、人工复核或降低置信度的方式保持结论表达的审慎性。

在智能调度和波形参数设置方面，智能体应围绕任务目标、目标状态、雷达资源、环境条件、当前工作模式和历史执行效果生成候选方案。认知雷达和近期资源管理研究都表明，雷达调度与波形选择本质上是多目标、受约束、反馈驱动的决策问题，需要在扫描与跟踪、精度与资源、性能与风险、当前收益与后续稳定性之间进行权衡。因此，智能体应输出完整候选方案包，包括推荐参数、推荐依据、适用条件、预期效果、约束检查、风险点、验证方式和回滚条件。阶段成果应包括雷达 Skills 能力包设计、研判结果结构模板、候选参数方案模板、参数推荐约束清单、输出质量检查规则、风险说明模板和典型场景推理样例。

### 2.4 工具调用、工作流门控与受控执行能力

工具调用与受控流程能力解决智能体如何进入工程系统的问题。业界实践已经形成一个清晰趋势：让模型负责理解、规划、解释和候选方案生成，让确定性工具负责计算、查询、仿真和校验，让工作流负责高风险流程、人工确认、检查点和回滚。雷达智能体尤其需要这种分工，因为参数设置、工作模式切换和控制请求适配都具有工程风险，需要通过受控工作流完成工程动作衔接。

工具调用是智能体工程中的“行动接口”。模型本身不应承担数据库查询、数值仿真、状态校验、参数边界判断等确定性工作，而应通过标准化工具获得结果。工具的价值在于把外部系统能力变成智能体可调用、可校验、可记录的动作。一个合格的工具应明确输入参数、输出结构、权限要求、超时策略、异常返回、证据编号和追溯引用。这样，模型调用工具后拿到的不是一段不可控文本，而是可以继续推理、可以展示、可以进入追溯记录的结构化结果。

工作流门控则是面向高风险流程的“工程保险机制”。它把参数调整、人工确认、控制请求适配、状态跟踪和异常回滚拆成明确节点，每个节点都有输入、输出、检查条件和状态记录。对于雷达智能体而言，工作流的作用不是削弱智能性，而是把智能体生成的候选方案转化为可审查、可确认、可回退的工程过程。这样，模型的开放式推理能力与雷达系统的确定性控制要求可以共存。

该能力需要将雷达专业能力划分为四类工具。第一类是只读分析工具，包括数据查询、目标分析、状态解释、知识检索和报告生成，主要服务目标研判和证据补充。第二类是仿真评估工具，包括参数效果估计、资源占用检查、历史效果对比和风险条件检查，主要服务候选方案验证。第三类是流程触发工具，用于把候选方案提交到工作流，进入参数边界校验、人工确认、检查点记录和状态反馈。第四类是控制请求适配工具，用于在受控流程中承接经审批的方案，与控制执行链路衔接。

受控流程应成为雷达智能体落地的核心机制。对于只读研判任务，系统可采用推理-行动-观察式工具调用，根据证据缺口动态查询和分析；对于参数推荐任务，系统需要通过仿真、历史效果和约束检查增强候选方案可信度；对于参数设置、工作模式切换和异常回滚任务，系统必须进入工作流，完成参数边界校验、仿真评估、人工确认、请求适配、状态跟踪和回滚准备。Agent Harness 应统一管理工具权限、参数边界、调用超时、失败重试、异常降级、审计记录和追溯引用。阶段成果应包括工具能力清单、工具调用结果规范、工具输入/输出校验规则、受控工作流样例、风险分级规则、参数调整闭环演示流程和异常回滚样例。

### 2.5 三阶分层记忆、追溯评估与持续演进能力

雷达智能体要从一次性辅助工具发展为长期可用的工程系统，必须把三阶分层记忆作为关键技术之一，并与追溯、评估和经验治理机制联合设计。三阶分层记忆不是简单的历史记录库，而是智能体持续运行所需的上下文管理和经验演进机制：短期记忆服务当前任务，中期记忆服务近期连续任务链，长期记忆服务经过评估的知识、案例、策略和经验资产。最新 Agent 评估实践强调，智能体质量需要同时考察最终输出、任务轨迹、工具调用、门控结果、证据引用、异常处理和输出质量。对于雷达智能体，追溯记录不仅服务审计，更是后续复盘、经验抽取、能力评估和策略优化的基础。

追溯可以理解为智能体的“飞行记录仪”。它不仅记录最终回答，还记录任务是如何被理解的、装载了哪些上下文、模型生成了什么中间结论、调用了哪些工具、工具返回了什么、工作流经过了哪些节点、人工做了哪些确认、最后输出依据是什么。没有追溯，智能体出错后只能看到结果，难以判断问题来自感知数据、上下文选择、模型推理、工具调用还是工作流节点；有了追溯，系统才能进行定位、复盘、评估和改进。

评估是智能体持续演进的质量控制机制。传统软件测试通常验证固定输入是否产生固定输出，而智能体系统具有一定生成性和不确定性，因此需要建立样例集、评分准则和轨迹级评估。雷达智能体的评估不能只看回答是否通顺，而要看目标研判是否有证据支撑、参数推荐是否满足约束、风险说明是否完整、工具调用是否必要、工作流是否合规、经验引用是否适用。评估结果反过来用于修正 Skills、优化上下文装载策略、补充工具能力和完善工作流门控。

该能力需要建立贯穿任务全过程的追溯对象。追溯记录应覆盖任务输入、上下文版本、证据项、模型输出、工具调用、工作流节点、审批动作、参数候选方案、控制请求适配结果、操作员反馈和执行效果。基于这些记录，系统可以开展任务级评估、轨迹级评估和经验级评估。任务级评估关注目标研判是否准确、参数推荐是否合理、风险表达是否充分；轨迹级评估关注工具选择是否正确、工作流是否合规、异常处理是否有效；经验级评估关注某次任务沉淀的经验是否可复用、适用条件是否清楚、是否存在与当前状态冲突的风险。

三阶分层记忆应采用短期记忆、中期任务链和长期经验资产的分层机制。短期记忆保存当前任务上下文、工具结果、工作流状态和待确认事项，支撑任务内连续推理；中期记忆维护近期目标、区域、参数方案、状态变化和操作反馈之间的关联，支撑连续跟踪和跨任务复盘；长期记忆保存经过评估的知识、案例、策略和经验资产，为后续任务提供检索增强和经验参考。长期经验应经过“记录-评估-抽取-确认-复用”的治理流程，再进入可复用资产。近期 memory-agent 研究也表明，智能体记忆能力需要关注准确检索、交互中学习、长程理解和选择性遗忘；这对雷达智能体尤其重要，因为过期经验、错误经验和不适用经验都可能影响参数推荐质量。阶段成果应包括追溯记录规范、评估样例集、三阶分层记忆对象定义、经验候选抽取规则、复盘评估样例、长期经验资产样例和经验受控复用机制。

上述五类能力共同构成雷达智能体的研究主线。任务理解与 Agent Harness 运行组织能力解决“任务如何进入智能体”；上下文工程与证据组织能力解决“模型基于什么推理”；雷达 Skills 与智能推理能力解决“智能体如何形成专业结论和候选方案”；工具调用、工作流门控与受控执行能力解决“候选方案如何进入工程链路”；三阶分层记忆、追溯评估与持续演进能力解决“系统如何积累经验、被验证、被复盘和持续改进”。这一路线把智能体最佳实践与认知雷达的反馈闭环统一起来，使雷达智能体成为可运行、可解释、可治理的工程系统。

## 3. 雷达智能体技术路线与阶段成果

雷达智能体技术路线以“双闭环”为主线展开：内层是 Agent 任务闭环，外层是认知雷达业务闭环。Agent 任务闭环包括任务触发、Agent Harness 任务实例化、上下文工程装载、LLM 与雷达 Skills 推理、工具/工作流受控调用、结构化结果发布、追溯记录、评估反馈和记忆回流。认知雷达业务闭环包括感知结果接入、态势理解、候选调度/参数方案生成、仿真与边界校验、人工确认、控制请求适配、执行反馈和效果复盘。两条闭环通过任务实例、证据链、候选方案、工作流状态和追溯记录进行衔接。

所谓“双闭环”，是把智能体自身的运行闭环和雷达业务的认知闭环对应起来。Agent 任务闭环关心一次智能体任务如何被接收、推理、执行、记录和改进；认知雷达业务闭环关心雷达如何根据环境、目标和执行反馈调整工作方式。两者的连接点是任务实例和候选方案：智能体把雷达感知和状态输入组织成任务实例，经过上下文工程和 Skills 推理生成研判结论与候选方案，再通过工具和工作流完成验证、确认和执行衔接，最后把执行效果和复盘结果写回记忆体系。这使雷达智能体既具备 Agent 的推理与工具协同能力，又服务于认知雷达的感知-行动-反馈循环。

在运行流程上，显控指令、感知结果事件和雷达状态告警首先进入 Harness，形成带任务编号、目标/区域引用、风险等级和上下文版本的任务实例。随后，系统按照任务类型装载上下文，组织感知证据、雷达状态、历史任务、专业知识、参数策略和数据质量标识。LLM 推理引擎在雷达 Skills 约束下进行任务理解、证据解释、执行路径选择和结果生成。只读研判任务通过工具调用补证和分析；参数推荐任务通过仿真评估和约束检查形成候选方案；控制相关任务进入工作流，完成参数边界校验、人工确认、请求适配、状态跟踪和异常回滚准备。任务结束后，系统将全过程写入追溯记录，并把可复盘内容进入三阶分层记忆和评估机制。

第一阶段形成基础运行与上下文能力验证成果。该阶段重点建设任务实例、Harness 状态流转、上下文工程、证据项、追溯记录和三阶分层记忆基础框架，验证智能体能否稳定接收显控任务、感知事件和状态告警，能否围绕目标、区域、时间窗口和风险等级形成任务上下文，能否记录可回放的任务轨迹。阶段成果包括任务对象规范、Harness 运行样例、上下文装载模板、证据链样例、追溯记录样例、三阶分层记忆框架和基础任务闭环原型。

第二阶段形成典型雷达智能场景验证成果。该阶段重点围绕当前环境目标研判、智能调度建议和波形参数推荐开展验证，检验智能体能否综合感知结果、雷达状态、环境约束、历史案例和专业知识生成专业、可解释、可审查的结果。目标研判场景应形成目标类别、行为特征、异常程度、证据链、置信度和处置建议；调度与波形参数推荐场景应形成候选方案、推荐依据、适用条件、预期效果、风险说明、验证方式和回滚条件。阶段成果包括雷达 Skills 能力包、典型任务样例库、目标研判演示样例、参数推荐演示样例、输出质量检查规则和显控展示样例。

第三阶段形成受控执行与持续演进验证成果。该阶段重点验证工具治理、工作流门控、人工确认、控制请求适配、复盘评估和经验沉淀能力，推动雷达智能体从“能生成建议”走向“能进入受控工程闭环”。系统应能够把参数仿真、边界校验、审批确认、执行反馈和异常回滚纳入工作流；能够基于任务追溯记录和执行效果生成复盘结论；能够将经过评估的任务经验沉淀为长期经验资产，并在后续相似任务中受控引用。阶段成果包括受控工作流样例、参数调整闭环演示、异常回滚样例、复盘评估报告样例、智能体评估样例集、经验资产样例和持续优化验证结果。

通过上述路线，雷达智能体研究从“模型接入”提升为“认知雷达任务闭环设计”，从“功能模块堆叠”提升为“上下文、工具、工作流、记忆和评估共同治理的智能体工程”。最终形成面向雷达内部任务的智能决策中枢：能够理解任务、组织证据、生成方案、进入受控流程、记录全过程，并在复盘和经验治理中持续演进。
