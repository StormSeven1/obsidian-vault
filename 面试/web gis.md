## 2. 核心架构与领导力追问 (难度：⭐⭐⭐⭐)

### Q 1: [架构设计] 在 Next.js App Router 架构下，如何处理“服务端组件 (RSC)”与“客户端渲染的地图实例 (Cesium/ThreeJS)”之间的状态同步？

- **考察点**：Next.js 最新架构的实战经验。RSC 无法直接访问客户端 DOM 和地图实例，如何优雅传值而不引起地图重新初始化是核心难题。
    
- **详细理想答案**：
    
    1. **架构分层**：将地图容器作为特殊的 `Client Component`。利用 Next.js 的 `Shared Layout` 特性，确保地图实例在页面路由跳转时不会被销毁（Persistent Instance）。
        
    2. **状态同步机制**：
        
        - **数据下行**：服务端获取的 GIS 业务数据通过 `Props` 传给地图组件。
            
        - **双向同步**：使用客户端状态管理（如 **Zustand** 或 **Jotai**）作为中转站。当 URL 参数变化时，地图组件监听变化并调用引擎 API（如 `viewer.camera.flyTo`）执行动作，而不是让 React 重新渲染组件。
            
        - **Server Actions**：用户在地图上的交互（如点击标注）通过 Server Actions 传回后端，更新数据库后，再通过流式响应驱动 UI 更新。
            
- **评价要点**：优秀的候选人会强调“**不要让地图实例进入 React 的渲染循环**”，即通过 `useRef` 持有实例，通过副作用（useEffect）手动调用地图 API。
    

### Q 2: [方案设计] 我们需要构建 Server-Driven UI (服务端驱动 UI)。请问对于高度动态的 GIS 监控大屏，你如何设计一套协议，让后端能控制前端三维图层的加载顺序、样式和交互逻辑？

- **考察点**：系统解耦与大规模业务场景下的可维护性。
    
- **详细理想答案**：
    
    1. **协议定义 (Schema)**：设计一套基于 JSON 的配置协议。包含：`LayerDefinition` (图层数据源、类型)、`StyleRule` (基于地理属性的过滤变色规则)、`EventLinkage` (交互联动逻辑)。
        
    2. **渲染转换器 (Renderer)**：前端实现一个中转引擎。将后端的 JSON 配置实时解析为 Cesium/ThreeJS 的指令。例如，后端下发 `{"action": "highlight", "id": "tower_01"}`，前端执行高亮 Shader。
        
    3. **动态更新流**：利用 WebSocket 推送增量配置，实现无需前端发布代码，即可远程控制大屏的图层显示隐藏、视角切换和预警特效。
        
- **评价要点**：看候选人是否具备“**配置化思维**”。高级开发者会考虑如何将“业务逻辑”从“渲染引擎”中剥离，让后端具备动态控制前端三维场景的能力。
    

## 3. 技术深度与性能攻坚 (难度：⭐⭐⭐⭐⭐)

### Q 3: [内存管理] 在 72 小时不间断运行的系统中，大量的实时 WebSocket 推送会导致严重的内存压力。你如何实现“零垃圾 (Zero-GC)”的更新策略？

- **考察点**：高并发场景下的浏览器底层性能调优。
    
- **详细理想答案**：
    
    1. **对象池 (Object Pooling)**：禁止在推送循环中 `new` 对象。预先创建 10,000 个标注对象，当数据更新时，循环使用已有对象，仅修改经纬度等数值，避免触发 JavaScript 的垃圾回收（GC）。
        
    2. **二进制协议 (TypedArrays)**：弃用 JSON 格式，改用 **Protocol Buffers**。直接在内存中操作 `Float32Array`，显著降低解析 CPU 损耗和内存占用。
        
    3. **多线程离屏渲染 (OffscreenCanvas)**：将所有数学计算和数据解析放在 Web Worker 中，利用 `SharedArrayBuffer` 共享内存。即使主线程卡顿，地图渲染依然能保持 60 fps。
        
- **评价要点**：关注候选人是否提到 **TypedArrays**。这是处理实时海量地理数据（如雷达波、动态轨迹）的分水岭技术。
    

### Q 4: [图形学深度] 当相机拉得非常远或非常近时，会出现“画面抖动”或“模型闪烁”。你如何从逻辑和 Shader 层面彻底解决？

- **考察点**：地理信息系统（GIS）中由于坐标数值巨大导致的精度丢失问题。
    
- **详细理想答案**：
    
    1. **抖动原因**：32 位浮点数在表示地球级经纬度时精度不足。当相机位置很大而模型相对于相机的偏移量很小时，计算误差会导致模型跳变。
        
    2. **Shader 层面解决方案**：
        
        - **RTC (Relative to Center)**：在传给 GPU 之前，将模型坐标减去相机位置（Camera-Relative Rendering）。确保 GPU 计算的坐标始终在 0 附近，保证精度。
            
        - **双精度计算模拟**：在 Shader 中使用两个 `float` 模拟一个 `double` 精度。
            
    3. **深度冲突 (Z-fighting) 方案**：使用 **Logarithmic Depth Buffer (对数深度缓冲区)**。让相机附近的深度精度更高，解决远景和近景模型重叠时的闪烁。
        
- **评价要点**：这是验证侯彦庆“深耕源码”含金量的核心题。如果他能说出“RTC”或“Camera-Relative”，则是顶级图形专家。
    

## 4. 针对候选人的“杀手锏”提问

### 针对 侯彦庆 (侧重考察：自研引擎深度)

- **提问**：你的引擎如何解决 **100 GB 无人机倾斜摄影数据** 的解析速度问题？在 Web Worker 中传输大量模型数据时，如何避免“主线程阻塞”？
    
- **答案关键点**：应提到 **Transferable Objects**（所有权转移而非拷贝）和 **WebAssembly (Wasm)**。优秀的答案会涉及到如何并行解析 3D Tiles 树节点，以及如何利用 GPU 的 `Compressed Textures` 减少显存占用。
    

### 针对 赵通 (侧重考察：前瞻性与迁移能力)

- **提问**：从 WebGL 迁移到 **WebGPU**，架构上最大的改动是什么？它能为我们的实时感知系统带来什么质的飞跃？
    
- **答案关键点**：WebGPU 提供了 **Compute Shaders**。这意味着前端可以利用 GPU 进行大规模的地理分析计算（如淹没分析、通视分析），而不必求助于后端。这是从“可视化”向“实时空间计算”的跨越。
