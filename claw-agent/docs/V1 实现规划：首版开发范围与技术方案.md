# V1 实现规划：首版开发范围与技术方案

## 0. 文档定位

本文档是第 2 章总体架构的首版落地规划。它回答一个问题：**第一版到底做什么、不做什么、怎么做？**

本文档不修改第 2 章的架构约束，而是在 §11 首版收缩指南的框架内，将架构概念映射到具体的代码模块、技术选型与实现顺序。

## 1. 技术选型

### 1.1 总体技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| **Agent Runtime** | LangGraph (StateGraph + Checkpointer) | 状态图天然映射 10 步主循环；checkpoint 覆盖 P0 durable execution；interrupt/resume 覆盖审批流 |
| **后端网关** | FastAPI + SQLAlchemy | 已有 NexusUI 后端基础；WebSocket 支持实时推送；与 LangGraph Python 生态同语言 |
| **前端** | Next.js 16 + shadcn/ui + Zustand | 已有 NexusUI 前端原型；地图/军标/态势渲染已就绪 |
| **持久化** | PostgreSQL | LangGraph Checkpointer 原生支持；一等对象持久化；JSONB 支持灵活 schema |
| **消息总线** | Redis Streams | 轻量事件总线；支持消费者组；背压可通过 XLEN 监控 |
| **模型调用** | Anthropic SDK + OpenAI SDK (双 provider) | 支持模型路由的基础：主推理用 Claude，轻量任务用 GPT-4o-mini |

### 1.2 从 DeerFlow 借鉴的设计

| DeerFlow 模式 | 本系统映射 | 借鉴方式 |
|--------------|-----------|---------|
| Middleware Chain (18 层有序管道) | 主循环步骤管道 | 保留管道机制，替换为 10 步主循环逻辑 |
| LangGraph Server + Checkpointer | Agent Runtime 基础设施 | 直接使用 LangGraph 的 PostgreSQL checkpointer |
| GuardrailProvider (插件化审批) | 边界关卡 | 扩展为 5 个关卡，每个关卡独立实现 |
| Skill 系统 (YAML frontmatter) | 技能载体 | 复用 YAML 技能定义格式 |
| Sub-Agent 并发控制 | 工作单元 (Worker) | 复用双线程池 + 并发上限模式 |
| DeerFlowClient (嵌入式调用) | 后端内部调用 agent runtime | 不经 HTTP，直接 Python 调用 |

### 1.3 不从 DeerFlow 借鉴的部分

| DeerFlow 设计 | 不采用原因 |
|--------------|-----------|
| IM 消息入口 (Feishu/Slack) | 本系统是事件驱动，不是对话驱动 |
| ThreadState 对话模型 | 本系统是 10 个一等对象 + 五层状态链 |
| SQLite 持久化 | 生产级需要 PostgreSQL |
| 前端 (Next.js 聊天界面) | 已有 NexusUI 更专业的态势感知前端 |
| Nginx 反代 + 多进程部署 | 首版用单进程 FastAPI + LangGraph 即可 |

## 3. V1 功能范围

### 3.1 做什么

严格遵循第 2 章 §11.4 的实现顺序，V1 分为 5 个迭代交付：

#### 迭代 1：事件入口 + 主循环骨架 + 基础生命周期

**目标**：事件能进来、Run 能创建、主循环能跑通、状态能持久化。

| 功能项 | 详细说明 | 涉及的一等对象 |
|--------|---------|--------------|
| 事件归一化 | 用户请求、模拟雷达告警、定时触发 → 统一 Event schema | Event |
| 事件总线 | Redis Streams，支持发布/消费/消费者组 | Event |
| 准入关卡 | 声明式规则：事件能否进入、是否创建 Run、是否唤醒 Mission | Event, Run, Mission |
| 10 步主循环 StateGraph | LangGraph StateGraph，10 个节点按 §6 顺序，节点间通过 State 传递一等对象 | 全部 |
| Run 生命周期 | created → running → completed/failed/suspended，支持 checkpoint | Run |
| Mission 基本生命周期 | created → active → suspended → completed，可被事件唤醒 | Mission |
| Checkpoint/恢复 (P0) | Run/Mission 级别的 PostgreSQL checkpoint，进程崩溃可恢复 | Run, Mission |
| 快速运行路径 | 跳过 §6.4/§6.5，直接执行，覆盖只读查询和低风险动作 | Run |

**验收标准**：
- 模拟雷达告警事件 → 准入关卡放行 → 创建 Run → 主循环跑完 → 结果写入数据库
- 进程中途 kill → 重启后从 checkpoint 恢复继续
- 用户查询 "当前态势" → 快速运行路径 → 返回 Committed State 快照

#### 迭代 2：执行关卡 + 技能载体 + 审批流

**目标**：计划能执行、动作受约束、审批能挂起/恢复。

| 功能项 | 详细说明 | 涉及的一等对象 |
|--------|---------|--------------|
| 执行关卡 | 检查动作类型、工具白名单、区域约束、审批要求 | Plan, AssetLease, ReviewRecord |
| 技能载体 | YAML 定义 + Python 执行函数，输入输出清晰 | Plan, Run |
| 白名单直接动作 | 低风险动作（如查询态势、切换视图）免审批执行 | Event |
| 审批挂起/恢复 | LangGraph interrupt + signal，等待操作员审批后 resume | ReviewRecord, Run |
| 资产租约基本版 | 占用/释放，先不做冲突仲裁 | AssetLease |
| 操作员审批 UI | 前端 ApprovalCard → approve/reject → resume Run | ReviewRecord |

**验收标准**：
- 高风险动作（如控制无人机）→ 执行关卡拦截 → 前端弹出审批卡 → 操作员审批 → resume → 执行
- 技能调用（如"分析目标轨迹"）→ 执行关卡放行 → 执行 → 结果写入 Working State
- 低风险动作（如"切换 2D 地图"）→ 直接执行，无需审批

#### 迭代 3：验证 + 提交关卡 + 已提交状态

**目标**：判断经过验证才能提交，已提交状态由控制内核独占写入。

| 功能项 | 详细说明 | 涉及的一等对象 |
|--------|---------|--------------|
| 轻量验证 | Schema 检查 + 策略合规性检查 + 冲突检测 | EvidenceGraph, ClaimSet, ReviewRecord |
| LLM-as-judge 验证 | 对高风险判断使用异质模型做独立验证 | ClaimSet, ReviewRecord |
| 提交关卡 | 验证结果检查 + 冲突状态检查 + 治理模式检查 | ClaimSet, CommittedState, ReviewRecord |
| 已提交状态 (窄版) | 只覆盖 `mission_status` / `risk_level` / `asset_operational` 三个领域 | CommittedState |
| 提交权威 | 只有 control_kernel.commit_authority 能写入 CommittedState | CommittedState, Trace |
| 状态链 UI 指示器 | 前端显示 Observed→Evidence→Claim→Working→Committed 进度 | 五层状态链 |

**验收标准**：
- 深度审议运行产出判断 → 验证通过 → 提交关卡放行 → 写入 CommittedState
- 验证不通过 → 判断保留在 Working State，不提交
- 执行载体尝试直接写 CommittedState → 被提交权威拒绝

#### 迭代 4：追溯记录 + 审计基础 + 操作员面板

**目标**：全链路可追溯，操作员能看到"系统做了什么、为什么这样做"。

| 功能项 | 详细说明 | 涉及的一等对象 |
|--------|---------|--------------|
| 追溯记录 (追加式) | 每次 Run 的每一步写入 Trace（事件、决策、动作、结果、审查） | Trace |
| Trace 查询 API | 按 Run ID / Mission ID / 时间范围查询 | Trace |
| 操作员面板 UI | Mission 详情、Run 时间线、当前状态链位置 | Run, Mission |
| 接管关卡 | 操作员可暂停/恢复/终止 Run，接管动作写入 ReviewRecord | ReviewRecord, Run |
| 治理维度切换 | 前端切换 Observe / Semi-Auto / Approval-Gated / Human-Controlled | 运行边界配置 |

**验收标准**：
- 任意一次 Run 完成后，能通过 API 查到完整 Trace
- 操作员在前端点击"接管" → Run 暂停 → 操作员手动操作 → 点击"恢复" → Run 继续
- 切换到 Human-Controlled 模式后，所有动作都需要审批

#### 迭代 5：深度审议 + 模型路由 (基础版)

**目标**：复杂态势能被深度分析，模型能按调用点路由。

| 功能项 | 详细说明 | 涉及的一等对象 |
|--------|---------|--------------|
| 深度审议运行 | 完整走完 §6.4 上下文装配 → §6.5 计划形成 | EvidenceGraph, ClaimSet, Plan |
| 上下文装配 | 从状态链 + 记忆中投影出模型可见上下文，受预算约束 | 五层状态链 + 记忆 |
| 计划形成 | LLM 生成结构化计划（调查计划或行动计划） | Plan |
| 工作单元 (Worker) 基础版 | 并发上限 1，受限 toolset，结果通过引用交接 | Plan, Run |
| 模型路由 (简化版) | 两个档位：主推理 (Claude) / 轻量任务 (GPT-4o-mini)，配置文件指定 | 无 (横切) |
| 记忆 (简化版) | 情景记忆：历史 Run 摘要；语义记忆：目标画像。存 PostgreSQL | 记忆侧 |

**验收标准**：
- 复杂雷达告警事件 → 升级为深度审议运行 → 上下文装配 → 生成计划 → 执行计划 → 验证 → 提交
- 主推理用 Claude，轻量验证用 GPT-4o-mini，可在配置中切换
- Worker 执行后，结果通过引用传递给主 Run，不共享上下文

### 3.2 不做什么

以下能力**不在 V1 范围内**，但架构骨架预留位置：

| 不做 | 原因 | 预计版本 |
|------|------|---------|
| 整合运行 (Consolidation Run) | 依赖足够的运行数据积累 | V2 |
| 工作流载体 (Workflow) | 首版先用 Skill 覆盖，Workflow 需要流程编排引擎 | V2 |
| CEP 复杂事件处理 | 首版只做简单阈值模式检测（如"N 分钟内同区域 M 次告警"） | V2 |
| 记忆巩固 (Dreaming) | 依赖整合运行 | V2 |
| 模型路由完整版 (多档位 + 降级策略 + 成本预算) | 首版只做两档位 | V2 |
| Trace 回放 UI | 首版只做追加式记录 + 查询 | V2 |
| 资产冲突仲裁 | 首版只做占用/释放，不做抢占 | V2 |
| 边界规则 DSL | 首版用声明式 YAML 配置 | V2 |
| 异质对抗验证 | 首版只做 LLM-as-judge | V3 |
| 评估闭环 (Evaluation) | 依赖足够的 Trace 积累 | V3 |

## 4. 关键技术决策

### 4.1 主循环的 LangGraph 实现

10 步主循环实现为一个 LangGraph StateGraph，每步是一个节点：

```python
# agent-runtime/graphs/main_loop.py (设计草案)

from langgraph.graph import StateGraph, END
from agent_runtime.state.objects import RunState  # 包含所有一等对象的引用

def build_main_loop() -> StateGraph:
    graph = StateGraph(RunState)

    # 10 步主循环节点
    graph.add_node("event_ingest", event_ingest)       # §6.1
    graph.add_node("boundary_check", boundary_check)   # §6.2
    graph.add_node("run_decision", run_decision)       # §6.3
    graph.add_node("context_assembly", context_assembly) # §6.4 (快速运行跳过)
    graph.add_node("plan_formation", plan_formation)    # §6.5 (快速运行跳过)
    graph.add_node("execute", execute)                  # §6.6
    graph.add_node("verify", verify)                    # §6.7
    graph.add_node("commit_and_publish", commit_and_publish) # §6.8
    graph.add_node("monitor_and_reentry", monitor_and_reentry) # §6.9
    graph.add_node("archive", archive)                  # §6.10

    # 顺序边
    graph.add_edge("event_ingest", "boundary_check")
    graph.add_edge("boundary_check", "run_decision")

    # 条件边：快速运行跳过 §6.4/§6.5
    graph.add_conditional_edges("run_decision", route_by_cognitive_depth)
    # 深度审议: run_decision → context_assembly → plan_formation → execute
    # 快速运行: run_decision → execute

    graph.add_edge("execute", "verify")

    # 条件边：只读无副作用可跳过验证
    graph.add_conditional_edges("verify", route_by_verification_level)

    graph.add_edge("commit_and_publish", "monitor_and_reentry")
    graph.add_edge("monitor_and_reentry", "archive")

    # 条件边：Mission 仍活跃则回到 event_ingest
    graph.add_conditional_edges("archive", route_by_mission_status,
        {"continue": "event_ingest", "end": END})

    return graph
```

### 4.2 关卡的确定性执行

关卡不经过 LLM，是纯 Python 判定逻辑：

```python
# agent-runtime/gates/admission.py (设计草案)

class AdmissionGate:
    """准入关卡：决定事件能否进入系统"""

    def check(self, event: Event, config: BoundaryConfig) -> GateDecision:
        # 1. 检查事件类型白名单
        if event.type not in config.allowed_event_types:
            return GateDecision(deny, reason="事件类型不在白名单")

        # 2. 检查是否需要创建深度审议运行
        if event.risk_level >= config.deliberative_threshold:
            return GateDecision(allow, run_type="deliberative")

        # 3. 检查是否需要挂起
        if event.requires_approval:
            return GateDecision(ask, reason="需要审批", suspend=True)

        return GateDecision(allow, run_type="fast")
```

### 4.3 NexusUI 前端对接

后端通过 WebSocket 推送运行时事件到前端：

```
Agent Runtime (LangGraph)
    │
    │ 每个 node 完成后发布事件
    ▼
Redis Streams (event bus)
    │
    │ FastAPI WebSocket handler 消费
    ▼
NexusUI WebSocket Client (nexus-client.ts)
    │
    │ 更新 Zustand stores
    ▼
UI 组件响应式更新
```

前端 Zustand stores 与一等对象的对齐：

| Store | 一等对象映射 | 新增字段 |
|-------|------------|---------|
| `track-store` | Event (触发源) + EvidenceGraph (证据) | 增加 `evidenceId`, `confidence` |
| `asset-store` | AssetLease (租约) | 增加 `leaseStatus`, `leasedBy`, `priority` |
| `zone-store` | CommittedState (一部分) | 关联 `commitId`, `committedAt` |
| `alert-store` | Event + ReviewRecord | 增加 `reviewStatus`, `gateDecision` |
| `app-store` | Run + Mission | 增加 `runId`, `runStatus`, `currentStep`, `cognitiveDepth` |
| `mission-store` (新) | Mission 生命周期 | 全新：`missionId`, `status`, `runs[]`, `checkpoint` |
| `trace-store` (新) | Trace | 全新：`runTrace[]`, `stepRecords[]` |

### 4.4 检查点/恢复实现

使用 LangGraph 的 PostgreSQL checkpointer：

```python
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

# 每个 Run/Mission 的状态自动持久化到 PostgreSQL
checkpointer = AsyncPostgresSaver.from_conn_string(DATABASE_URL)

# 恢复：通过 thread_id + checkpoint_id 恢复到任意点
# 审批挂起：interrupt 后状态保留在 checkpointer
# 操作员审批后：通过 thread_id resume
```

## 6. 验证方案

### 6.1 端到端测试场景

| 场景 | 覆盖的迭代 | 验证要点 |
|------|-----------|---------|
| 模拟雷达告警 → 自动研判 → 态势更新 | 1→2→3 | 事件驱动完整闭环 |
| 用户查询 "3 号区域态势" | 1 | 快速运行路径 |
| 无人机巡逻任务 → 发现异常 → 审批 → 执行 | 1→2→3→4 | 审批挂起/恢复 + Mission 持续 |
| 复杂多源告警 → 深度分析 → 提交判断 | 1→2→3→5 | 深度审议完整路径 |
| 操作员接管 → 手动操作 → 恢复自动运行 | 4 | 接管关卡 + 审计链完整 |
| 进程崩溃 → 重启 → 从检查点恢复 | 1 | Durable execution |

### 6.2 单元测试要求

- 每个关卡：测试 allow/deny/ask 三种决策路径
- 主循环：测试快速运行 vs 深度审议运行的路径选择
- 提交权威：测试非授权写入被拒绝
- 状态链：测试 Observed→Evidence→Claim→Working→Committed 的推进规则
- Checkpoint：测试 save → kill → restore 的完整性

## 7. 里程碑与交付节奏

| 里程碑 | 迭代 | 核心交付物 | 预计工期 |
|--------|------|-----------|---------|

## 8. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| LangGraph checkpoint 性能不足 | 首版 Run 量级小；监控 checkpoint 写入延迟；必要时切换到自定义 checkpointer |
| 状态链实现过于复杂 | 首版 B 级对象窄版，字段收窄到最小可用 |
| 前端改造影响现有 NexusUI 功能 | 新增组件独立目录 (`mission/`, `governance/`, `evidence/`)，不修改现有组件 |
| 关卡逻辑频繁变更 | 声明式 YAML 配置，不改代码即可调整规则 |
| 模型调用成本过高 | 模型路由确保轻量任务用小模型；上下文预算限制 token 消耗 |
