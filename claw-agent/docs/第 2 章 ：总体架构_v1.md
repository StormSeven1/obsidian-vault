## 1. 架构目标

总体架构的目标**不是**扩展更多业务 Agent、增加更多并列运行链路、或堆叠更多概念层级，而是：

> **在系统愿景与需求基线的约束下，构建一个长期稳定、治理统一、状态清晰、可逐步演进的系统骨架。**

为达成这一目标，架构必须同时满足以下条件：系统能承载事件驱动的持续运行而非仅服务单轮对话；支持快慢不同认知深度下的任务执行；将权限、审批、风险与回滚边界前置为运行时约束而非事后补丁；将状态、证据、执行、验证与审计纳入同一控制体系而非各自为政。与此同时，系统应在不增加中心节点的前提下支持技能、工作流、工作单元与长周期任务的持续扩展；让学习与演进作为后处理回路存在，不进入实时关键路径；并将运行时合同——检查点/恢复、边界规则、上下文纪律、提交权威——提升为一等架构约束，而不是实现层留给后续章节的"建议"。

首要原则：

> **在保证能力闭环完整的前提下，让系统只保留少数稳定抽象，并让每个抽象只承担一种主要职责。**

## 2. 架构总定义

**最小定义**（先抓住主干，再展开细节）：

> 本系统本质上是：**在运行边界约束下，由控制内核治理、由深度审议规划、由执行平面执行、由状态平面承载真相、由操作员监督接管的单一控制平面。**

完整定义如下：

> **一个以控制内核为最小可信内核、以边界关卡作为确定性执行边界、以快速运行 / 深度审议运行 / 任务持续运行为认知深度模式、以工作流 / 技能 / 工作单元为执行载体、以观测事实 / 证据 / 判断与假设 / 工作草稿 / 已提交状态为单一真相链、由验证 / 审批 / 仲裁 / 审计 / 上下文纪律 / 模型路由六项横切治理能力贯穿、并以整合运行在后处理回路中承担能力演进的态势感知智能控制平面。**

该定义包含七项关键判断：系统只有一个控制平面，不是多个并列主脑；快慢差异体现在运行深度而非系统数量；执行形式体现在工作流 / 技能 / 工作单元而非业务场景 Agent；长周期任务解决持续性问题而非引入新的系统中心；任何关键状态变更都必须回到统一提交与审计链；演进在后处理回路中发生，不进入实时关键路径；核心运行时合同与骨架结构同等重要，不可降级为实现细节。

## 3. 七运行平面

架构采用七个稳定运行平面。每个平面只承担一种主要职责，平面之间通过一等运行对象交互。

其中，**核心运行平面**（4 个）承载系统主运行链：世界接口 → 边界判定 → 治理裁决 → 认知与执行。**支撑平面**（3 个）承载状态、观测与演进，为核心平面提供服务但不参与主运行链的直接决策。

### 3.1 世界接口平面

世界接口平面负责系统与外部世界之间的统一接入与输出，是系统的唯一进出关口。

输入侧涵盖用户请求、雷达、视频、GIS、数据库、知识库、定时任务、策略触发、资产状态回调、执行回执、历史案例与外部资料等异构来源。输出侧覆盖 WebGIS、相机云台、雷达控制、无人机任务下发、通知告警、报表及外部业务系统 API。世界接口**只负责**协议适配、数据标准化、指令回执、能力注册、异常转换以及事件模式检测与时序关联，**不负责**业务判断与状态提交——这些职责由下游平面承担。

值得强调的是，世界接口不仅是协议转换器。对于态势感知系统而言，多源事件的时序关联、模式检测（如"5 分钟内同一区域 3 次雷达告警"）和事件聚合是基础能力。这些复杂事件处理（CEP）能力在事件归一化之后、边界检查之前发生，产出增强后的事件或衍生事件。CEP 的具体规则表达方式和检测引擎属于实现层选择，本文档只锁定"CEP 是世界接口的职责之一"。

### 3.2 运行边界平面

运行边界平面负责定义系统在技术、权限、风险、策略上可以执行到什么程度，统一管理权限边界、资产作用域、区域与时间窗约束、网络访问约束、高风险动作准入、审批停点、回滚前提以及凭证与调用范围。

其设计意图在于将「能不能做」「什么时候必须停下来问人」这两类判断从智能规划中独立出来，使运行边界不依赖模型的临场判断。本平面在主循环的多个点位被查询（详见 §6），而非一次性前置判断——边界的评估是持续的、嵌入主循环的。

在实现层，设计层的运行边界必须落到明确的执行面（Enforcement Surface）上，不能停留在抽象概念。至少包括以下五个关卡：

| 执行面 | 触发时机 | 主要作用 | 主要涉及对象 |
|---|---|---|---|
| **入口 / 准入关卡** | 事件接入之后、运行决策之前 | 决定事件是否允许进入系统、是否允许创建深度审议运行、是否需要挂起等待 | 事件 / 运行实例 / 长周期任务 |
| **执行关卡** | 计划步骤执行前 | 检查动作类型、工具、资产、区域、时间窗、审批要求 | 计划 / 资产租约 / 审查记录 |
| **资产控制关卡** | 资产租约获取与动作下发前 | 检查资产是否可占用、是否冲突、是否允许抢占 | 资产租约 / 长周期任务 / 运行实例 |
| **提交关卡** | 判断升级到已提交状态前 | 检查验证结果、冲突状态、治理模式、冻结指令 | 判断与假设集 / 审查记录 / 已提交状态 |
| **接管关卡** | 操作员发起接管/停止/回滚时 | 确保人工接管、停止、恢复、回滚进入统一治理链 | 审查记录 / 运行实例 / 长周期任务 / 已提交状态 |

这些关卡在实现上可体现为中间件、钩子、策略引擎、审批拦截器或提交审查器，本文档不锁定具体技术形态，但锁定"**所有关键动作必须命中明确关卡**"这一要求。

### 3.3 控制内核

控制内核是系统的最小可信内核，也是唯一必须始终在线、始终一致、始终可恢复的治理中心。它统一承担事件接入、事件去重合并与优先级、运行模式判定、运行实例生命周期、资源锁与冲突仲裁、审批流挂起与恢复、提交审查、审计主链以及熔断降级回滚等职责，但不承担复杂开放推理、具体业务分析与领域逻辑。

控制内核的核心定位可以概括为一句话：它负责**治理权威（authority）**，不负责**智能推理（intelligence）**，也不负责**业务逻辑（domain logic）**。它是裁决者，不是参与者。

> **即使高级智能单元暂时不可用，系统仍能维持基本治理、标准流程执行、关键状态一致性与安全边界。**

控制内核是单一治理权威，但不等于单一"大类"或单一"巨型进程"。为防止实现时膨胀，首版就应按以下五个子职责组拆分模块边界：

1. **入口与运行准入**：负责事件接入、去重、合并、优先级处理与运行实例创建准入。
2. **生命周期与调度**：负责运行实例与长周期任务的生命周期、挂起恢复、回调重入、调度与超时。
3. **策略与审查协调**：负责运行边界查询、审批流编排、人工接管与仲裁协同。
4. **提交与审计权威**：负责提交审查、已提交状态写入、追溯记录主链与审计出口。
5. **弹性与降级控制**：负责熔断、降级、恢复、回滚与健康状态监控。

后续实现可以把这五组拆成独立模块、服务或 actor，但它们在语义上仍共同构成唯一的治理权威，不能形成多个彼此竞争的"控制中心"。

### 3.4 审议与计划平面

审议与计划平面负责复杂理解、上下文装载、证据组织、调查计划与行动计划形成，其职责范围涵盖自然语言理解、上下文装载与裁剪、调查计划与行动计划生成、复杂态势研判、技能 / 工作流 / 工作单元编排以及长周期任务的阶段性监督与重规划。

本平面内的协调者是**受控的认知编排器**，不是万能总控——它可以发散分析，但不能绕过控制内核直接修改全局主状态。本平面内部进一步的子组件划分（如上下文装配、计划生成、编排、内嵌自检、失败反思）由《认知核》章节展开，本文档不锁定。

### 3.5 执行平面

执行平面的职责是把计划转化为实际动作或稳定流程执行。系统只保留三类稳定执行载体：

- **工作流（Workflow）**：面向高频、流程稳定、可代码化、确定性高的任务。
- **技能（Skill）**：面向输入输出清晰、可复用、能力边界明确、不必长期持有全局状态的专门能力。
- **工作单元（Worker）**：面向一次性复杂分析、并行探索、多数据源综合、长任务中的受限子任务。

所有执行载体必须受上下文、权限、预算与写权限约束，**不得直接写入已提交状态**。直接动作与 UI 执行器作为极简动作出口归属本平面，但不视为独立载体类型——它是无需规划的直接调用，由控制内核转发至世界接口层。

### 3.6 状态与知识平面

状态与知识平面承载系统的世界模型与长期知识。它在架构图上是一个平面，但内部明确分为两个子区：运行时状态侧的五层状态链承载实时运行时真相的推进与治理；记忆侧的四类长期记忆承载跨运行实例的知识积累与检索。两者职责不同：状态回答"现在怎样"，记忆回答"过去怎样、下次怎样"。

#### 运行时状态侧

五层状态链构成系统的单一真相链，自底向上依次为：

1. **观测事实（Observed Facts）**：原始观测与外部输入。
2. **证据图（Evidence Graph）**：证据、来源、支持 / 反驳关系。
3. **判断与假设（Claims / Hypotheses / Projections）**：当前判断、候选解释、待验证结论，以及基于当前证据对未来状态的推演预测（如"目标预计 15 分钟后进入禁飞区"）。
4. **工作草稿（Working State）**：执行草稿、计划草稿、中间结果。
5. **已提交状态（Committed State）**：对外生效的单一真相与主状态。

这一结构的核心意义在于三层分离：把事实与判断分开，把判断与提交结论分开，把中间工作态与最终全局真相分开。

**与 Endsley 态势感知三级模型的对齐**

本五层状态链覆盖了 Endsley 态势感知模型的三个层级。第 1 级（感知，Perception）对应观测事实加证据图，负责感知环境中的元素及其属性；第 2 级（理解，Comprehension）对应判断与假设，负责理解元素的意义并形成对当前态势的判断；第 3 级（预测，Projection）对应判断与假设集中的推演预测子类，负责基于当前理解推演近期未来状态。

推演预测不是独立的状态层，而是判断与假设集中的一种特殊 claim_type。推演预测与当前判断共享相同的证据基础和验证流程，不需要独立的治理链路。但推演预测在认知层面是独立的——它是"将会怎样"，不是"现在怎样"。Endsley 的研究表明，76% 的态势感知错误来自感知层（第 1 级），但真正致命的决策失误往往来自缺乏预测（第 3 级）。本设计将推演预测纳入判断与假设的治理链路，确保推演预测也经过证据支撑和验证审查。

**与 JDL 数据融合模型的映射**

| JDL 层级 | 职责 | 本系统对应 |
|-----------|------|-----------|
| 第 0 级（预处理） | 信号预处理、坐标变换 | 世界接口协议适配 |
| 第 1 级（目标提炼） | 目标检测、定位、跟踪 | 观测事实 → 证据图 |
| 第 2 级（态势评估） | 关系识别、态势聚合 | 证据图 → 判断与假设 |
| 第 3 级（影响评估） | 威胁评估、后果预测 | 判断与假设（推演预测）+ 计划形成 |
| 第 4 级（过程优化） | 自适应资源调度 | 模型路由 + 整合运行 |

JDL 第 4 级对应的不是单一对象，而是系统通过模型路由和整合运行实现的自我优化能力——即"调度传感器、分配算力、优化流程"的能力。

#### 记忆侧

长期知识同样归属本平面，建议分为四类：指令记忆（Instruction Memory）存放规则、策略、区域约束与操作员偏好；情景记忆（Episodic Memory）存放具体案例、历史事件与失败记录；语义记忆（Semantic Memory）存放经验的泛化知识、目标画像与地物常识；能力记忆（Capability Memory）存放系统自身拥有的技能与工作流元数据。记忆用于支持上下文装载、相似案例召回与能力演进，**不替代实时状态治理**。

**态势模型**

五层状态链在任何时刻的快照，构成了系统的**态势模型（Situation Model）**——即系统当前对世界"正在发生什么"的整体理解。态势模型不是第 11 个对象，而是证据图 + 判断与假设集 + 已提交状态在给定时刻的联合视图。它为操作员提供"当前态势一目了然"的视图基础，为上下文装配（§6.4）提供"当前理解的整体轮廓"而非零散片段，为任务持续运行提供跨运行实例的连续态势基线。

态势模型的具体 UI 表现和查询接口由《审计、观测、回放与评估》章节展开。本文档只锁定"态势模型是状态链的实时联合视图，不是独立存储对象"。

### 3.7 操作员与可观测平面

操作员与可观测平面将人纳入运行时而非置于系统之外。该平面承载 UI 意图、审批与拒绝、接管 / 停止 / 恢复、任务时间线、追溯记录与回放、阶段性解释以及诊断与评估等能力，确保系统不仅「能跑」，还能被审查、被理解、被接管、被持续优化。

## 4. 一等运行对象

为防止架构继续围绕 Agent 名称扩张，系统围绕以下十个一等运行对象建模。所有平面通过这些对象交互，不通过直接调用。

| 对象 | 作用 |
|------|------|
| **事件（Event）** | 统一输入对象：用户请求、雷达告警、视频异常、定时触发、审批结果、执行回调在入口处都先被归一化为事件 |
| **运行实例（Run）** | 一次处理过程的运行对象，分任务运行与整合运行两类。任务运行承担实时任务；整合运行承担背景演进 |
| **长周期任务（Mission）** | 长生命周期业务目标对象，可跨多个任务运行持续存在，由事件与调度器共同推进 |
| **证据图（Evidence Graph）** | 证据组织对象，表达证据来源、支持关系、冲突关系与完整性 |
| **判断与假设集（Claim Set）** | 当前判断与候选假设集合，表达系统「当前认为可能成立的结论」 |
| **计划（Plan）** | 调查计划或行动计划对象，必须是独立工件，而不是某次模型输出的副产品 |
| **资产租约（Asset Lease）** | 资产占用对象，表达对相机、无人机、雷达等资源的租约、冲突与优先级 |
| **审查记录（Review Record）** | 审批、验证、拒绝、人工接管、回滚建议等治理对象 |
| **已提交状态（Committed State）** | 当前对外生效的全局真相对象 |
| **追溯记录（Trace）** | 系统为何这样判断、规划、执行、提交的全过程记录对象 |

具体 schema 与状态机由《运行时对象与状态模型》章节展开，本文档只锁定对象名称与总体职责。

## 5. 正交运行模式维度

系统的运行模式不是一组平铺的通道（lane），而是四组正交维度。任一具体运行都是四个维度的组合，不能在维度之间"长出新的一层"。

### 5.1 触发维度

系统可由用户触发、外部事件触发、定时触发、执行回调触发、策略 / 记忆触发等方式唤醒。触发方式决定运行实例的入口参数和初始上下文，但不影响后续的认知深度、治理模式与执行载体选择——这正是"正交"的含义。

### 5.2 认知深度维度

认知深度维度反映系统对当前任务投入多少认知资源，分为三档：

- **快速运行（Fast Run）**：适用于只读查询、UI 操作、已缓存结果、白名单低风险动作、标准工作流入口触发等场景。不创建开放式工作单元、不做重型自由规划。
- **深度审议运行（Deliberative Run）**：适用于多源证据整合、复杂研判、高风险动作前分析、多资产协同决策、需要补证与重规划的任务。需要上下文装载、计划形成，可创建受限工作单元，结果经过验证与提交审查。
- **任务持续运行（Mission Continuation）**：适用于持续跟踪、长时监视、多阶段巡逻、长任务推进等场景。**不是第三个主脑**，而是「长周期任务状态 + 调度机制 + 可被事件反复唤醒的深度审议运行」。

任何快速运行在出现歧义、风险上升或失败时必须**立即升级**到深度审议运行。升级是单向的，不存在深度审议运行主动降为快速运行。

### 5.3 治理维度

治理维度决定系统在当前任务中有多大的自主执行空间，分为四档：观察与建议（Observe / Advise）只观察与建议，不执行任何动作；半自动（Semi-Auto）允许低风险动作自动执行；审批门控（Approval-Gated）要求关键动作经审批后继续；人工控制（Human-Controlled）由人工接管，系统转为辅助角色。治理维度的默认值由运行边界平面决定，不允许在审议与计划平面临场更改。

### 5.4 执行载体维度

执行载体维度决定任务以何种形式落地：直接动作 / UI 执行器处理极低风险低时延动作；工作流（Workflow）承载稳定流程；技能（Skill）承载专门能力；工作单元（Worker）承载受限复杂分析单元。执行载体的选择由计划阶段决定，受治理维度和边界规则的约束。

## 6. 运行主循环

系统标准运行闭环分十步。十步内部顺序不可更改，叙事上可分为**主运行链**（8 步）和**持续链**（2 步）。主运行链（§6.1 → §6.8）覆盖事件接入 → 边界检查 → 运行决策 → 上下文装配 → 计划形成 → 执行 → 验证 → 提交与发布，是每次任务的完整闭合路径，不可跳步。持续链（§6.9 → §6.10）覆盖监控与重入及归档，处理回调重入、长周期任务持续与追溯记录归档。此外，§8 整合运行作为独立演进回路与主循环完全分离。

每一步明确在哪个平面发生、读写哪些一等对象。

| 步骤 | 名称 | 主要发生平面 | 读写的一等对象 |
|------|------|----------------|----------------|
| 6.1 | **事件接入** | 世界接口 → 控制内核 | 写事件 |
| 6.2 | **边界检查** | 运行边界 | 读事件 / 指令记忆；挂起或放行 |
| 6.3 | **运行决策** | 控制内核 | 创建运行实例（快速 / 深度审议）或唤醒长周期任务 |
| 6.4 | **上下文装配** | 深度审议 | 读证据图 / 判断与假设集 / 记忆 / 指令；写模型可见上下文 |
| 6.5 | **计划形成** | 深度审议 | 写计划（或绑定现有工作流） |
| 6.6 | **执行** | 执行 | 读计划；调用直接动作 / 工作流 / 技能 / 工作单元；写工作草稿 / 证据 |
| 6.7 | **验证** | 横切验证介入 | 读计划 / 证据图 / 工作草稿 / 策略；写审查记录 |
| 6.8 | **提交与发布** | 控制内核 | 从判断升级写入已提交状态；触发 UI 意图 / 通知 / 动作 |
| 6.9 | **监控与重入** | 控制内核 | 等待回调 / 新事件 / 调度触发；若长周期任务仍活跃则继续循环 |
| 6.10 | **归档** | 控制内核 → 状态与知识 | 写追溯记录；运行实例状态归档 |

主循环有四项关键设计决策。第一，**边界检查在计划形成之前**：先确定能做什么，再讨论怎么做，防止无效规划浪费资源。第二，**验证在提交之前**：任何判断升级到已提交状态前必须经过独立验证，确保"做了不等于认了"。第三，**学习回路不在主循环内**：§6.10 归档只做归档；真正的学习与能力演进在 §8 整合运行中独立进行。第四，快速运行可跳过 6.4 / 6.5，但 6.1 / 6.2 / 6.3 / 6.6 / 6.7 / 6.8 / 6.10 不能省——只读无副作用的快速运行可跳过 6.7 验证，但任何有副作用的动作（包括白名单低风险动作）必须经过至少轻量验证或显式白名单检查，不允许在快路径上形成治理豁免。

**实现映射**：主循环在工程落地时，至少应映射为以下关卡 / 权威链：

- `6.1-6.3`：入口 / 准入关卡 + 运行权威
- `6.4-6.5`：深度审议运行时 + 上下文投影
- `6.6`：执行关卡 + 资产控制关卡
- `6.7`：独立验证
- `6.8`：提交关卡 + 提交权威
- `6.9-6.10`：生命周期 / 调度 + 审计权威

也就是说，第 2 章的主循环并不要求"一个大循环函数"实现，而是要求实现层能清晰回答：每一步由谁负责、在哪个关卡被拦住、由谁拥有最终写权限。

**OODA 映射**

本系统的 10 步主循环是 OODA 循环在 AI-native 控制系统中的具体实例化。观察（Observe）对应 §6.1 事件接入，外部世界、资产与用户输入归一化为事件；判断（Orient）对应 §6.2 边界检查 + §6.3 运行决策 + §6.4 上下文装配，先判定"能不能做"，再决定"以什么深度做"，再装载上下文形成当前态势理解；决策（Decide）对应 §6.5 计划形成，基于判断产物形成调查计划或行动计划；行动（Act）对应 §6.6 执行，把计划转化为实际动作。

在此之外，本系统对经典 OODA 增加了两项关键扩展。验证（Verify，§6.7）在行动和提交之间插入独立审查，防止"做了就认"；提交（Commit，§6.8）在写入全局真相前强制治理检查，保证"认了才生效"。监控（Monitor，§6.9）则等待反馈并观察行动的效果，回到观察阶段形成闭环。

## 7. 横切治理能力

所有运行平面与主循环共享以下六项横切治理能力。"横切"意味着这些能力不属于任何单一平面，但每个平面都必须配合它们。

### 7.1 验证

验证能力独立于执行者，至少应覆盖四个维度：证据充分性——证据是否足以支撑判断；策略合规性——是否符合运行边界；冲突检测——是否与现有已提交状态、资产租约或其他运行实例冲突；结果验证——动作执行结果是否与预期一致。

验证与评估必须严格分离。验证是运行时能力，对单次动作和单次判断负责；评估是离线能力，对系统整体质量、回归检测和能力升级负责。评估包括追溯记录评分、工具选择评估、论证提取评估、升级评估与持续回归测试集等。两者不可混淆。验证的具体分档（轻量 schema 检查 / LLM-as-judge / 异质对抗 / 人工复核）由《证据、判断与验证》章节展开；评估的具体框架由《审计、观测、回放与评估》章节展开。

### 7.2 审批与人工接管

审批与人工接管必须存在于统一控制平面，不能散落在执行单元内部。系统必须提供审批挂起与恢复、人工接管、停止、降级、回滚等能力，确保任何需要人类介入的场景都经过统一的治理链路。

### 7.3 资源仲裁

关键资产和关键状态通过统一租约、锁与仲裁机制管理，涵盖资产租约、已提交状态的写锁、长周期任务主计划锁以及审批对象锁。仲裁的目的是防止并发运行实例之间的资源冲突和状态竞争。

### 7.4 审计与回放

所有关键运行对象与关键动作必须可审计、可回放。追溯记录以对象级别记录（事件 / 运行实例 / 长周期任务 / 计划 / 判断 / 审查 / 提交），支持按对象 ID 回放。系统应采用行业标准语义约定（如 OpenTelemetry GenAI），确保审计数据可被通用工具链消费。

### 7.5 上下文纪律

模型可见上下文、任务运行状态、本地系统状态、长期记忆必须分层装载，避免上下文膨胀与角色混乱。这里有三条核心边界：系统运行状态（一等对象）不等于模型可见上下文；长期记忆必须通过检索进入模型上下文，不默认加载；工作单元之间不共享上下文，通过引用与文件交接结果。

上下文合同大纲如下（具体实现由《认知核》和《审计》章节展开）：

| 合同项 | 要求 |
|--------|------|
| **对象投影规则** | 一等对象不直接进入提示词（prompt）；只有投影进入。投影规则按任务类型定义 |
| **按需检索** | 长期记忆不默认注入，通过检索按任务需要装载 |
| **上下文预算** | 每种运行实例类型有上下文 token 预算；超出时触发压缩 / 摘要 |
| **工作单元交接** | 工作单元之间通过引用和文件交接结果，不共享上下文 |
| **监控/通知剥离** | 监控、通知、展示信息必须从主推理上下文中剥离，通过独立通道传递 |
| **敏感信息隔离** | 凭证、密钥等永远不进入模型上下文 |

### 7.6 模型路由

不是每个决策都需要同等级别的模型。模型路由作为横切能力统一管理每个调用点（事件解析、上下文裁剪、计划生成、编排决策、内嵌自检、独立验证、失败反思、场景归一化等）的模型档位选择、降级策略与成本预算。

其设计意图有三：让模型升级 / 替换 / 降级变成配置变更，不波及上层逻辑；让异质验证与同档推理分离，避免同模型自证；让成本与时延 SLO 可按调用点独立管理。具体档位选择由《模型平面：模型路由》章节展开，本文档只锁定"模型路由是横切能力"这件事。

## 8. 整合运行：演进回路的独立对象

### 8.1 为什么不在主循环里

主循环必须保持对时延敏感、可审计、可接管。把「学习」放进主循环会产生三个问题：拖慢实时路径；让失败的学习污染正在服务的任务；让能力演进与当前请求的推理混淆，破坏审计清晰度。

因此，**学习与能力演进必须作为独立的后处理运行对象存在**，与任务运行共享运行实例这一抽象，但生命周期、触发机制、权限边界完全独立。

### 8.2 整合运行的定义

> **一类由调度器周期触发、在严格受限权限下运行的后台运行实例，用于从归档的任务运行 / 证据 / 追溯记录中蒸馏长期记忆、提议新能力、建议策略优化。**

### 8.3 触发机制

整合运行采用三重门控机制，按成本从低到高依次检查：时间门控确保距上次整合运行已超过配置阈值；物料门控确保累积未处理的任务运行归档数量达到阈值；负载门控确保无关键长周期任务占用、无关键审批挂起且控制内核空闲。任一门控不满足则本轮不触发。

### 8.4 权限边界

整合运行必须是严格受限的。它对任务运行归档、证据图、追溯记录与记忆只读；只允许写入记忆（情景 / 语义 / 能力）和能力提议队列；严禁写入已提交状态、直接注册生产技能 / 工作流或修改指令记忆。

### 8.5 产出形态

整合运行产生的不是直接上线的能力，而是**候选提议**，包括新语义记忆条目候选、新技能候选（带使用场景与输入输出契约草案）、新工作流候选（带稳定性证据）以及策略调整建议（如某区域误报率高，建议收紧阈值）。**所有候选必须经过操作员复核（人工升级关卡）才升级为生产能力**，这是系统演进可控性的关键锁。

### 8.5b 升级流水线

整合运行的产出升级不只是"人工复核"，而应经过完整的六阶段流水线：候选生成（整合运行产出候选提议）→ 离线评测（如新技能的输入输出测试）→ 追溯评分（检查候选是否改善了过去任务的关键追溯指标）→ 影子 / 灰度（在受限环境中小范围试用）→ 操作员复核（决定是否升级）→ 正式升级（升级为生产能力）。

首版可以先实现候选生成 → 操作员复核 → 正式升级的简化路径，但架构应锁定完整的 6 阶段流水线作为终态。

### 8.6 与主循环的关系

```
任务运行主循环：
  事件 → ... → 归档(追溯记录) ──────┐
                                     │
                                     ▼
                              任务运行归档池
                                     │
                                     │ 调度器按三重门控触发
                                     ▼
                          ┌─────────────────────┐
                          │   整合运行           │
                          │   定向 → 采集 →     │
                          │   整合 → 裁剪       │
                          └─────────────────────┘
                                     │
                                     ▼
                              能力提议队列
                                     │
                                     │ 操作员复核
                                     ▼
                          记忆 / 技能 / 工作流 / 策略
```

具体阶段内部实现（定向 / 采集 / 整合 / 裁剪）由《记忆与演进》章节展开，本文档只锁定「它是独立运行实例、有严格权限边界、产出必须人工复核后才升级」。

## 9. 复杂度控制原则

为保证整体架构保持优雅且可落地，以下七项原则作为骨架纪律长期遵守。

### 9.1 不按业务场景建立常驻 Agent

「周界异常」「巡检」「多目标跟踪」「雷达研判」都是能力编排结果，**不是独立系统骨架**。业务场景的多样性应通过技能、工作流与工作单元的组合来应对，而非为每个场景创建独立的常驻 Agent。

### 9.2 不做两套治理体系

快任务、慢任务、长任务共享同一控制平面、同一状态治理、同一审计主链。无论任务认知深度如何，治理规则必须一致。

### 9.3 不让执行单元直接写全局主状态

工作流 / 技能 / 工作单元只能产生观测、中间结果、证据、建议；已提交状态只能通过统一提交链修改。这是防止状态混乱的根本防线。

### 9.4 不混淆四个正交维度

快速运行 / 深度审议运行 / 任务持续运行是认知深度维度；工作流 / 技能 / 工作单元是执行载体维度；观察 / 半自动 / 审批门控 / 人工控制是治理维度；用户 / 事件 / 定时 / 回调 / 策略 / 记忆触发是触发维度。任何新需求必须能落在这四个维度的组合里，不能在维度之间「长出新的一层」。

### 9.5 不把学习与演进压入首版关键路径

演进在整合运行中独立发生，且必须经过人工复核才升级。学习不能阻塞或污染实时任务。

### 9.6 不让模型可见上下文等同于系统运行状态

系统一等对象是真相，模型上下文是投影。真相以对象为准，投影按任务定制。两者混淆将导致模型行为不可控。

### 9.7 每个概念只承担一种主要职责

若某个概念开始承担多种职责（如「总控」同时是事件路由器、审批判决者、任务编排器、提交仲裁者、记忆管理者），必须拆分——但拆分必须落到已有平面，不创造新平面。

## 10. 本章锁定的架构约束

以下约束由本文档锁定。后续章节可以在这些约束内细化，**不允许突破**。突破必须回到本文档修订。

### 10.1 平面约束

系统恰有 **7 个运行平面**，不增不减。平面之间只通过一等运行对象交互，不通过直接调用。控制内核是唯一必须始终在线的治理中心。运行边界在智能规划之前生效。

### 10.2 对象约束

系统有 **10 类一等运行对象**，不在本文档之外新增。任何系统状态变更必须对应一个一等对象的状态迁移。已提交状态只能由控制内核通过提交审查写入。

### 10.3 模式维度约束

运行模式由四个正交维度的组合表达，不构造新的第五维度。快速运行升级到深度审议运行是单向的。任务持续运行不是独立主脑，是可重入的深度审议运行加上持久化长周期任务状态。

### 10.4 主循环约束

主循环恰有 **10 步**，顺序与平面归属不可更改。边界检查必须早于计划形成。验证必须早于提交。学习与演进不在主循环内。

### 10.5 横切约束

共有 **6 项横切治理能力**：验证 / 审批 / 仲裁 / 审计 / 上下文纪律 / 模型路由。横切能力不依附于任何单一平面，但每个平面都必须配合。模型路由是横切能力，不是审议与计划平面的内部细节。

### 10.6 演进约束

演进通过整合运行作为独立后处理运行实例承担。整合运行的产出必须经过操作员复核才能升级为生产能力。整合运行禁止写已提交状态、禁止直接注册生产技能。

### 10.7 合同约束

以下运行时合同与骨架约束同等重要，后续章节不可弱化：

**检查点 / 恢复合同**：任务持续运行的持久化执行（durable execution）是 P0 基础设施。检查点最小粒度、暂停/中断/恢复语义、审批挂起后恢复机制由第 7 章锁定，但本章锁定其优先级为 P0，不允许降级为 P1。

**边界规则合同**：每个关卡必须有明确的输入模型、输出模型、决策语义（允许/拒绝/询问/覆盖）、决策记录（写入审查记录 / 追溯记录）以及拒绝后恢复路径。由第 4 章锁定细节。

**上下文合同**：对象投影规则、上下文预算、工作单元交接、监控剥离由 §7.5 的合同大纲约束，由第 5 章（认知核）和第 11 章（审计）展开。

**提交合同**：已提交状态的写入权归控制内核独有；写入前必须经过验证和提交关卡；写入后必须触发发布、通知和审计。

**评估合同**：评估独立于验证，是系统持续改进的基础闭环，包括追溯评分、工具选择评估与持续回归测试。由第 11 章锁定。

**运行时预算 / 背压合同**：系统必须有截止预算（deadline budget）、队列深度限制、工作单元并发上限与审批积压策略。超出限制时触发降级或拒绝。由后续运行时章节展开，本章锁定其为一等约束。

## 11. 首版实现收缩指南

本章锁定的是总体架构终态，不等于首版必须一次性把所有对象、能力、治理细节都做到最丰满。为防止实现时走向"大爆炸式并行开发"，首版应遵守以下收缩策略。

### 11.1 收缩原则

首版的核心原则是"不删骨架，先窄后宽"：7 平面 / 10 对象 / 4 维度 / 10 步主循环保持不变，但允许对象先以窄字段、窄状态机落地，不允许把对象语义吞并进其他对象。实现顺序上，优先把准入、执行、提交三类关卡做硬，再提升深度审议能力。时间线上，先保证事件、执行、验证、提交、审计链闭合，学习与演进严格后置。

### 11.2 对象三级分档

对象分档的目的不是删对象，而是控制首版的实现深度。

| 分档 | 对象 | 首版要求 |
|---|---|---|
| **A级：首版完整一等对象** | 事件 / 运行实例 / 长周期任务 / 计划 / 审查记录 / 资产租约 | 独立持久化、明确状态机/生命周期、进入主控制链 |
| **B级：首版窄版一等对象** | 证据图 / 判断与假设集 / 已提交状态 | 必须独立存在，但字段、关系、领域范围可先收窄；重点先支撑核心态势闭环 |
| **C级：首版先做可写可追溯，后做丰满能力** | 追溯记录 | 首版必须追加式记录全链路，但回放 UI、诊断视图、聚合分析能力可后续增强 |

### 11.3 能力窄版策略

首版实现时，建议主动收窄以下能力：边界规则先使用声明式配置，不引入通用 DSL；已提交状态先覆盖少数高价值领域（如 `mission_status` / `risk_level` / `asset_operational`）；直接动作只允许白名单低风险动作，不扩张为第四类通用执行框架；工作单元并行度默认从 0 或极小值起步，待租约 / 审查 / 验证稳定后再扩；整合运行只生成记忆 / 技能 / 工作流候选与策略建议，不自动上线任何能力；检查点/恢复首版至少支持长周期任务级别的检查点 + 恢复以及审批挂起后的恢复，完整的回退/分支/回放可在后续版本补充。

### 11.4 门控优先级与实现先后顺序

首版不应平均推进所有能力，而应按以下顺序实现：

1. **准入关卡 + 运行实例 / 长周期任务基本生命周期 + 检查点/恢复基础语义**
2. **执行关卡 + 资产租约 + 审批/人工接管**
3. **验证 + 提交关卡 + 已提交状态窄版**
4. **追溯记录主链 + 审计/回放基础能力**
5. **整合运行候选提议链**

若上述顺序被打乱，最常见的结果是：智能能力先上线，治理链和审计链后补，最终系统行为不可解释、不可接管、不可回滚。

### 11.5 首版对控制内核的实现纪律

首版实现时必须显式按 §3.3 的五个子职责组划分代码边界。实现形式上允许一个进程内的多个模块、多个 actor / service 或同一存储上的不同权威层，但不允许用一个"总控类"同时承载所有职责，不允许让深度审议直接写已提交状态，不允许让审批、仲裁、提交审查散落到执行单元内部。

## 13. 架构结论

基于系统愿景、需求基线与运行模式，当前总体架构总结为：

> **一个单一控制平面下的分层自治系统：以运行边界约束执行边界；以控制内核承担最小可信治理；以审议与计划平面承担复杂理解与计划；以工作流 / 技能 / 工作单元作为执行载体；以观测事实 / 证据 / 判断 / 工作草稿 / 已提交状态作为单一真相链；通过验证 / 审批 / 仲裁 / 审计 / 上下文纪律 / 模型路由六项横切能力保证系统可控、可解释、可接管、可恢复；并以整合运行作为独立后处理运行对象承担能力演进，与实时关键路径完全分离。**

该架构同时满足覆盖首期真实业务闭环、控制系统复杂度不因场景增长而自然裂变、为后续状态模型/治理机制/任务对象/记忆/评估提供稳定骨架、让演进作为长期回路存在不干扰实时关键路径四项要求。因此，这一总体架构是当前阶段兼顾能力完整性、结构优雅性与工程可落地性的推荐方案。

### 13.1 优雅性测试

本架构是否成立，不只看概念数量，还看能否一眼回答以下问题：

1. 谁决定事件能进入系统 → 准入关卡（控制内核）
2. 谁决定动作能执行 → 执行关卡（运行边界）
3. 谁决定真相能提交 → 提交关卡（控制内核）
4. 长任务如何暂停与恢复 → 检查点/恢复合同（P0）
5. 人工如何接管而不破坏审计链 → 接管关卡 + 审查记录
6. 系统如何持续变好而不污染实时路径 → 改进回路（整合运行）
7. 推演预测如何被治理 → 判断与假设中的推演预测 + 验证

如果这些问题需要依赖额外口头解释，说明架构仍不够优雅。
