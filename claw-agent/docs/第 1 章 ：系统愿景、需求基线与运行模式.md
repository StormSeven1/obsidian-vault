# 态势感知 AI-native Agent 系统设计 · 第 1 章：系统愿景、需求基线与运行模式

## 1. 文档目的

本文档用于在总体架构设计之前，统一系统的愿景、需求边界、运行模式与复杂度控制原则。

本文档回答四个基础问题：

1. 我们到底要构建什么系统
2. 这个系统必须满足哪些核心需求
3. 这个系统应以哪些运行模式运转
4. 这些需求是否支持当前讨论出的总体架构，并保证其足够优雅、可控、可落地

本文档不讨论具体实现细节、接口字段和模块内部算法，而是作为后续总体架构、状态模型、任务对象、治理机制等章节的共同前提。

---

## 2. 系统愿景

本系统的愿景，不是构建一个会聊天的态势助手，而是构建一个**在受控边界内持续运行的人机协同态势闭环系统**。

系统不应只在用户提问时被动响应，而应能够围绕多源态势信息持续运行，主动感知环境、识别异常、组织证据、形成判断、提出或执行动作、监督执行过程，并在重复场景中沉淀稳定能力。

系统必须长期围绕以下闭环运转：

> **感知 → 发现 → 研判 → 计划 → 联动 → 监督 → 解释 → 复盘**

因此，系统的核心价值不是“回答了什么”，而是能否稳定完成面向现实世界任务的态势闭环。

---

## 3. 系统定位

### 3.1 本质定位

本系统应被定义为：

> **一个面向态势感知任务闭环的智能控制平面**

它不是纯对话系统，也不是若干业务 Agent 的集合，而是一个以统一治理、统一状态、统一审计为前提的长期运行系统。

### 3.2 驱动方式

系统必须是**事件驱动**的，而不是对话驱动的。

对话只是输入通道之一。系统还必须同时处理：

- 用户请求
- 外部告警与感知事件
- 定时触发
- 执行过程回调
- 审批结果
- 策略或记忆触发

### 3.3 自治定位

系统默认应运行在**半自治**模式下：

- 低风险动作可自动执行
- 中高风险动作必须进入审批或人工确认
- 任意执行过程都必须支持接管、停止、恢复、降级与回滚

系统未来可以按区域、资产、任务类型、风险级别逐步提升自治等级，但自治能力必须由系统治理机制控制，不能依赖模型临场判断。

### 3.4 建模原则

系统必须围绕**运行对象**建模，而不是围绕业务场景名称或 Agent 名称建模。

系统真正的一等对象应包括：

- Event
- Run
- Mission
- Evidence
- Claim
- Plan
- State
- Review
- Trace

“周界异常”“雷达研判”“巡检”“持续跟踪”等都应被视为这些对象在不同约束下形成的编排结果，而不是独立架构中心。

## 4. 需求基线

为了避免架构被愿景无限拉大，系统需求应分为 P0、P1、P2 三层。

### 4.1 P0：首版必须满足的需求

#### 4.1.1 多源世界接入

系统必须统一接入以下对象：

- 用户输入
- 数据库与知识库
- 雷达航迹与告警
- 视频流、截图与感知结果
- 无人机实时状态与任务状态
- GIS 图层、区域、地理约束
- 历史事件与任务日志
- 外部业务系统与文档资料

#### 4.1.2 可控资产统一纳管

系统必须统一管理至少以下可控资产与动作出口：

- WebGIS
- 相机云台与抓拍
- 雷达模式控制
- 无人机任务下发与回调
- 告警/通知系统
- 报表生成
- 外部业务系统 API

#### 4.1.3 统一事件闭环

无论是用户请求、外部事件、定时任务还是执行回调，都必须进入同一控制体系，由统一运行入口处理，不能演化成彼此独立的多套处理链路。

#### 4.1.4 快慢协同能力

系统必须同时支持：

- 低时延查询与低风险响应
- 复杂研判与多步分析
- 长生命周期任务的持续推进

这意味着系统必须具备不同认知深度下的运行模式，但不应因此裂变为多套独立系统。

#### 4.1.5 行动闭环能力

系统不能只停留在“给出建议”，还必须支持：

- 调用资产执行动作
- 监督执行状态
- 接收执行回执
- 根据执行结果继续推进任务

#### 4.1.6 人类治理能力

系统必须原生支持：

- 审批
- 拒绝
- 接管
- 停止
- 恢复
- 回滚

人不是系统外的观察者，而是运行时参与者。

#### 4.1.7 显式状态与证据管理

系统不能只依赖聊天上下文或零散日志维持真相，必须维护：

- 原始观测
- 证据关系
- 当前判断与候选假设
- 工作态与中间结果
- 已提交主状态

#### 4.1.8 并发与仲裁能力

系统应支持并发分析、并发查询、并发任务推进，但涉及关键资产、关键任务主状态、关键风险结论时，必须统一仲裁。

#### 4.1.9 审计与解释能力

系统必须支持全链路审计与结构化解释，至少能够说明：

- 为什么产生该结论
- 依据了哪些证据
- 触发了哪些动作
- 哪些动作经过了审批
- 当前任务推进到了哪里

---

### 4.2 P1：应预留但可分阶段落地的需求

- 长生命周期任务的 checkpoint / resume
- 受限并行 worker
- 动态 UI 与任务时间线
- 相似案例召回与结构化记忆装载
- 执行后的结果复核与独立 verifier
- 对任务过程进行阶段性解释与回放

---

### 4.3 P2：需要预留空间但不进入首版关键路径的需求

- workflow / skill 自动沉淀
- 策略优化建议
- 能力缺口发现与新增能力建议
- 更高等级的自治策略自适应
- 更复杂的跨任务长期学习机制

---

## 5. 非目标边界

为了保证系统保持优雅与可落地，以下内容在当前阶段不应作为首版硬目标：

1. 不追求完全无人值守的全域自治
2. 不按业务场景建立长期常驻 Agent 群
3. 不要求所有任务都进入开放式 agent loop
4. 不让学习与进化进入实时关键路径
5. 不让模型直接拥有高风险动作最终裁决权
6. 不让快路径绕过统一治理与状态提交

这些边界不是降低 ambition，而是防止架构在早期阶段失控。

---

## 6. 运行模式

系统的运行模式不应被理解为一组平铺的 lane 名称，而应被拆解为几组正交维度。

### 6.1 触发维度：系统为何被唤醒

系统可由以下方式触发：

- 用户触发
- 外部事件触发
- 定时触发
- 执行回调触发
- 策略 / 记忆触发

这意味着系统从设计上必须支持非对话输入持续进入同一运行框架。

### 6.2 认知深度维度：系统需要思考到什么程度

#### 6.2.1 Fast Run

适用于：

- 简单查询
- UI 操作
- 已缓存结果读取
- 白名单低风险动作
- 标准 workflow 的入口触发

特点：

- 不创建开放式 worker
- 不做重型自由规划
- 优先通过直接动作或标准 workflow 完成
- 若出现歧义、风险上升或执行失败，立即升级

#### 6.2.2 Deliberative Run

适用于：

- 多源证据整合
- 复杂目标研判
- 高风险动作前分析
- 多资产协同决策
- 需要补证、消歧、重规划的任务

特点：

- 需要上下文装载
- 需要计划形成
- 可调用 skill / workflow / worker
- 结果在提交前必须经过验证与审查

#### 6.2.3 Mission Continuation

适用于：

- 持续跟踪
- 长时监视
- 多阶段巡逻
- 伴随回调持续推进的任务
- 长生命周期目标管理

Mission Continuation 不是第三个独立主脑，而是：

> **Mission 状态 + 调度机制 + 可被事件反复唤醒的 Deliberative Run**

它解决的是持续性问题，而不是引入第二套系统中心。

### 6.3 治理维度：系统拥有哪些执行权

系统至少应支持以下治理模式：

- Observe / Advise：只观察与建议
- Semi-Auto：低风险动作自动执行
- Approval-Gated：需审批后继续
- Human-Controlled：人工接管，系统转为辅助

### 6.4 执行载体维度：系统依赖什么完成任务

系统中的执行载体应收敛为少数几类稳定对象：

#### 6.4.1 Direct Action / UI Executor

用于极低风险、低复杂度、低时延的直接动作。

#### 6.4.2 Workflow

用于：

- 高频
- 流程稳定
- 可代码化
- 确定性较高

#### 6.4.3 Skill

用于：

- 输入输出清晰
- 可复用
- 具备明确专门能力边界
- 不必长期持有全局状态

#### 6.4.4 Worker

用于：

- 一次性复杂分析
- 并行探索
- 复杂数据综合
- 长任务监督中的局部子任务

Worker 必须是受限的：

- 受上下文约束
- 受预算约束
- 受权限约束
- 不得直接提交全局主状态

## 7. 由需求反推的架构约束

如果上述愿景、需求基线与运行模式成立，那么总体架构必须满足以下约束。

### 7.1 必须存在单一控制平面

所有事件必须进入同一治理入口，所有关键状态变更必须通过同一提交流程完成。系统不能因为快任务、慢任务或不同场景而形成多套互不一致的治理中心。

### 7.2 必须将运行边界与智能编排分离

系统必须先定义：

- 什么可以做
- 什么必须审批
- 什么不允许做
- 什么条件下必须停止或降级

然后再讨论模型如何理解与规划。运行边界不能依赖模型 prompt 临场决定。

### 7.3 必须允许快路径，但快路径不能绕过治理

低时延响应是刚需，但快路径只能减少认知深度，不能绕过：

- 权限检查
- 审批门
- 状态提交规则
- 统一审计

### 7.4 必须显式维护证据与主状态

系统不能把“当前事实”“当前判断”“当前主状态”都混在聊天上下文中，而必须有独立的数据与治理对象。

### 7.5 Mission 必须是持久化业务对象，而不是第二主脑

Mission 的本质是任务持续性与重入能力，不应被设计成另一个平行系统中心。

### 7.6 执行载体必须少而稳定

workflow、skill、worker、direct action 已足以覆盖当前需求。不要继续扩张更多并列执行角色，否则会让系统骨架持续膨胀。

### 7.7 学习与进化必须脱离实时关键路径

能力演进很重要，但它应作为后处理和长期优化闭环存在，而不应反向控制首版运行时设计。

### 7.8 模型上下文与系统运行上下文必须分离

不是所有系统上下文都应该塞进模型窗口。系统必须保留本地运行状态、任务状态、审计状态与模型可见上下文之间的边界。

## 8. 对当前讨论架构的裁决

基于上述愿景、需求和运行模式，当前讨论后收敛出的总体架构是合理的，而且已经接近当前阶段的最佳实践。

它的合理性在于：

1. 它保留了统一控制平面和统一治理
2. 它承认快慢差异，但没有裂变成两套独立系统
3. 它将 workflow、skill、worker 视为执行载体，而不是彼此竞争的系统中心
4. 它将 Mission 正确放回到持久化任务对象的位置
5. 它要求显式状态、证据、审计与验证，不再把系统真相寄托在聊天上下文上

### 8.1 当前架构中应被保留的判断

- 系统是事件驱动，而不是纯对话驱动
- 必须存在统一控制平面
- 必须支持快慢协同
- 总体上应控制权收敛、智能性发散
- 执行单元应动态组合，而不是按业务场景固化 Agent
- 必须支持审批、接管、熔断、回滚、审计

### 8.2 当前架构中应进一步收敛的地方

为了让架构更优雅、更可落地，应进一步明确：

1. Fast / Deliberative / Mission Continuation 是**认知深度模式**，不是并列子系统
2. Workflow 是**执行载体**，不是独立运行模式
3. Mission 是**持久化任务对象**，不是第二主脑
4. 审批、权限、风险边界属于**治理维度**，不应与运行模式混合
5. worker 应被定义为**受限执行单元**，而不是泛化为系统内任意智能分身

### 8.3 为什么说这版架构是优雅的

它之所以优雅，不是因为概念最少，而是因为：

- 每个核心概念只承担一种主要职责
- 需求维度与架构维度能够一一对应
- 控制复杂度的方式是收敛抽象，而不是削减能力
- 为未来能力增长保留了空间，但没有把未来全部能力塞入首版关键路径

因此，这版架构是当前阶段**最小但完整**的方案：

- 足够覆盖首期真实业务闭环
- 足够支撑后续能力演进
- 不会因为业务场景增加而自然裂变为越来越多的并列 Agent

---

## 9. 本章结论

本系统的需求已经足以支持一个明确的总体设计方向。

这个方向不是：

- 为每个场景设计一个 Agent
- 为快慢任务分别设计一套系统
- 把学习、进化、长期自治全部压进首版运行时

而是：

> **以单一控制平面为中心，以统一治理、统一状态、统一审计为基础，以 Fast Run / Deliberative Run / Mission Continuation 作为认知深度模式，以 Workflow / Skill / Worker 作为执行载体，以显式证据、状态、审批和验证机制保证系统可控、可接管、可恢复、可演进。**

这一定义可以作为后续《总体架构》《运行时对象与状态模型》《治理与审批机制》《任务执行单元》《记忆与演进机制》等章节的共同基础。

---

## 10. 后续章节建议顺序

建议在本章之后，按以下顺序继续展开系统设计：

1. 总体架构
2. 运行时对象与状态模型
3. 治理、审批与执行边界
4. 执行载体：workflow / skill / worker
5. Mission 持久化与重入机制
6. 证据、验证与提交机制
7. 记忆与能力演进
8. 审计、解释与回放

本章到此为止。总体架构骨架、运行平面、一等运行对象、主循环与横切治理能力，统一在第 2 章《总体架构》中展开，以避免形成两套并行的架构真源。
