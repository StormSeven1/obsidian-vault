0. 文档定位

本文档定义态势感知 AI 智能体（Agent）系统的总体架构。它面向评审者回答三个问题：系统面对的核心设计挑战是什么，本架构为什么能够解决这些挑战，以及这套设计的优雅性、反应速度、扩展性和智能成长能力体现在哪里。
态势感知智能体与普通问答智能体的根本差异在于，它运行在连续变化的现实世界中。系统需要同时接收用户输入、传感器告警、地图变化、资产回调、历史研判结果和外部系统更新；需要在大量事件中判断什么值得关注；需要维护长期任务上下文；需要形成可追溯、可更新、可推演的态势理解；还需要在使用过程中持续沉淀经验，让能力随任务积累而增强。
因此，本文档只锁定总体架构主干、核心抽象和关键运行机制，不展开字段级 schema、接口签名、模型提示词、算法细节、完整状态机、提交协议字段和具体 UI 交互。与原版本相比，本修订版只在总体层补充四类生产化原则：降级恢复、未知高影响事件升级、经验资产治理和跨工作空间版本意识。更细的状态机、对象字段、提交仲裁、评估流程和工程实现细节，应在后续运行时架构章节展开。
需要特别说明的是，本章中的七个核心架构构件是长期稳定的架构角色，不等同于七个微服务、七个代码模块或七个独立智能体。工程实现时，它们会进一步落成事件、工作空间、工作空间变化、注意力决策、运行实例、能力调用、态势补丁、追溯片段和经验资产等运行时对象。总体架构回答“系统为什么这样组织”，运行时架构回答“系统如何真正跑起来”。

1. 架构命题：为什么需要任务态势智能操作系统

态势感知系统需要的不是一个能够回答问题的聊天入口，而是一个能够在真实任务环境中持续工作的智能操作系统。它必须连接外部世界，维护任务上下文，在事件流中分配注意力，调用合适能力完成理解与行动，并将每次运行沉淀为未来可复用的经验资产。

如果系统只围绕单次会话设计，它会缺乏任务连续性：用户追问、告警更新、资产回调和历史分析结果难以自然汇合。如果系统只围绕固定流程设计，它会缺乏敏捷性：简单事件被迫走重流程，复杂事件又难以动态补证和重规划。如果系统只围绕工具调用设计，它会缺乏真正智能：能力越多，选择越混乱，上下文越膨胀。如果系统只围绕数据库状态设计，它会缺乏态势理解：事实、证据、判断、预测和不确定性无法形成一个可推演的世界模型。

任务态势智能操作系统正是为解决这些问题而设计。“任务态势”强调系统围绕长期任务和实时态势持续运行；“活态”则保留在活态任务工作空间中，用于描述任务上下文会随着事件、证据、回调和用户意图不断更新。系统以活态任务工作空间承载长期上下文，以注意力循环处理连续事件流，以能力检索动态选择合适能力，以态势世界模型表达对象、关系、证据、判断和预测，以经验引擎把运行历史转化为系统的智能复利。

同时，任务态势智能操作系统必须把基础可靠性作为架构属性，而不是事后补丁。当算力资源紧张、模型不可用、外部能力异常或事件流暴涨时，系统应支持降级、延迟、取消、恢复或人工接管，而不是让任务长时间挂起。这里的可靠性是总体架构边界，不要求 V1 一次性实现完整容灾体系。

本架构的主线可以概括为：

感知世界
  → 维持任务工作空间
  → 分配注意力
  → 调用能力理解与行动
  → 更新态势世界模型
  → 从经验中持续进化

这条主线体现了本架构的核心取舍：治理能力、权限边界、降级恢复和审计机制必须存在，但它们作为系统可靠运行的横向支撑；总体架构的中心问题，仍然是如何让系统形成持续理解、快速反应、能力扩展和经验复利。

2. 设计哲学

2.1 工作空间优先，而不是会话优先

传统聊天式智能体以对话会话为中心。任务态势智能操作系统以任务工作空间为中心。一次对话、一次告警、一次工具调用、一次资产回调，都只是任务工作空间中的事件。真正长期存在的是任务目标、态势模型、证据包、运行历史、用户意图、可用能力和经验记忆。

这意味着系统的核心不是“回答用户问题”，而是“维护并推进一个长期任务语义场”。用户输入只是进入这个语义场的一种方式；传感器告警、资产回调、历史任务结果和外部系统更新，也同样会改变这个语义场。

2.2 注意力优先，而不是流程优先

态势感知系统面对的是连续、多源、异构的事件流。如果每个事件都进入完整推理流程，系统会变慢、变重，也会表现得不够聪明。真正智能的系统必须先判断哪些变化值得关注，哪些只是背景噪声，哪些只需要更新状态，哪些需要快速反应，哪些值得进入深度审议。

因此，本系统将注意力循环（Attention Loop）作为核心架构机制。它不是普通规则引擎，也不是新的重型智能体，而是系统的认知资源分配器。它让系统具备“重点感”和“敏锐感”。

注意力循环不能只奖励“已知模式”。低历史相似度不应自动等价于低重要性。对于高新颖性、高不确定性、高潜在影响或无法被现有模式解释的事件，系统应触发观察、补证、快速研判、深度审议或人工复核，而不是把它们默认为背景噪声。

2.3 能力检索优先，而不是工具罗列优先

能力越多，智能体不一定越聪明。如果所有技能、工具、资产适配器、剧本和历史经验都平铺给模型，系统会变得迟钝、混乱、难以预测。真正 AI 原生的能力系统必须支持按任务、区域、资产、风险、证据缺口和历史经验动态检索能力。

因此，计划器不直接面对全部能力，而是面对经过能力检索（Capability Retrieval）过滤和排序后的候选能力集。系统的扩展性不来自“能注册多少工具”，而来自“能否在合适时刻选择合适能力”。

2.4 态势模型优先，而不是数据库状态优先

系统状态不是一堆表。态势智能需要一个可查询、可投影、可推演的态势世界模型（Situation Model）。它不仅记录“发生了什么”，还要表达当前有哪些对象、对象之间有什么关系、哪些证据支持当前判断、哪些判断仍然不确定、哪些未来状态值得预警，以及还缺哪些关键证据。

态势记忆（Situation Memory）是存储层，态势世界模型是运行时联合视图。前者保存事实、证据、判断和历史，后者面向任务运行提供当前世界理解。

态势世界模型还需要具备版本意识。多个工作空间可以共享事实，但长时间运行和高风险判断不能无条件基于旧事实提交已确认态势。进入已确认态势前，系统应检查关键事实、证据和判断前提是否发生影响性变化；如已变化，应重新验证、重规划或进入人工仲裁。

2.5 经验复利优先，而不是简单归档优先

追溯记录（Trace）的价值不只是审计。每一次运行、失败、审批、修正、误报、漏报和成功处置，都是系统能力增长的原材料。经验引擎（Experience Engine）的职责不是把历史保存起来，而是把历史转化为可复用资产：情景记忆、目标画像、区域经验、技能候选、剧本候选、策略建议、评估样例和回归测试集。

智能不是一次性设计出来的，而是在长期运行中通过经验复利逐步形成的。

经验复利必须建立在经验资产治理之上。经验引擎产物默认是候选资产，不是可信能力。情景摘要、模式标签、能力路由信号、技能候选和剧本候选都应保留来源、评估状态和发布状态；未经评估或复核的经验资产不能直接改变生产能力或关键路由策略。

2.6 运行合同优先，而不是抽象叙事优先

总体架构需要保持清晰和优雅，但不能停留在概念叙事。每个核心构件都必须最终落成可测试、可追踪、可恢复的运行合同。例如，注意力循环不只是“判断重要性”，而应输出结构化注意力决策；能力系统不只是“注册工具”，而应提供能力 manifest、风险等级、审批策略和评估结果；态势世界模型不只是“状态集合”，而应通过态势补丁和提交机制控制哪些判断可以进入已确认态势。

因此，本架构在总体层保持七个稳定构件，在实现层要求关键决策以结构化对象表达，并进入追溯记录。架构的优雅不来自概念数量少，而来自核心边界稳定、状态所有权清晰、运行链路可回放、异常路径可恢复、经验资产可治理、态势提交可校验

3. 总体架构

任务态势智能操作系统由七个核心架构构件构成：任务网关（Mission Gateway）、活态任务工作空间（Living Mission Workspace）、注意力循环（Attention Loop）、智能体运行底座（Agent Harness）、能力系统（Capability System）、态势世界模型（Situation Model）和经验引擎（Experience Engine）。控制脊柱（Control Spine）作为横向机制贯穿全链路，但不作为主流程叙事中心。

[图片]
系统包含两条关系明确的链路：一条是实时任务链路，负责感知、理解、响应和更新态势；另一条是后台经验回路，负责从追溯记录和运行结果中沉淀经验资产，并将成熟能力回流到能力系统。
需要强调的是，下图是系统能力边界图，不是每个事件的默认串行执行链路。真实运行时必须坚持短链路优先：能在即时路径解决的事件，不进入快速研判路径；能在快速研判路径解决的问题，不进入深度审议路径；能异步沉淀的经验，不阻塞实时任务链路。
[图片]
这张图表达三个关键判断。第一，即时路径不默认进入完整智能体运行底座，它主要负责即时状态更新和轻量反馈。第二，经验引擎是后台经验回路，不阻塞实时任务链路。第三，态势世界模型是实时世界模型，态势记忆是其存储支撑，二者不能混为一个对象。

从生产运行角度看，还需要补充三个约束。第一，注意力循环既要识别常见高风险事件，也要识别无法解释的新颖事件。第二，实时链路必须允许降级运行，避免算力资源紧张或外部能力异常时任务长时间挂起。第三，跨工作空间共享事实时，需要具备版本意识，避免长时间运行基于过期事实提交已确认态势。

从工程角度看，这张图还隐含一个关键约束：实时链路中的每一次关键判断，都应生成可追踪的运行对象。例如事件标准化生成 Event，工作空间更新生成 WorkspaceDelta，注意力循环生成 AttentionDecision，能力调用生成 CapabilityInvocation，态势更新生成 SituationPatch，运行过程生成 TraceSpan。后续章节将把这些对象定义为运行时架构的核心合同。

3.1 运行时链路收缩原则
本架构不是长链路架构，而是短链路优先、按需升级的分层运行架构。七个核心构件定义系统能力边界，三条运行路径定义真实执行路径。
Hot Path:
事件 → 标准化 → 轻量注意力判断 → 事实更新 / UI 提示

Warm Path:
工作空间变化 → 注意力判断 → 少量能力检索 → 技能 / 轻量模型 → 候选判断

Deep Path:
任务触发 / 高风险事件 → 工作空间投影 → Agent Harness → 验证 → 态势补丁提交
控制脊柱、经验引擎、提交校验和弹性控制是横向或后台机制，不应被实现成所有请求的同步串行节点。只有高风险、复杂、冲突、长时间运行或未知高影响事件，才需要进入完整深度链路。

4. 七个核心架构构件

七个核心架构构件构成系统的稳定主干。任务网关负责让系统接入真实世界；活态任务工作空间负责承载任务语义场；注意力循环负责分配认知资源；智能体运行底座负责复杂理解、计划、行动和验证；能力系统负责能力生态；态势世界模型负责连续态势理解；经验引擎负责智能复利。

这些构件不是七个微服务，也不是七个必须独立部署的模块，而是七个长期稳定的架构角色。后续实现可以把它们拆成多个服务、模块、actor 或任务队列，但语义边界不应混淆。

控制脊柱不列为第八个原语。它是贯穿机制，负责权限、租约、审批、提交、审计、预算、背压、降级和恢复。它必须存在，但不应抢占 AI 原生主线。

为避免实现中出现概念滑移，本架构保留六条边界约束：

Workspace ≠ Memory
Workspace ≠ Model Context
Situation Model ≠ Database Tables
Attention Loop ≠ Planner
Control Spine ≠ 普通审计模块
Experience Engine ≠ 自动上线技能生成器

同时需要避免四类运行时误解：低相似度不代表低风险，经验资产不等于可信能力，运行快照不等于最新世界状态，降级运行不等于任务失败。更细的字段、状态和仲裁策略应进入后续运行时设计，而不是在总体架构中展开。

活态任务工作空间是任务视角，不是事实源。模型上下文是工作空间、态势模型、能力候选和历史经验的投影，不是工作空间本身。态势世界模型是当前世界理解，不是一组无语义状态表。注意力循环负责资源分配和路径选择，不负责完整任务计划。控制脊柱提供真实执行边界，不只是事后审计。经验引擎产出候选资产，不能绕过评估和发布流程直接改变生产能力。

5. 任务网关（Mission Gateway）

任务网关是系统接入真实世界的统一入口。它不是普通 API 网关，而是智能体与操作员、传感器、地图、资产、外部系统和任务表面的连接层。它的目标是让智能体进入真实工作流，而不是被关在一个单独的聊天框里。
[图片]
 从输入侧看，任务网关接收用户请求、雷达告警、视频异常、GIS 事件、数据库变更、定时触发、外部系统回调、资产执行回执和操作员审批结果。不同来源的输入在这里被标准化为事件，并带上来源、时间、空间范围、目标引用、资产引用、父运行引用和界面上下文等信息。

从输出侧看，任务网关负责把系统判断和动作映射到不同任务表面，例如地图态势层、告警卡片、审批卡片、任务时间线、证据面板、风险解释面板、报告视图和复盘视图。移动端提醒、语音播报、实时画布等形态可以作为后续扩展，而不作为当前架构的默认覆盖范围。任务表面（Mission Surfaces）是任务网关面向用户的呈现形态，不作为独立架构构件。

任务网关的核心要求是事件原生（event-native）和表面感知（surface-aware）。所有输入都应先进入事件体系；同一态势判断在不同表面需要不同表达方式：地图关注空间位置和轨迹，告警卡关注优先级和建议动作，审批卡关注后果和可选操作，复盘视图关注证据链与判断演化。

运行状态外显、降级提示、人工复核提示等 UI 细节应在任务表面设计章节展开。第 2 章只保留任务网关作为事件入口和任务表面连接层的角色。

6. 活态任务工作空间（Living Mission Workspace）

活态任务工作空间是智能体的认知工作台。它不是简单上下文缓存，也不是一次运行实例的状态对象，而是围绕任务持续存在的活态工作空间。系统的中心不是运行实例，而是工作空间；运行实例可以结束，工作空间持续存在。
[图片]
不过，工作空间不能被实现成一个巨大数据容器。它主要拥有任务目标、当前焦点、活跃假设、证据引用、运行引用、局部时间线、上下文投影配置和待处理动作。事件、实体、资产、能力、态势状态和历史记忆并不被复制进工作空间，而是通过引用和投影进入工作空间。换言之，工作空间是任务视角，不是事实源。

一个工作空间需要持续回答几个问题：当前任务正在朝什么目标推进？新事件对当前态势有什么影响？最重要的证据缺口是什么？哪些假设需要验证？哪些能力或资产可以帮助补证？用户当前关心什么？历史上类似情况如何处理？这些问题的持续维护，就是系统产生“智能感”的基础。

模型可见上下文也不能直接等同于工作空间。模型上下文来自工作空间的投影，不同任务、不同路径和不同能力调用拥有不同的投影规则。即时路径可能只需要一个轻量变化摘要，快速研判路径需要少量证据和候选技能，深度审议路径才需要完整的任务目标、关键证据、活跃假设、能力候选和历史案例。

工作空间运行队列、预算、失败状态、检查点引用等运行时可靠性信息，应由 Agent Harness、调度器或控制脊柱管理，不应把工作空间实现成运维状态容器。第 2 章仍强调工作空间的核心语义：它是任务视角，不是事实源，也不是运行时杂项容器。

7. 工作空间拓扑与路由

真实系统运行时通常会同时存在多个工作空间。它们不是彼此隔离的数据岛，而是围绕同一个全局态势世界形成的不同任务视角。比较合理的拓扑是：全局态势世界模型作为共同世界底座，任务工作空间承载长期任务，事件簇工作空间承载一次异常事件簇，交互上下文承载用户临时交互意图。
[图片]


这里有一个重要原则：用户输入不应默认创建新的工作空间。大多数用户输入只是交互上下文或交互线程，它绑定到当前地图选区、告警卡片、任务详情页、目标详情页或全局查询上下文。只有当用户发起一个独立分析任务，例如“分析过去 24 小时 A 区域所有异常并生成报告”，才可能创建短期分析型工作空间。

7.1 工作空间的区分边界

工作空间不按输入来源区分，而按任务语义边界区分。任务工作空间的边界来自长期任务目标，例如持续监控某区域或跟踪某个目标。事件簇工作空间的边界来自相关事件簇，例如同一区域在短时间内连续出现多源告警。交互上下文的边界来自用户当前操作位置和问题意图，它通常依附于已有工作空间或全局态势查询。

因此，一个事件进入系统时，首先要判断它是否已有明确父链路。如果工具结果、工作单元回调、资产回执或审批结果带有 parent_run_id、parent_action_id 或 callback_ref，它们直接回到原运行实例所属工作空间，不需要重新推理。如果用户输入来自具体界面，系统应携带 selected_region、selected_target、selected_alert、selected_mission、current_workspace_id、visible_map_extent 和 recent_run_id 等上下文，优先绑定当前交互对象。

没有显式绑定、也没有清晰交互上下文的独立事件，才进入相关性评分。评分依据包括任务是否匹配、目标是否重叠、空间是否重叠、时间是否接近、是否补充证据缺口、是否关联同一资产、是否命中当前用户焦点，以及事件类型是否与工作空间任务类型兼容。V1 不需要复杂模型，规则权重即可落地。

score(event, workspace) =
  task_score
+ ancestry_score
+ target_score
+ spatial_score
+ temporal_score
+ evidence_gap_score
+ asset_score
+ operator_focus_score
+ event_type_score

高分事件路由到对应工作空间；中分事件作为候选证据挂起；低分但重要的事件创建事件簇工作空间或候选事件；低分且不重要的事件只更新背景态势或进入未归属事件池。创建新工作空间是最后选择，不是默认选择。需要强调的是，低相关性不代表低重要性；当事件呈现高新颖性、高不确定性或高潜在影响时，即使无法匹配既有工作空间，也应进入候选观察、快速研判、深度审议或人工复核路径。

7.2 多事件并发处理

多个事件可能在同一时间窗口内到达。系统不应逐条触发模型推理，而应先形成工作空间变化批次。这个批次记录新增事件、变化对象、新证据、回调结果、用户意图、资产状态变化、受影响假设和证据缺口变化。注意力循环面对的是变化批次，而不是孤立事件。

这种处理方式可以避免重复推理，也能让用户输入、系统告警和历史回调自然合并。例如，用户在地图上询问“3 号区域刚才怎么回事？”，同时雷达告警进入 3 号区域，前一个轨迹分析工作单元也返回结果。系统不应把它们拆成三个无关运行实例，而应合并到同一个工作空间变化，再由注意力循环决定是更新当前研判、启动快速研判路径，还是升级到深度审议路径。

当多个工作空间同时产生变化批次时，注意力调度器统一调度。它根据风险等级、紧迫度、任务优先级、用户当前关注、关键证据回调、是否影响已确认态势、系统负载和已有活跃运行实例等因素，决定是只更新工作空间、合并到已有运行实例、启动即时/快速研判/深度审议路径，还是延迟处理或打断当前处理。

7.3 事实共享，理解局部

多个工作空间共享同一个世界，但拥有不同任务视角。事实应全局共享，上下文按工作空间投影，判断可以局部形成，已确认态势全局唯一，经验跨工作空间沉淀。

全局共享对象包括事件存储、观测事实、证据存储、实体注册表、资产状态、已确认态势、能力注册表、经验记忆和追溯记录。工作空间局部状态则包括任务目标、注意力焦点、活跃假设、证据引用、证据缺口、活跃运行实例、待处理动作、工作空间时间线、局部判断和上下文投影缓存。

这一区分非常关键。多个工作空间可以引用同一个事件或证据，但它们围绕不同目标形成不同解释。一个区域异常工作空间关心“这是否构成区域异常”，一个目标跟踪工作空间关心“这是否改变目标长期行为画像”。同一事件可以服务多个问题，但不能让多个工作空间对同一问题重复启动完整处理链。

事实全局共享需要配套版本意识。长时间运行和高风险判断应知道自己基于哪一版关键事实和证据形成判断；如果运行期间关键事实发生影响性变化，系统应重新验证、重规划或进入人工仲裁。该机制只在长运行、高风险判断和已确认态势提交时触发，不应成为所有事实更新的同步阻塞点。
7.4 多工作空间关联下的重复处理控制

一个事件可以关联多个工作空间，但不能让多个工作空间对同一事件、同一问题重复启动完整处理链。本系统采用单事件入库、多工作空间投影、按任务意图选择主处理方的模式。

事件存储中只有一份事件事实
        │
        ▼
工作空间路由器生成分发计划
        │
        ├── 主处理方：某个任务意图下的主工作空间
        ├── 相关方：接收事件引用或证据链接
        ├── 观察方：只观察状态变化
        └── 抑制方：不触发处理

事件只在全局事件存储中写入一次。多个工作空间通过 workspace_event_link 建立引用，关系可以是主处理方、相关方、观察方、候选相关或抑制处理。需要注意的是，“一个事件只能有一个主处理工作空间”这个说法不够严谨。更准确的约束是：同一个 event_id + task_intent 只能有一个主处理工作空间。同一个事件如果在不同任务意图下被处理，可以有不同主处理方，但它们处理的问题必须不同。
[图片]
例如，目标 T-17 出现在 A 区域异常事件中。对于 incident_assessment 这个任务意图，A 区域事件簇工作空间可以是主处理方；对于 target_profile_update 这个任务意图，T-17 目标工作空间可以是主处理方。二者共享事件和证据，但处理目标、上下文投影和输出判断类型不同，因此不是重复处理。

主处理方的选择遵循几个简单优先级：显式父链路优先，更具体的工作空间优先，已有活跃处理优先，当前用户焦点优先，风险与时效优先。任务工作空间作为父级工作空间时，通常接收事件簇工作空间的摘要和状态变化，不重复执行同一事件的研判链。

运行分发必须进一步通过 dedupe_key 防止重复运行。去重键应包含 event_cluster_id、owner_workspace_id、target_refs、spatial_scope、time_window 和 task_intent。如果已有相同问题的活跃运行实例，系统应合并进去，而不是新建运行实例；如果已有完成运行实例但新事件显著改变态势，则启动后续运行，而不是重复原始分析。

V1 的落地策略应保持务实：绝大多数事件靠显式引用和界面上下文绑定，少部分独立事件靠时空、目标、任务和证据缺口评分，无法判断的事件进入候选池或延迟归属。工作空间路由不需要一开始就完全智能，等追溯记录积累后，再由经验引擎学习更好的路由规则。
7.5 跨工作空间一致性与态势提交
多个工作空间共享事实时，系统必须避免长时间运行基于过期事实提交已确认态势。为此，长运行和高风险判断应绑定态势版本；进入已确认态势前，应检查关键事实、证据和判断前提是否发生影响性变化。
如果变化不影响当前判断，可以继续提交；如果变化影响核心假设，应重新验证或重规划；如果不同工作空间对同一已确认态势产生冲突，应进入人工仲裁。
这组机制只在高风险判断、长时间运行和已确认态势提交时触发，不应成为所有事实更新的同步阻塞点。具体快照对象、提交协议、冲突类型和仲裁策略在后续运行时架构章节展开。

8. 注意力循环（Attention Loop）

注意力循环是系统在事件流中分配认知资源的机制。它决定一个事件或一组工作空间变化是被忽略、合并、只更新状态、触发即时响应、进入快速研判、升级深度审议，还是提醒操作员关注。
[图片]
注意力循环的输入来自新事件、当前态势世界模型、工作空间当前焦点、活跃假设、证据缺口、任务优先级、区域风险、资产状态、历史相似案例、用户最近意图和系统当前负载。它的输出不是最终答案，而是路径选择、资源调度、强制升级、降级处理和人工提醒决策。

注意力循环默认应是轻量机制。V1 可以用规则、权重评分、索引查询和简单聚类实现；后续再引入轻量模型、向量检索、异常检测和历史模式。它不能一开始就变成另一个开放式深度推理智能体，否则会破坏系统的反应速度。

为了避免注意力循环成为无法评估的“神秘调度器”，它在运行时必须输出结构化的注意力决策（AttentionDecision）。注意力决策至少应包含路径选择、优先级、理由编码、去重键、合并策略、上下文预算、时间预算、能力标签、人类提醒级别和审计级别。换言之，注意力循环的输出应是调度合同，而不是一段自然语言解释。

对于未知高影响事件，注意力决策还应记录新颖性、不确定性、潜在影响和是否触发强制升级。具体评分字段、阈值和 reason code 应在注意力决策合同章节展开，不在总体架构中定义。

WorkspaceDelta
  → AttentionScorer
  → AttentionDecision
  → DispatchPlan

这种设计使注意力循环可以被测试、回放、调参和评估。系统可以事后回答：为什么这个事件被忽略，为什么这组变化被合并，为什么它进入深度审议，为什么没有提醒操作员，以及类似事件未来是否应该调整策略。

8.1 未知高影响事件强制升级
注意力循环不得仅以历史相似度或已知模式命中率决定事件重要性。当事件呈现高新颖性、高不确定性、高潜在影响、多源弱信号汇聚，或无法被已知背景模型解释时，即使相关性评分较低，也应进入观察、补证、快速研判、深度审议或人工复核路径。
该机制的目标不是让所有未知事件都进入深度审议，而是避免“找不到匹配模式”被误判为“低价值噪声”。V1 可以采用简单规则落地：未知但低影响的事件进入观察队列，未知且高潜在影响的事件进入快速研判或人工复核，只有复杂、高风险或证据冲突的事件才进入深度审议。

9. 三条运行路径

系统响应速度不能只依赖模型速度，而应成为架构属性。任务态势智能操作系统通过即时路径、快速研判路径和深度审议路径分层处理事件，使简单变化走轻路径，常见判断走快速路径，复杂问题才进入深度审议。
[图片]
即时路径（Hot Path）面向毫秒到秒级响应。它处理告警展示、态势快照更新、已知规则命中、UI 状态刷新、事件合并和用户当前关注区域的轻量提示。即时路径不做开放式推理，也不默认进入完整智能体运行底座。它的典型流程是：事件经过注意力循环后形成工作空间变化，直接更新态势世界模型或任务表面。

事件 → 注意力循环 → 工作空间变化 → 态势世界模型 / 任务表面

快速研判路径（Warm Path）面向秒级到十几秒响应。它处理简单证据合并、风险初判、轻量技能调用、历史相似案例检索和短摘要生成。快速研判路径以能力检索、技能、检索和轻量模型为主，不默认进入复杂规划。

事件 / 工作空间变化
  → 注意力循环
  → 能力检索
  → 技能 / 检索 / 轻量模型
  → 候选判断
  → 工作空间更新

深度审议路径（Deep Path）面向复杂态势、关键任务和高价值判断。它处理多源证据冲突、高风险动作前分析、多资产协同、长周期任务重规划、工作单元并行调查和未来态势预测。只有深度审议路径默认进入完整智能体运行底座。

事件 / 任务触发
  → 注意力循环
  → 加载活态任务工作空间
  → 能力检索
  → 上下文投影
  → 计划生成
  → 工作单元 / 技能 / 工具
  → 证据 / 判断
  → 验证
  → 态势世界模型更新
  → 经验捕获

路径可以升级，也可以提前终止。一个普通告警可以先走即时路径更新态势；如果短时间内同区域多次出现，则升级为快速研判路径；如果证据冲突或风险升高，则进入深度审议路径。相反，如果快速研判路径已经给出足够可靠的低风险判断，不应为了“显得智能”而进入深度审议。

路径也可以降级。当模型服务拥塞、外部工具不可用、能力不可用或任务预算不足时，系统应根据任务风险选择降级策略：即时路径继续更新事实，快速路径使用轻量能力或缓存态势，深度路径暂停、排队、缩小上下文、转入只读研判或交给人工复核。降级不等于失败，但降级结果必须与完整研判结果区分。

10. 智能体运行底座（Agent Harness）

智能体运行底座是系统执行复杂智能任务的运行基础设施。它不是万能大脑，也不是业务场景智能体，而是支持上下文装配、计划生成、能力调用、工作单元管理、验证、重试、沙箱和追踪的通用底座。
[图片]
智能体运行底座还应支持基础运行控制，例如超时、取消、失败恢复和必要的检查点。第 2 章只提出这些可靠性要求，不定义完整 RunState、AgentRun 字段或检查点 schema。

智能体运行底座主要服务深度审议路径，也为快速研判路径提供部分轻量能力。它内部包含上下文构建器、计划器、能力检索器、工具与技能执行器、工作单元管理器、验证器、重试与恢复管理器、沙箱和追溯记录发射器。这里的能力检索器是智能体运行底座在调用能力系统前执行的能力选择机制；它逻辑上属于运行时，数据上依赖能力系统、工作空间、态势世界模型和经验记忆。

上下文构建器从活态任务工作空间、态势世界模型、态势记忆和候选能力中生成模型可见上下文。它不能全量注入工作空间，也不能全量注入长期记忆。它必须根据不同运行路径和任务目标选择上下文预算，只装载与当前问题相关的证据、假设、能力说明和历史案例。

计划器负责生成结构化计划，例如调查计划、补证计划、风险分析计划、资产使用建议、多工作单元分析计划和报告生成计划。计划器面对的是检索后的候选能力，而不是全部系统能力。工作单元管理器则用于管理隔离子任务，例如并行分析不同区域、对不同假设分别补证、检索历史相似案例或生成目标画像。工作单元之间不共享完整上下文，只通过引用和结果交接。

验证器的职责不只是安全审查，也包括智能质量控制。它判断证据是否足够、是否存在反证、是否遗漏关键变量、计划是否可执行、预测是否有依据、报告是否与证据一致。沙箱则为实验性分析、候选技能测试、代码执行、仿真和非生产能力评估提供隔离环境，使系统可以探索而不污染真实态势。

完整运行实例状态机、检查点内容、恢复策略和回滚范围，应在运行时架构章节展开。第 2 章只保留原则：深度审议路径和高风险任务应可超时、可取消、可追踪，必要时可恢复，避免长任务因模型、工具或资源异常而无限挂起。

11. 能力系统（Capability System）与能力检索（Capability Retrieval）

能力系统是系统的能力生态。它让系统通过新增、组合和检索能力来扩展，而不是通过增加新的常驻智能体来扩张。系统支持的能力类型包括任务角色配置、技能、工具、资产适配器、工作单元、钩子、任务剧本和插件包。
[图片]
任务角色配置定义智能体在某类任务中的角色、风格、偏好和领域边界。技能是输入输出清晰、可复用、可评估的专业能力，例如目标轨迹分析、多源告警聚合、风险等级初判、历史相似案例检索和态势摘要生成。工具是原子操作能力，例如查询轨迹、查询地图图层、读取资产状态或创建任务卡。资产适配器连接真实资产或外部系统。工作单元承载隔离子任务。钩子是生命周期扩展点。任务剧本是稳定任务流程。插件包则是能力打包和分发单元。

能力系统的关键不在于能注册多少能力，而在于能否检索和装配合适能力。能力检索读取任务目标、当前态势、事件类型、区域、资产类型、证据缺口、风险等级、历史相似案例、用户偏好和当前路径类型，输出候选能力集。计划器和技能执行器只面对这个候选能力集。

能力选择应优先考虑任务相关性、历史效果、上下文成本、解释性和路径约束。快速路径限制能力数量，避免上下文膨胀；深度路径可以组合任务剧本、工作单元和多个技能；候选能力的使用效果会被追溯记录记录，并由经验引擎转化为后续的能力路由信号。

工程实现时，能力系统不应把完整能力说明一次性塞入模型上下文。更合理的方式是渐进加载：能力元数据常驻，命中后加载能力 manifest，调用前加载 schema、约束和少量示例，执行时再进入沙箱、工具或资产适配器。这种方式可以让能力规模增长，而不让上下文成本线性膨胀。

能力 manifest 除了描述输入输出、风险等级、审批策略、沙箱策略和评估结果，也应声明基本运行可靠性策略，例如超时、重试、降级和健康状态。能力检索不应只考虑相关性，也应考虑当前路径约束和能力可用性。具体 manifest 字段在能力系统章节展开。

12. 态势世界模型（Situation Model）

态势世界模型是系统对当前世界的结构化理解。它不是简单数据库状态，也不是模型上下文，而是可查询、可投影、可推演和可验证的运行时世界模型。
[图片]
态势世界模型至少表达地图图层、区域边界、关键对象、对象轨迹、对象关系、资产状态、任务状态、观测事实、证据关系、判断与假设、风险评估、未来预测、不确定性、证据缺口和已确认态势。它的核心视图可以概括为：

观测事实 → 证据 → 判断 / 预测 → 已确认态势

观测事实记录原始观测和标准化事实。证据记录证据项、来源、置信度、支持关系和反驳关系。判断 / 预测记录当前判断、候选假设、风险评估和未来预测。已确认态势记录已经被系统确认并可被外部依赖的态势状态。

真正智能的系统不只输出结论，还要知道自己不知道什么。因此，态势世界模型必须显式表达哪些判断置信度低、哪些证据互相冲突、哪些关键证据缺失、哪些预测需要继续观测，以及哪些区域或对象需要补充感知。

态势记忆是态势世界模型的存储支撑。它保存观测事实、证据、判断、预测、已确认态势和历史情景片段。态势世界模型则在运行时形成当前世界视图、活跃假设、不确定性、证据缺口和未来投影。

为了避免智能体直接污染真实态势，态势更新不应由模型直接写入最终状态。更合理的方式是所有技能、工具和智能体运行先产生态势补丁（SituationPatch），再经过验证、权限和提交机制，才进入已确认态势。候选判断、低置信预测和未验证证据可以进入工作空间或候选层，但不能被外部系统当作已确认事实依赖。

态势更新还应具备提交前校验原则。对于高风险判断、长时间运行或跨工作空间影响较大的判断，进入已确认态势前应检查关键事实和证据是否已经发生影响性变化。具体快照对象、补丁提交协议和冲突处理策略在态势模型运行时章节展开。

13. 经验引擎（Experience Engine）

经验引擎是系统的智能复利引擎。它从追溯记录、运行结果、失败案例、用户修正、任务后果、验证结果和评估结果中提取经验，将其转化为可复用的记忆、技能、剧本、策略建议、能力路由信号和评估资产。
[图片]
它产生的不是直接上线的能力，而是候选资产。典型产物包括情景片段、模式、目标画像、区域经验、技能候选、任务剧本候选、策略建议、评估样例、回归测试集和能力路由信号。这些资产需要经过离线评估、仿真或影子运行，并在必要时经过人工复核，才能进入生产能力系统。

经验引擎带来三类复利。认知复利让系统越来越理解某区域常见异常、某目标类型常见行为、某类告警常见误报原因和哪些证据最关键。能力复利让系统能把重复分析步骤沉淀为技能，把稳定处置路径沉淀为任务剧本，把高价值专项分析沉淀为工作单元模板。交互复利让系统逐步理解操作员更关心证据链、风险等级、下一步建议还是简洁摘要。

经验引擎不阻塞实时任务。实时路径只消费已经被验证、评估、复核并发布的能力资产。V1 不应实现完整自动技能生成，而应先实现“追溯记录 → 情景摘要 → 模式标签 → 评估样例候选”的最小路径。

为了保证长期安全性，经验资产应分级管理。原始追溯记录只读保存，情景摘要和模式标签可以进入检索，能力路由信号可以影响候选排序但需要监控，技能候选和剧本候选只能影子运行，经过评估和发布的能力才能进入生产调用。经验引擎的价值在于让系统越来越好，而不是让系统绕过治理自动改变自身行为。
13.1 经验资产治理
经验资产默认不可信，必须经过溯源、评估和发布流程，才能影响生产系统。风险不只来自自动生成技能，也来自错误情景摘要、错误模式标签、错误能力路由信号和错误目标画像。这些资产即使不直接执行动作，也可能影响检索排序、上下文投影、注意力评分和能力选择。
第 2 章只定义经验资产的基本生命周期：
原始追溯记录
  → 候选经验资产
  → 影子运行 / 评估资产
  → 已发布能力资产
  → 隔离 / 废弃资产
经验资产治理遵循三条原则：
1. 任何经验资产都应能追溯到来源运行、证据或人工修正。
2. 任何影响能力排序、上下文投影或生产调用的经验资产都应经过评估或复核。
3. 任何被证明错误的经验资产都应支持隔离、回滚和影响面追踪。
L0-L5 分级表、ExperienceAsset 字段、评估指标、隔离队列和周期性复测机制，应在经验引擎设计章节展开。

14. 任务事件生命周期与任务表面

系统不采用固定主循环作为唯一表达，而采用事件化生命周期。每个关键动作都发布生命周期事件，使注意力循环、能力系统、追溯记录、经验引擎和钩子能够订阅这些事件。
核心生命周期事件包括：事件接收、事件标准化、工作空间更新、注意力决策、路径选择、运行启动、能力调用、证据产生、判断更新、态势提交、运行完成和经验捕获。更细的提交前版本检查、冲突仲裁、经验评估和经验发布事件，应在运行时架构章节展开。
事件化生命周期带来四个价值。轻量事件可以只触发轻量处理，不必进入完整推理链；钩子、技能、任务剧本和经验引擎可以自然扩展系统行为；追溯记录可以形成清晰时间线；注意力循环和经验引擎可以从事件流中学习模式。
[图片]
任务表面是任务网关的呈现形态。当前核心任务表面包括地图态势层、告警卡片、审批卡片、任务时间线、证据面板、风险解释面板、报告视图和复盘视图。移动端提醒、语音播报、实时画布（Live Canvas）等形态可以作为后续扩展，而不作为当前架构的默认覆盖范围。同一态势判断在不同表面需要不同表达：地图强调位置、范围和轨迹；告警卡强调优先级、风险和建议动作；审批卡强调后果、证据和可选操作；任务时间线强调事件顺序和决策链；复盘视图强调判断演化和经验提取。
任务表面应能区分候选判断、已确认态势、降级判断和人工复核状态，避免操作员把候选结果误解为已确认态势。具体展示方式不在本章展开。

15. 控制脊柱（Control Spine）

控制脊柱是横向贯穿机制，不是架构主叙事中心。它确保系统在拥有强智能和高扩展性的同时，仍然保持可控、可审计、可接管、可恢复。
[图片]
控制脊柱包含权限、资产租约、审批、验证、提交、审计、预算、背压、降级和恢复。它的原则很简单：智能体可以生成候选判断，但不能直接把判断变成已确认态势；智能体可以调用能力，但关键副作用动作必须经过相应控制点；智能体可以学习经验，但新能力进入生产必须经过评估和发布流程；智能体可以在资源紧张时降级运行，但不能把降级结果伪装成完整研判。

控制脊柱不放在主流程中央，是因为本系统的核心设计挑战首先是态势理解、注意力分配、能力选择和经验复利。控制脊柱的作用是为这些智能能力提供可靠边界，使系统在长期运行中保持可控、可审计、可恢复，而不是替代智能形成过程本身。更细的接管、回滚、幂等和治理策略应在运行边界与治理章节展开。

需要强调的是，控制脊柱虽然不作为主叙事中心，但在实现上必须是一等公民。权限、审批、沙箱、预算、审计和提交机制不能只靠模型提示词保证，而应落实在能力调用、资产适配器、态势提交和外部副作用动作的执行边界上。每个能力都应声明风险等级、审批策略、沙箱策略和审计级别；每次关键调用都应写入追溯记录。
15.1 弹性控制
弹性控制负责避免系统在算力紧张、模型不可用、外部工具异常或事件流暴涨时长时间挂起。它不改变七个核心架构构件，而是作为控制脊柱的一部分贯穿能力调用、运行实例、任务队列和态势提交。
第 2 章只要求弹性控制至少覆盖以下原则：
- 超时：任务和能力调用不能无限等待。
- 重试：短暂失败可以重试，但必须有上限。
- 降级：复杂能力不可用时，应尽量保留轻量研判、只读查询或缓存态势。
- 排队：低优先级任务可以延迟，高优先级任务可以抢占。
- 人工接管：系统无法可靠判断时，应允许操作员接手。
- 恢复：异常结束的关键任务应保留可追溯状态，必要时可继续或重跑。
熔断键、健康探测、运行健康状态对象、队列策略、回滚策略和具体恢复状态机，应在控制脊柱或运行时可靠性章节展开。
15.2 人工接管
当系统无法可靠判断、关键能力不可用、证据不足、存在未知高影响事件，或外部副作用动作需要明确授权时，应允许人工接管。人工接管不是失败，而是控制脊柱的一种正常路径。
人工接管后，系统应保留上下文、证据链、候选判断、失败原因和建议动作，使操作员不是从零接手，而是在结构化状态上继续决策。

16. V1 收缩策略

V1 不追求完整任务态势智能操作系统，而是证明最小智能闭环成立：系统可以从多源事件进入，更新活态任务工作空间，经注意力循环判断路径，调用合适技能进行快速研判，更新态势世界模型，生成追溯记录，并把运行经验沉淀为后续可复用资产。
[图片]
V1 应优先实现任务网关的核心入口、活态任务工作空间的最小结构、工作空间路由器的显式绑定和相关性评分、注意力循环的简单重要性评分、即时路径的事件入库和态势更新、快速研判路径的 2 到 3 个核心技能、深度审议路径的最小深度审议、能力检索的基础规则、态势世界模型的观测事实 / 证据 / 判断 / 已确认态势窄版、追溯记录核心链路，以及经验引擎的情景摘要提取最小版。

V1 不应急于实现完整插件生态、自动技能生成、复杂工作单元并发、完整任务剧本引擎、复杂经验引擎、多通道全覆盖和复杂预测模型。V1 的成功标准也不应是“模型表现得多聪明”，而是系统是否能维护一个活态任务工作空间，是否能区分哪些事件需要注意，是否能走即时、快速研判和深度审议不同路径，是否能按需选择能力，是否能形成结构化态势模型，是否能从运行中生成可复用经验。

从验收角度看，V1 至少应证明五件事。第一，一个独立告警能够被路由到正确工作空间，或者进入候选事件池，而不是默认创建无关运行实例。第二，同一时间窗口内的用户输入、告警和回调能够合并为一次工作空间变化。第三，系统能够根据注意力评分选择即时路径、快速研判路径或深度审议路径。第四，快速研判路径能够通过能力检索选择合适技能，而不是把全部工具暴露给模型。第五，态势世界模型能够形成“观测事实 → 证据 → 判断 → 已确认态势”的最小闭环。
16.1 V1 最小可靠性边界
V1 不需要实现完整生产级可靠性体系，但必须证明最小可靠性边界成立：
1. Hot Path 不进入 Agent Harness。
2. Warm Path 只调用少量白名单能力。
3. Deep Path 至少支持 timeout、cancel 和 trace。
4. 候选判断与已确认态势分层。
5. 经验引擎只生成候选摘要，不直接影响生产能力。
6. 未知高影响事件可以进入观察队列、快速研判或人工复核。
完整 circuit breaker、snapshot version check、capability health score、experience quarantine、checkpoint/resume、cross-workspace conflict arbitration 可作为 V1.5 或后续运行时章节目标，不作为 V1 的默认验收范围。

17. 架构复杂度控制原则

任务态势智能操作系统的复杂度控制不靠删除智能能力，而靠稳定抽象边界。

新增业务场景不应催生新的常驻智能体，而应优先通过任务角色配置、技能、任务剧本和能力检索扩展。所有事件不应默认进入深度推理，注意力循环必须先做路径判断。能力系统不能变成工具垃圾场，新增能力必须有描述清晰的 manifest、适用场景、输入输出、评估方式和调用边界。

工作空间不能变成上下文大杂烩，它应保持为结构化任务视角，模型上下文必须由上下文投影生成。态势世界模型不能退化为状态表，它必须表达对象、关系、证据、判断、预测、不确定性和证据缺口。经验引擎不能变成归档任务，它必须产出可复用资产。控制脊柱不应抢占主叙事，它必须服务于智能形成过程，为系统的长期运行提供可靠边界。

运行时可靠性机制也不能膨胀成新的主架构。降级、恢复、快照、提交校验和经验隔离都应作为现有构件的运行合同存在，而不是引入新的常驻智能体或新的主流程。复杂度控制的原则是：保留短链路优先，必要时按需升级；增加边界约束，而不是增加新的“大脑”。

18. 架构优雅性测试

本架构是否足够优雅，可以用七个问题检验。

第一，系统是否有清晰中心？答案应是活态任务工作空间，而不是会话或单次运行实例。第二，系统如何避免变慢？答案应是注意力循环和三条运行路径分流。第三，系统如何扩展能力？答案应是能力系统和能力检索，而不是新增常驻智能体。第四，系统如何形成真正理解？答案应是态势世界模型对对象、关系、证据、判断、预测和不确定性的表达。

第五，系统如何越用越强？答案应是经验引擎将追溯记录转化为记忆、技能、剧本、评估资产和能力路由信号。第六，系统如何进入真实工作流？答案应是任务网关和任务表面把智能体嵌入地图、告警、任务时间线、移动端和外部系统。第七，系统如何保持简洁？答案应是七个核心架构构件稳定不变，不为每个业务场景发明新主脑。

如果这些问题需要大量额外解释，说明架构表达仍然不够清晰。

还可以增加一组工程可落地性测试。第一，每个关键判断是否有结构化对象承载？第二，每个状态写入是否有明确所有者？第三，每个能力调用是否能被追溯和评估？第四，每个外部副作用是否有控制点？第五，每次运行失败后是否能恢复、重试或回放？如果这些问题答不上来，说明架构还停留在理念层，需要进入运行时设计补齐。

本轮修订后，可增加三项简化可靠性测试。第一，当核心能力不可用时，系统是否能降级、延迟或转人工，而不是无限等待？第二，当事件无法匹配历史模式时，系统是否能识别新颖性和潜在影响，而不是默认忽略？第三，当错误经验进入候选资产时，系统是否能隔离、回滚或至少停止其继续影响生产能力？

19. 架构结论

本系统的最终目标不是构建一个更复杂的智能体平台，而是构建一个真正 AI 原生的态势任务智能操作系统。

它的核心主干可以概括为七个核心架构构件：任务网关、活态任务工作空间、注意力循环、智能体运行底座、能力系统、态势世界模型和经验引擎。它的反应速度来自三条运行路径。它的扩展能力来自能力检索。它的长期智能来自“追溯记录 → 情景片段 → 模式 → 能力候选”的经验复利。它的可靠性由控制脊柱横向保障，但主叙事不再是治理，而是智能。

这套架构让系统具备六个关键特征：反应快，上下文稳，能力强，理解深，会进化，可恢复。反应快来自注意力循环和路径分层；上下文稳来自活态任务工作空间；能力强来自能力系统和能力检索；理解深来自态势世界模型；会进化来自经验引擎；可恢复来自控制脊柱、短链路优先、降级恢复和经验治理。

因此，本架构适合作为态势感知 AI 智能体系统的总体架构基线。
