# 第 3 章：运行时架构与 V1 落地设计

## 任务态势智能操作系统的工程化运行蓝图

---

## 0. 文档定位

第 2 章定义了任务态势智能操作系统的总体架构基线，回答系统为什么需要以任务工作空间、注意力循环、能力检索、态势世界模型和经验引擎为核心。本章承接该架构基线，进一步回答工程落地问题：系统运行时有哪些核心对象，事件如何进入和流转，工作空间如何路由和去重，注意力决策如何表达，三条运行路径如何实现，能力系统如何被检索和调用，态势世界模型如何安全更新，V1 应该怎样收缩并验收。

本章不是重新设计一套架构，而是把第 2 章中的架构角色落成可实现、可测试、可追踪的运行时合同。它的目标不是一次性穷尽所有工程细节，而是建立一套足够稳定的工程骨架，使后续模块、数据模型、服务拆分、接口定义和任务实施都能在同一套语义下展开。

本章遵循三个原则。第一，先定义运行对象，再定义服务模块。对象是系统语义的稳定核心，服务可以随着部署形态变化。第二，先定义状态流转，再定义模型提示词。模型只是某些路径中的执行组件，不能成为系统唯一的流程表达。第三，先完成 V1 最小闭环，再扩展复杂能力。V1 的目标是证明架构成立，而不是提前实现完整操作系统。

---

## 1. 总体运行时视图

从运行时看，任务态势智能操作系统可以分成八个工程层次：入口层、任务层、调度层、执行层、能力层、态势层、经验层和控制层。它们分别对应第 2 章中的核心构件，但不要求一一对应为独立服务。

```text
入口层：Mission Gateway
  负责外部输入接入、事件标准化、任务表面输出和外部系统回调。

任务层：Workspace Runtime
  负责工作空间创建、绑定、更新、投影和工作空间变化批次生成。

调度层：Attention Runtime
  负责变化评分、路径选择、去重、合并、优先级和调度合同生成。

执行层：Agent Harness / Skill Runtime
  负责上下文装配、计划、技能调用、工具执行、工作单元、验证和恢复。

能力层：Capability Registry / Capability Retrieval
  负责能力注册、能力描述、能力索引、能力检索、风险策略和版本管理。

态势层：Situation Model / Situation Memory
  负责观测事实、证据、判断、已确认态势、不确定性和证据缺口的存储与投影。

经验层：Trace / Experience Pipeline
  负责追溯记录、运行总结、情景片段、模式标签、评估样例和能力路由信号。

控制层：Control Spine
  负责权限、审批、预算、背压、沙箱、资产租约、提交和审计。
```

这八层形成一个事件驱动的运行时主线：外部输入先变成标准化事件，事件被路由到工作空间并形成工作空间变化，注意力循环基于变化输出注意力决策，调度器根据决策选择即时路径、快速研判路径或深度审议路径，执行结果以态势补丁的形式进入验证和提交，最终更新态势世界模型，并把过程写入追溯记录，供经验引擎离线沉淀。

```text
Raw Input
  → Event
  → Workspace Routing
  → WorkspaceDelta
  → AttentionDecision
  → DispatchPlan
  → Hot / Warm / Deep Path
  → CapabilityInvocation / WorkUnit
  → SituationPatch
  → Validation
  → Situation Commit
  → Trace
  → ExperienceArtifact
```

运行时架构的核心不是“模型循环”，而是这些对象之间的状态转移。模型可以参与能力选择、计划生成、证据分析和摘要生成，但系统的可控性来自结构化对象、明确写入者、可回放状态和控制脊柱。

## 2. 核心运行时对象

本系统的关键运行对象包括 Event、Workspace、WorkspaceEventLink、WorkspaceDelta、AttentionDecision、DispatchPlan、Run、WorkUnit、CapabilityManifest、CapabilityInvocation、SituationPatch、TraceSpan 和 ExperienceArtifact。它们构成系统的工程语义骨架。

| 对象 | 作用 | 创建者 | 消费者 |
|---|---|---|---|
| Event | 标准化外部输入或内部回调 | Mission Gateway | Workspace Router / Attention Runtime |
| Workspace | 长期任务语义场 | Workspace Runtime | Attention Runtime / Agent Harness / UI |
| WorkspaceEventLink | 事件与工作空间的关系 | Workspace Router | Workspace Runtime / Dedupe |
| WorkspaceDelta | 一个时间窗口内的工作空间变化批次 | Workspace Runtime | Attention Runtime |
| AttentionDecision | 路径选择与资源调度合同 | Attention Runtime | Dispatcher / Control Spine |
| DispatchPlan | 具体调度计划 | Dispatcher | Hot / Warm / Deep Path Runtime |
| Run | 一次智能运行实例 | Agent Harness | Workspace / Trace / Experience Pipeline |
| WorkUnit | 深度路径中的隔离子任务 | Agent Harness | Skill Runtime / Tool Runtime |
| CapabilityManifest | 能力描述、约束与治理信息 | Capability Registry | Capability Retrieval / Agent Harness |
| CapabilityInvocation | 一次能力调用记录 | Agent Harness / Skill Runtime | Trace / Situation Model |
| SituationPatch | 对态势世界模型的候选更新 | Skill / Agent / Validator | Situation Model / Control Spine |
| TraceSpan | 可追溯运行片段 | 全链路组件 | Experience Pipeline / Audit |
| ExperienceArtifact | 经验候选资产 | Experience Engine | Capability System / Eval Pipeline |

这些对象中，最重要的是 Event、WorkspaceDelta、AttentionDecision 和 SituationPatch。它们分别对应“发生了什么”“工作空间发生了什么变化”“系统准备如何处理”“态势世界应该如何更新”。只要这四类对象设计清楚，系统就不会退化为不可观测的黑盒 agent。

### 2.1 Event

Event 是系统接收外部世界和内部回调的统一事实入口。用户输入、传感器告警、地图事件、资产回执、工具结果、审批结果、定时任务和外部系统更新都应先标准化为 Event。

```text
Event {
  event_id
  event_type
  source_type
  source_ref
  occurred_at
  received_at
  spatial_scope
  target_refs
  asset_refs
  payload
  parent_run_id?
  parent_action_id?
  callback_ref?
  ui_context?
  confidence?
  raw_ref
}
```

Event 只表达输入事实，不直接表达系统判断。它可以有置信度和来源质量，但不应包含已经被智能体生成的最终结论。判断应在证据、候选判断或态势补丁中表达。

### 2.2 Workspace

Workspace 是长期任务语义场，不是事实数据库，也不是模型上下文。它持有任务目标、当前焦点、活跃假设、证据引用、运行引用、待处理动作和上下文投影配置。

```text
Workspace {
  workspace_id
  workspace_type
  mission_id?
  task_intents
  goals
  focus
  active_hypotheses
  evidence_refs
  evidence_gaps
  active_run_refs
  pending_actions
  local_timeline_refs
  projection_profile
  owner_context
  status
}
```

Workspace 的核心价值是维持任务连续性。运行实例可以结束，工作空间不能因为一次运行结束就丢失任务语义。工作空间中的事实引用应指向全局事件存储、证据存储、态势记忆和追溯记录，而不是复制事实。

### 2.3 WorkspaceEventLink

WorkspaceEventLink 表达一个事件与一个工作空间之间的关系。它是多工作空间系统避免重复处理的关键对象。

```text
WorkspaceEventLink {
  link_id
  event_id
  workspace_id
  task_intent
  relation_type = owner | related | observer | candidate | suppressed
  reason_codes
  score
  created_at
}
```

同一个事件可以链接多个工作空间，但在同一个 `event_id + task_intent` 下只能有一个 owner。其他工作空间只能作为 related、observer、candidate 或 suppressed 关系存在。这个约束保证事实可以共享，但同一问题不会被多个工作空间重复完整处理。

### 2.4 WorkspaceDelta

WorkspaceDelta 是一个工作空间在短时间窗口内的变化批次。注意力循环面对的是 WorkspaceDelta，而不是孤立 Event。

```text
WorkspaceDelta {
  delta_id
  workspace_id
  time_window
  event_refs
  new_observation_refs
  changed_entity_refs
  changed_asset_refs
  updated_hypotheses
  evidence_gap_changes
  user_intent?
  callback_refs
  prior_run_refs
  summary
}
```

WorkspaceDelta 的价值在于合并并发变化。例如同一区域短时间内出现多个告警、用户同时发起询问、上一轮分析刚返回结果，这些不应触发多个孤立推理，而应合成一个变化批次，由注意力循环统一判断。

### 2.5 AttentionDecision

AttentionDecision 是注意力循环的结构化输出。它不是自然语言解释，而是调度合同。

```text
AttentionDecision {
  decision_id
  workspace_id
  delta_id
  path = hot | warm | deep | update_only | delay | suppress
  priority
  reason_codes
  dedupe_key
  merge_target_run_id?
  capability_tags
  context_budget
  time_budget
  escalation_condition?
  human_notify_level
  audit_level
  created_at
}
```

AttentionDecision 让系统可以被测试和回放。后续可以统计不同 reason_codes 的准确性、不同 path 的成本收益、不同 dedupe_key 的合并效果，以及哪些决策经常被操作员修正。

### 2.6 DispatchPlan

DispatchPlan 把注意力决策转换为具体执行计划。它决定走哪条运行路径、调用哪些运行时组件、是否需要排队、是否合并到已有运行、是否触发审批或通知。

```text
DispatchPlan {
  dispatch_id
  decision_id
  path
  run_policy
  queue_policy
  merge_policy
  selected_capability_refs
  required_controls
  deadline
  budget
}
```

DispatchPlan 不负责推理，它负责把调度合同落到具体执行环境。

### 2.7 Run 与 WorkUnit

Run 是一次智能运行实例，通常由 Warm Path 或 Deep Path 创建。Hot Path 可以不创建完整 Run，只写入轻量 TraceSpan。Run 记录一次运行的目标、上下文投影、能力调用、计划、结果、验证和状态。

```text
Run {
  run_id
  workspace_id
  parent_run_id?
  task_intent
  path
  objective
  input_delta_id
  context_projection_ref
  plan_ref?
  capability_invocation_refs
  work_unit_refs
  status
  result_ref?
  trace_ref
}
```

WorkUnit 是 Deep Path 中的隔离子任务，用于并行补证、分别验证假设、调用专项技能或处理不同对象。WorkUnit 应共享引用和结果，不共享完整上下文。

```text
WorkUnit {
  work_unit_id
  run_id
  objective
  scoped_context_ref
  capability_refs
  input_refs
  output_refs
  status
  error?
}
```

### 2.8 CapabilityManifest 与 CapabilityInvocation

CapabilityManifest 描述能力本身，CapabilityInvocation 记录一次能力调用。两者分离后，系统既能治理能力，也能评估能力效果。

```text
CapabilityManifest {
  capability_id
  name
  capability_type
  description
  input_schema
  output_schema
  applicable_event_types
  applicable_task_intents
  required_context
  risk_level
  approval_policy
  sandbox_policy
  evaluation_method
  version
  owner
  status
}
```

```text
CapabilityInvocation {
  invocation_id
  run_id
  capability_id
  input_refs
  output_refs
  started_at
  finished_at
  status
  cost
  error?
  trace_ref
}
```

CapabilityManifest 面向能力检索和治理，CapabilityInvocation 面向追溯、调试、评估和经验沉淀。

### 2.9 SituationPatch

SituationPatch 是态势世界模型的候选更新。技能、工具和智能体运行不直接修改已确认态势，而是生成补丁，经过验证和提交后才进入态势世界模型。

```text
SituationPatch {
  patch_id
  source_run_id?
  source_invocation_id?
  target_entities
  observations_added
  evidence_added
  judgments_proposed
  predictions_proposed
  confidence_delta
  uncertainty_delta
  evidence_gaps_added
  evidence_gaps_resolved
  requires_validation
  commit_status = proposed | validated | rejected | committed
}
```

SituationPatch 使态势更新可审计、可回滚、可比较。候选判断和已确认态势必须分层，否则系统很容易把模型推断误当成事实。

### 2.10 TraceSpan 与 ExperienceArtifact

TraceSpan 是运行过程中的可追溯片段。每次事件接收、路由、注意力决策、能力检索、能力调用、验证和提交都应产生 TraceSpan。

```text
TraceSpan {
  span_id
  trace_id
  parent_span_id?
  span_type
  actor
  input_refs
  output_refs
  started_at
  finished_at
  status
  reason_codes
  cost
  error?
}
```

ExperienceArtifact 是经验引擎从追溯记录中提取出的候选资产。

```text
ExperienceArtifact {
  artifact_id
  artifact_type = case_fragment | pattern_label | eval_sample | routing_signal | skill_candidate | playbook_candidate
  source_trace_refs
  summary
  applicability
  confidence
  validation_status
  promotion_status
}
```

ExperienceArtifact 默认不能直接进入生产能力系统，必须经过评估、影子运行或人工复核。

## 3. 事件进入系统后的生命周期

系统采用事件化生命周期，而不是固定大循环。每个输入先标准化为 Event，再进入工作空间路由、注意力决策、路径执行、态势补丁和经验沉淀。

```text
1. Raw Input 接入
2. Event 标准化
3. Workspace Routing
4. WorkspaceEventLink 建立
5. WorkspaceDelta 生成
6. AttentionDecision 输出
7. DispatchPlan 生成
8. Hot / Warm / Deep Path 执行
9. CapabilityInvocation / WorkUnit 产生结果
10. SituationPatch 生成
11. Validation 与 Control Spine 检查
12. Situation Model Commit
13. Trace 完整记录
14. Experience Pipeline 离线消费
```

### 3.1 Raw Input 到 Event

Mission Gateway 负责把所有输入变成统一 Event。输入来源可以是用户、传感器、地图、外部系统、资产回调、工具结果或审批结果。标准化时必须保留原始引用 raw_ref，确保后续可以追溯原始来源。

### 3.2 Event 到 WorkspaceDelta

Workspace Router 根据显式父链路、界面上下文、目标引用、空间范围、任务意图和证据缺口，确定事件与工作空间的关系。路由结果不应直接触发推理，而是先建立 WorkspaceEventLink，并在短时间窗口内生成 WorkspaceDelta。

### 3.3 WorkspaceDelta 到 AttentionDecision

Attention Runtime 读取 WorkspaceDelta、当前态势世界模型、工作空间焦点、任务优先级、风险等级、系统负载和历史相似案例，输出 AttentionDecision。这个决策决定后续路径，而不是直接决定最终答案。

### 3.4 AttentionDecision 到 DispatchPlan

Dispatcher 根据 AttentionDecision 生成 DispatchPlan。它负责把 path、priority、dedupe_key、merge_policy、budget、required_controls 等字段落到执行队列、运行实例或合并策略中。

### 3.5 执行路径到 SituationPatch

Hot Path 通常直接更新轻量状态或生成低风险 SituationPatch。Warm Path 通过能力检索和轻量技能生成候选判断。Deep Path 通过上下文投影、计划、工作单元和多能力调用生成更完整的证据、判断和建议。所有态势更新都应以 SituationPatch 的形式提交。

### 3.6 SituationPatch 到 Situation Model

SituationPatch 经过验证和控制脊柱检查后，才能进入态势世界模型。低风险观测事实可以自动提交，高风险判断、外部副作用或已确认态势更新应经过更严格的验证或人工审批。

### 3.7 Trace 到 ExperienceArtifact

所有关键动作都进入 Trace。经验引擎异步消费 Trace，生成情景片段、模式标签、评估样例、能力路由信号、技能候选和剧本候选。实时链路不等待经验引擎完成。

## 4. 工作空间路由与重复处理控制

工作空间路由的目标不是给每个事件创建新空间，而是把事件挂到最合适的任务语义场中。默认原则是：显式父链路优先，界面上下文优先，已有工作空间优先，创建新工作空间最后。

### 4.1 路由优先级

```text
1. parent_run_id / parent_action_id / callback_ref 明确存在
   → 直接回到原运行实例所属工作空间。

2. ui_context 明确绑定 selected_region / selected_target / selected_alert / selected_mission
   → 优先绑定当前交互对象对应工作空间。

3. event.target_refs 或 spatial_scope 与已有工作空间高度重叠
   → 按相关性评分选择候选工作空间。

4. 事件具有独立高风险或新事件簇特征
   → 创建事件簇工作空间。

5. 事件低相关且低风险
   → 只更新背景态势或进入候选事件池。
```

### 4.2 相关性评分

V1 可以用规则权重完成路由评分。

```text
score(event, workspace) =
  task_score
+ ancestry_score
+ target_score
+ spatial_score
+ temporal_score
+ evidence_gap_score
+ asset_score
+ operator_focus_score
+ event_type_score
```

每个分项都应产生 reason_code，不能只输出一个分数。reason_code 是后续调试、评估和经验学习的基础。

### 4.3 主处理方约束

系统必须保证同一问题不会被多个工作空间重复完整处理。核心约束是：

```text
同一个 event_id + task_intent 只能有一个 owner workspace。
```

一个事件可以在不同 task_intent 下有不同 owner。例如同一目标出现异常时，区域事件簇工作空间可以负责 incident_assessment，目标工作空间可以负责 target_profile_update。二者共享事实，但处理的问题不同。

### 4.4 去重键

运行调度时必须使用 dedupe_key 防止重复运行。

```text
dedupe_key = hash(
  event_cluster_id,
  owner_workspace_id,
  task_intent,
  target_refs,
  spatial_scope,
  time_window
)
```

如果已有相同 dedupe_key 的活跃 Run，新的 WorkspaceDelta 应合并到该 Run 或作为补充输入，而不是新建 Run。如果已有完成 Run，但新事件显著改变风险、证据或结论，则启动 follow-up Run，而不是重复原始分析。

### 4.5 合并策略

合并策略至少包括三类：

```text
merge_into_active_run
  合并到已有活跃运行。

append_as_followup
  作为已有运行的后续变化。

start_new_run
  启动新的运行实例。
```

注意力决策应明确 merge_policy，避免调度器自行猜测。

## 5. 注意力决策合同

注意力循环的核心输出是 AttentionDecision。这个对象连接工作空间变化和运行路径，是系统能否快速、稳定、可解释的关键。

### 5.1 输入

Attention Runtime 的输入包括：

```text
WorkspaceDelta
Workspace 当前焦点
Situation Model 当前态势
任务优先级
区域风险等级
目标重要性
证据缺口
历史相似案例
系统负载
已有活跃 Run
控制脊柱预算
```

### 5.2 输出

AttentionDecision 至少包含以下字段：

```text
AttentionDecision {
  decision_id
  workspace_id
  delta_id
  path
  priority
  reason_codes
  dedupe_key
  merge_target_run_id?
  capability_tags
  context_budget
  time_budget
  escalation_condition?
  human_notify_level
  audit_level
}
```

其中 path 的取值包括：

```text
hot
  即时路径，只做状态更新、轻量提示或低风险补丁。

warm
  快速研判路径，调用少量技能、检索和轻量模型。

deep
  深度审议路径，进入完整 Agent Harness。

update_only
  只更新工作空间或态势事实，不触发智能运行。

delay
  延迟处理，等待更多事件或系统资源。

suppress
  抑制处理，只保留追溯记录。
```

### 5.3 评分维度

V1 注意力评分可以采用规则评分。

```text
attention_score =
  novelty_score
+ risk_score
+ urgency_score
+ uncertainty_score
+ relevance_score
+ evidence_value_score
+ operator_focus_score
- system_load_penalty
- duplicate_penalty
```

评分结果不应直接等同于路径。路径选择还要结合规则阈值和控制策略。例如低风险但用户正在关注的变化可以走 Hot Path 提示，高风险且证据冲突的变化应走 Deep Path，低相关但高风险事件可以创建事件簇工作空间并进入 Warm Path。

### 5.4 升级与提前终止

AttentionDecision 可以定义 escalation_condition。例如：

```text
如果 5 分钟内同一区域同类告警超过阈值，Warm Path 升级为 Deep Path。
如果关键证据回调与当前判断冲突，重新触发 AttentionDecision。
如果 Warm Path 已得到高置信低风险判断，则不进入 Deep Path。
```

这种机制避免系统在简单问题上浪费资源，也避免复杂态势停留在轻路径。

## 6. 三条运行路径的工程实现

三条路径不是三个概念，而是三个不同成本、不同延迟、不同控制强度的运行时模式。

### 6.1 Hot Path：即时路径

Hot Path 面向毫秒到秒级响应。它不默认调用大模型，不默认创建完整 Run，不默认调用外部资产。

输入：

```text
Event
WorkspaceDelta
AttentionDecision(path=hot 或 update_only)
```

处理：

```text
事件入库
工作空间轻量更新
态势快照刷新
UI 提示或告警卡更新
低风险 SituationPatch 生成
TraceSpan 记录
```

输出：

```text
Workspace 更新
轻量 SituationPatch
任务表面更新
TraceSpan
```

Hot Path 的典型场景是已知规则命中、UI 状态刷新、低风险事实更新、告警展示、重复事件合并和背景态势刷新。

### 6.2 Warm Path：快速研判路径

Warm Path 面向秒级到十几秒响应。它可以调用少量技能、检索和轻量模型，但不进入复杂规划。

输入：

```text
WorkspaceDelta
AttentionDecision(path=warm)
Capability tags
Context budget
Time budget
```

处理：

```text
能力检索
上下文轻量投影
调用 1-3 个技能或检索能力
生成候选判断
形成证据摘要
生成 SituationPatch
写入 Run 和 Trace
```

输出：

```text
候选判断
证据摘要
SituationPatch
Run Result
Trace
```

Warm Path 的典型场景是风险初判、多源告警聚合、短摘要、相似案例检索、常见误报判断和轻量补证。

### 6.3 Deep Path：深度审议路径

Deep Path 面向复杂态势、关键任务和高价值判断。它进入完整 Agent Harness，可以规划、拆分工作单元、调用多种能力、执行验证和生成报告。

输入：

```text
Workspace
WorkspaceDelta
AttentionDecision(path=deep)
Candidate capabilities
Situation Model projection
Historical cases
Control constraints
```

处理：

```text
上下文投影
计划生成
工作单元拆分
能力调用
工具 / 资产适配器调用
证据整合
反证检查
验证
SituationPatch 生成
审批或提交
经验捕获
```

输出：

```text
结构化计划
证据包
候选判断 / 预测
建议动作
SituationPatch
Run Result
完整 Trace
经验候选资产
```

Deep Path 的典型场景是多源证据冲突、高风险动作前分析、多资产协同、长期任务重规划、复杂报告生成、未来态势预测和关键事件复盘。

### 6.4 路径能力边界

| 能力边界 | Hot Path | Warm Path | Deep Path |
|---|---|---|---|
| 大模型调用 | 默认不允许 | 可用轻量模型 | 可用完整模型 |
| 能力调用 | 低风险固定能力 | 少量检索后技能 | 多能力组合 |
| 工作单元 | 不创建 | 通常不创建 | 可创建 |
| 外部资产 | 默认不允许 | 只读或低风险 | 需控制脊柱约束 |
| 态势写入 | 低风险事实 | 候选判断 | 完整补丁 + 验证 |
| 审批 | 通常不需要 | 视风险而定 | 高风险必须检查 |
| Trace | 轻量 | 标准 | 完整 |

## 7. 能力系统落地

能力系统的工程目标是让系统能按需发现、选择、调用、评估和治理能力。能力不是简单工具列表，而是带有描述、输入输出、风险、适用场景、评估方式和版本的生产资产。

### 7.1 能力类型

V1 支持的能力类型可以先收缩为五类：

```text
skill
  输入输出清晰的专业分析能力。

tool
  原子操作能力，例如查询、读取、创建任务卡。

retriever
  检索能力，例如查历史案例、证据、地图图层。

asset_adapter
  外部资产或系统适配器。

playbook
  稳定任务流程模板，V1 可先只定义不完整实现。
```

Subagent、复杂插件包、自动生成技能、复杂钩子系统可以后置。

### 7.2 CapabilityManifest

每个能力必须有 manifest。

```text
CapabilityManifest {
  capability_id
  name
  description
  capability_type
  input_schema
  output_schema
  applicable_event_types
  applicable_task_intents
  applicable_regions?
  applicable_asset_types?
  required_context
  context_cost
  latency_class
  risk_level
  approval_policy
  sandbox_policy
  evaluation_method
  historical_success_rate?
  version
  owner
  status = draft | shadow | production | deprecated
}
```

### 7.3 能力检索流程

能力检索不应只靠文本相似度，而应结合任务、事件、态势和治理信息。

```text
Task Intent
  → Event Type
  → Workspace Goals
  → Situation Context
  → Evidence Gaps
  → Risk Level
  → Path Type
  → Historical Effectiveness
  → Candidate Capabilities
```

检索结果应包含排序理由：

```text
CapabilityCandidate {
  capability_id
  rank
  match_reasons
  expected_context_cost
  expected_latency
  risk_level
  required_controls
}
```

### 7.4 渐进加载

能力说明应分层加载，避免上下文膨胀。

```text
第一层：能力 metadata 常驻索引
第二层：命中后加载 manifest
第三层：调用前加载 schema、约束和少量示例
第四层：执行时进入技能、工具、沙箱或资产适配器
```

Warm Path 通常只加载少量 manifest 和 schema。Deep Path 可以加载更完整的能力说明、历史案例和任务剧本。

### 7.5 能力评估与降级

每次 CapabilityInvocation 都应记录效果。能力系统至少跟踪：

```text
调用成功率
平均延迟
平均成本
输出被接受率
输出被修正率
验证失败率
人工否决率
适用场景分布
```

效果差的能力应被降权、进入影子模式或要求重新评估。能力系统不能只支持新增能力，也必须支持降级、禁用、回滚和版本锁定。

## 8. 态势世界模型的最小实现

V1 不应试图一次实现完整态势世界模型。最小闭环只需要六类对象：Observation、Evidence、Judgment、ConfirmedSituation、Uncertainty 和 EvidenceGap。

### 8.1 V1 态势对象

```text
Observation
  标准化观测事实。来自事件、传感器、工具结果或人工输入。

Evidence
  可支持或反驳判断的证据项，包含来源、置信度和关系。

Judgment
  系统或人工形成的候选判断，例如风险初判、异常类型、原因解释。

ConfirmedSituation
  经过验证或审批后可被外部依赖的已确认态势。

Uncertainty
  对当前判断不确定性的结构化表达。

EvidenceGap
  当前判断仍缺少的关键证据。
```

### 8.2 态势更新原则

态势更新必须遵循四条原则：

```text
事实与判断分离。
候选判断与已确认态势分离。
模型推断与外部观测分离。
补丁提交与最终状态分离。
```

系统可以快速产生候选判断，但不能把候选判断直接变成已确认态势。

### 8.3 SituationPatch 提交流程

```text
Skill / Agent / Tool Output
  → SituationPatch(proposed)
  → Validator
  → Control Spine
  → SituationPatch(validated or rejected)
  → Situation Model Commit
  → Trace
```

验证器检查：

```text
证据是否存在
证据是否支持判断
是否存在反证
置信度是否合理
是否影响已确认态势
是否需要人工审批
是否符合权限与预算
```

### 8.4 态势投影

不同运行路径和任务表面不应直接读取全量态势世界模型，而应读取投影。

```text
Hot Path Projection
  当前事件、已知对象、低风险状态。

Warm Path Projection
  相关事件、关键证据、少量历史案例、证据缺口。

Deep Path Projection
  任务目标、对象关系、证据链、活跃假设、历史案例、能力候选。

Mission Surface Projection
  面向地图、告警卡、审批卡、报告和复盘的不同视图。
```

投影机制是控制上下文成本和用户呈现质量的关键。

## 9. Trace、评估与经验闭环

Trace 是经验引擎的原料，也是系统可审计、可调试、可评估的基础。没有高质量 Trace，经验引擎就只能做模糊归档，无法形成真正复利。

### 9.1 Trace 记录范围

V1 至少记录以下事件：

```text
Event 接收与标准化
Workspace 路由结果
WorkspaceDelta 生成
AttentionDecision 输出
DispatchPlan 生成
Capability Retrieval 结果
CapabilityInvocation 输入输出引用
Run 状态变化
SituationPatch 生成与提交
Validation 结果
人工修正或审批结果
ExperienceArtifact 生成
```

Trace 不一定保存所有原始内容，但必须保存足够引用，确保后续可以回放。

### 9.2 经验产物流

经验引擎的最小链路是：

```text
Trace
  → Run Summary
  → Case Fragment
  → Pattern Label
  → Eval Sample Candidate
  → Capability Routing Signal
```

更高级的链路是：

```text
Pattern Label
  → Skill Candidate
  → Shadow Evaluation
  → Human Review
  → Production Capability
```

V1 只要求完成前一条链路，不要求自动生成生产技能。

### 9.3 经验资产分级

| 等级 | 产物 | 线上使用方式 |
|---|---|---|
| L0 | 原始 Trace | 只读、审计、回放 |
| L1 | 情景摘要 / 模式标签 | 可检索，不直接决策 |
| L2 | 能力路由信号 | 可影响排序，需要监控 |
| L3 | 技能候选 / 剧本候选 | 只能影子运行 |
| L4 | 已评估技能 / 已发布剧本 | 可生产调用 |
| L5 | 关键任务能力 | 需要审批、版本锁定和回滚 |

### 9.4 评估样例

每次有价值的运行都可能成为评估样例候选。评估样例应包含：

```text
输入事件和工作空间状态
期望路径选择
期望能力选择
关键证据
正确判断或人工修正结果
不应触发的错误行为
成本与时延约束
```

这样系统才能逐步建立回归测试集，而不是靠人工主观感受判断智能是否变好。

## 10. 控制脊柱的运行时落点

控制脊柱不是一个事后审计模块，而是一组贯穿能力调用、态势提交、外部动作和经验发布的执行边界。

### 10.1 控制点

V1 至少需要以下控制点：

```text
能力调用前控制
  检查权限、风险等级、审批策略、预算和沙箱要求。

外部资产调用前控制
  检查资产租约、操作权限、环境约束和回滚策略。

SituationPatch 提交前控制
  检查是否影响已确认态势，是否需要验证或审批。

经验资产发布前控制
  检查评估结果、版本、责任人和回滚方案。
```

### 10.2 风险等级

能力和动作可以先分成五类：

```text
read
  只读查询，低风险。

write_internal
  写入内部任务对象或工作空间。

write_situation
  修改态势世界模型。

external_side_effect
  影响外部系统或资产。

destructive_or_irreversible
  破坏性或不可逆动作。
```

不同风险等级对应不同审批和审计要求。

### 10.3 审批策略

```text
none
  无需审批。

auto
  系统规则自动批准。

operator
  操作员审批。

supervisor
  更高权限审批。

blocked
  当前环境禁止执行。
```

审批策略应写在 CapabilityManifest 和 ActionPolicy 中，而不是藏在模型提示词里。

---

## 11. V1 最小闭环

V1 的目标不是完整实现 Mission Agent OS，而是证明架构主线成立。建议 V1 选择一个高价值、边界清晰的场景：区域异常告警进入系统后，系统能路由、合并、研判、更新态势、生成追溯和沉淀经验。

### 11.1 V1 主场景

```text
一个区域告警进入
  → 标准化为 Event
  → 路由到区域任务工作空间
  → 与用户当前关注区域和历史回调合并为 WorkspaceDelta
  → Attention Loop 选择 Warm Path
  → Capability Retrieval 选择 2-3 个技能
  → 技能生成候选判断和证据摘要
  → 生成 SituationPatch
  → 验证后更新态势世界模型
  → 任务表面展示告警解释和建议
  → Trace 记录全链路
  → Experience Engine 生成情景摘要和模式标签
```

### 11.2 V1 必做能力

V1 至少实现：

```text
Event 标准化
Workspace 创建与绑定
WorkspaceEventLink
WorkspaceDelta 生成
AttentionDecision 规则评分
Hot Path 状态更新
Warm Path 快速研判
基础 Capability Registry
基础 Capability Retrieval
2-3 个核心 Skill
SituationPatch
V1 态势世界模型
TraceSpan
Run Summary
Case Fragment / Pattern Label
```

### 11.3 V1 暂不做能力

V1 暂不实现：

```text
完整插件生态
自动技能生成
复杂任务剧本引擎
复杂多工作单元并发
全自动经验进化
复杂预测模型
多通道全覆盖
复杂权限矩阵
跨组织能力市场
```

这些能力可以设计接口预留，但不能成为 V1 交付前置条件。

### 11.4 V1 推荐技能

V1 可选 2-3 个核心技能：

```text
多源告警聚合技能
  将同一区域、同一时间窗口、相关目标的多条告警聚合为事件簇。

风险等级初判技能
  根据事件类型、区域风险、目标重要性和历史模式给出初步风险等级。

历史相似案例检索技能
  检索过去类似态势，返回处置结果、误报原因和关键证据。
```

这三个技能可以覆盖 Warm Path 的最小价值闭环。

## 12. V1 验收标准

V1 验收不应只看模型回答是否自然，而应看系统闭环是否成立。

### 12.1 功能验收

V1 至少证明：

```text
1. 独立告警能够标准化为 Event。
2. Event 能够被路由到正确 Workspace 或候选事件池。
3. 同一时间窗口内的用户输入、告警和回调能够合并为 WorkspaceDelta。
4. AttentionDecision 能够选择 hot、warm、deep、update_only、delay 或 suppress。
5. Warm Path 能够通过 Capability Retrieval 选择合适技能。
6. 技能输出能够形成 SituationPatch。
7. SituationPatch 能够经过验证并更新态势世界模型。
8. 任务表面能够展示判断、证据和不确定性。
9. Trace 能够记录关键过程。
10. Experience Engine 能够生成情景摘要或模式标签。
```

### 12.2 架构验收

V1 还应证明：

```text
Workspace 没有变成事实数据库。
Situation Model 没有退化为普通状态表。
Attention Loop 没有变成完整 Planner。
Agent Harness 没有吞掉所有路径。
Capability System 没有变成工具垃圾场。
Experience Engine 没有绕过评估直接发布能力。
Control Spine 没有停留在事后审计。
```

### 12.3 性能与成本验收

V1 应定义基础指标：

```text
Hot Path 平均延迟
Warm Path 平均延迟
Deep Path 平均耗时
单事件平均能力调用数
上下文平均 token 成本
重复运行率
错误路由率
人工修正率
SituationPatch 接受率
能力调用成功率
```

这些指标不需要一开始达到完美，但必须被记录。否则系统无法判断自己是否在变好。

## 13. 推荐工程分层

服务拆分可以根据团队规模和部署环境调整，但语义上建议保留以下模块边界。

```text
mission-gateway
  接入外部输入，标准化事件，输出任务表面更新。

workspace-runtime
  管理 Workspace、WorkspaceEventLink、WorkspaceDelta。

attention-runtime
  计算注意力评分，生成 AttentionDecision。

dispatcher
  根据 AttentionDecision 生成 DispatchPlan 并触发路径执行。

agent-harness
  处理 Deep Path 的上下文投影、计划、工作单元和验证。

skill-runtime
  处理 Warm Path 和 Deep Path 中的技能调用。

capability-registry
  管理 CapabilityManifest、版本、状态和评估信息。

capability-retriever
  根据任务和态势检索候选能力。

situation-runtime
  管理 SituationPatch、态势提交和态势投影。

trace-runtime
  收集 TraceSpan，提供回放和审计。

experience-pipeline
  异步消费 Trace，生成经验候选资产。

control-spine
  提供权限、审批、预算、沙箱和审计策略。
```

这些模块可以合并部署。例如 V1 可以把 workspace-runtime、attention-runtime 和 dispatcher 放在同一个服务中，把 capability-registry 和 capability-retriever 放在同一个服务中，把 trace-runtime 和 experience-pipeline 先做成轻量异步任务。但语义边界应保持清晰。

---

## 14. V1 数据存储建议

V1 不需要复杂基础设施，但需要避免把所有状态塞进一个通用表。建议至少区分以下存储域：

```text
事件存储
  Event、Raw Ref、事件索引。

工作空间存储
  Workspace、WorkspaceEventLink、WorkspaceDelta。

态势存储
  Observation、Evidence、Judgment、ConfirmedSituation、EvidenceGap、SituationPatch。

能力存储
  CapabilityManifest、CapabilityInvocation、能力评估信息。

运行存储
  Run、WorkUnit、DispatchPlan、AttentionDecision。

追溯存储
  Trace、TraceSpan、输入输出引用。

经验存储
  ExperienceArtifact、模式标签、评估样例候选。
```

其中 Event、Trace 和 SituationPatch 是最重要的可回放对象。只要它们设计正确，后续即使服务拆分调整，系统仍然可以恢复和分析历史运行。

## 15. 失败恢复与幂等

Agent 系统在真实环境中一定会遇到超时、工具失败、重复回调、模型错误、审批中断和外部系统不一致。因此 V1 就应建立基础恢复机制。

### 15.1 幂等键

至少这些对象需要幂等控制：

```text
Event：source_ref + occurred_at + event_type
WorkspaceEventLink：event_id + workspace_id + task_intent
AttentionDecision：delta_id + dedupe_key
Run：dedupe_key + path + task_intent
CapabilityInvocation：run_id + capability_id + input_hash
SituationPatch：source_run_id + target_entities + patch_type
```

### 15.2 失败处理

失败处理分为：

```text
retryable
  临时错误，可重试。

recoverable
  可降级、可切换能力、可等待回调。

requires_human
  需要人工判断或审批。

terminal
  不可恢复，记录失败并进入经验样例。
```

每次失败都应进入 Trace，并被经验引擎消费。失败不是噪声，而是系统改进的重要素材。

## 16. 本章结论

第 2 章定义了任务态势智能操作系统的总体架构，第 3 章将它落成运行时工程蓝图。两章之间的关系是：第 2 章负责稳定抽象，第 3 章负责运行合同。

本章最核心的结论有五个。

第一，系统运行的基本单位不是模型消息，而是 Event、WorkspaceDelta、AttentionDecision、Run、CapabilityInvocation、SituationPatch 和 TraceSpan。第二，注意力循环必须输出结构化调度合同，不能停留在自然语言判断。第三，能力系统必须以 manifest、检索、风险、评估和版本为核心，不能退化为工具列表。第四，态势世界模型必须通过 SituationPatch 安全更新，不能让模型直接污染已确认态势。第五，经验引擎必须从 Trace 中提取候选资产，并经过评估和发布流程后再回流能力系统。

V1 的成功标准不是把所有能力做全，而是证明最小智能闭环成立：事件可以进入，工作空间可以维护，注意力可以分流，能力可以按需选择，态势可以结构化更新，过程可以追溯，经验可以沉淀。只要这个闭环成立，系统就可以在真实使用中逐步扩展能力，而不是靠一次性设计赌完整未来。
