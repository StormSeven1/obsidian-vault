## 1）总架构图：一张图讲清“单一控制平面”

这张最重要，适合放在介绍开头。

### 提示词

clean enterprise architecture infographic, premium AI system design poster, vector style, ultra clear layout, white background, layered boxes, thin glowing blue-gray connection lines, highly readable labels, elegant technical composition  
  
Draw a high-level system architecture diagram for a situational awareness AI-native agent system.  
The visual core must be a SINGLE CONTROL PLANE, not multiple parallel agents.  
Center the diagram around "Control Kernel" as the unique governance authority.  
Around it, show the seven stable operating planes:  
1) World Interface Plane at the bottom, connected to external world, sensors, GIS, radar, video, database, operators, assets  
2) Runtime Boundary Plane above World Interface, showing five gates: Admission Gate, Execution Gate, Asset Control Gate, Commit Gate, Override Gate  
3) Deliberation and Planning Plane on the left of the Control Kernel  
4) Execution Plane on the right of the Control Kernel  
5) State and Knowledge Plane below as the truth foundation, containing Runtime State and Memory  
6) Operator Surface and Observability Plane above as human supervision, approval, takeover, replay, evaluation  
7) Consolidation Run as a separate background evolution loop, visually detached from the real-time path  
  
The diagram must emphasize:  
- one governance center  
- boundary before action  
- execution cannot directly write committed truth  
- state and knowledge as the system truth base  
- human operator in the loop  
- learning loop completely separated from real-time critical path  
  
Use clean arrows and explicit layer separation.  
Show a short subtitle: "Single Control Plane under Runtime Boundary".  
Use minimal but readable English labels.  
professional, elegant, presentation-grade

### 这张图要表达的意思

文档明确要求系统是 **一个单一控制平面**，而不是多个并列主脑；同时，运行边界必须早于智能规划生效，Consolidation Run 必须与实时关键路径分离

## 2）功能组成图：突出七平面各自职责

这张图比“总架构图”更偏组件说明。

### 提示词

clean enterprise architecture functional composition diagram, premium vector infographic, dark navy background, cyan and white labels, balanced modular composition, modern defense-tech interface aesthetic  
  
Create a functional composition diagram for a situational awareness AI-native control system.  
Show seven major planes as large functional modules with distinct responsibilities:  
  
- World Interface Plane: protocol adapters, data normalization, CEP, command receipts  
- Runtime Boundary Plane: permissions, policy, asset scope, risk constraints, approval stops, rollback preconditions  
- Control Kernel: run admission, lifecycle, scheduling, arbitration, commit authority, degradation control  
- Deliberation and Planning Plane: context assembly, evidence organization, plan formation, complex reasoning, mission replanning  
- Execution Plane: workflow, skill, worker, direct action  
- State and Knowledge Plane: observed facts, evidence graph, claim set, working state, committed state, plus episodic / semantic / instruction / capability memory  
- Operator Surface and Observability Plane: approval, takeover, replay, trace, diagnostics, evaluation  
- Separate Consolidation Run loop: archive analysis, memory distillation, skill proposal, workflow proposal, policy suggestion  
  
Use a clean modular grid, with each block containing 3 to 5 concise sub-capabilities.  
Make it look like a serious architecture poster used in an executive product review.  
Do not make it look like a software class diagram.  
Emphasize functional boundaries and disciplined separation of responsibilities.

### 这张图为什么值得画

第 2 章最强的一点，就是“**每个平面只承担一种主要职责**”，并且要求后续章节都落回这 7 个平面里，不能随意长出新层

## 3）运行流程图：10 步主循环

这张图最适合帮助用户理解系统“是怎么跑起来的”。

### 提示词

professional process flowchart infographic, enterprise AI runtime loop, vector diagram, clean white background, strong directional arrows, minimal color accents, highly structured layout  
  
Draw a 10-step runtime loop for a situational awareness AI-native system.  
The flow must be linear and disciplined, not chaotic.  
Show the following exact sequence:  
  
1. Event Intake  
2. Boundary Check  
3. Run Decision  
4. Context Assembly  
5. Plan Formation  
6. Execution  
7. Verification  
8. Commit and Publish  
9. Monitor and Re-enter  
10. Archive  
  
Visually emphasize these architecture laws:  
- Boundary Check happens BEFORE Plan Formation  
- Verification happens BEFORE Commit  
- Commit is controlled by the Control Kernel  
- Fast path may skip some deep deliberation steps but cannot skip core governance  
- learning does NOT happen inside this loop  
  
Use side annotations to indicate which plane owns each step:  
World Interface, Runtime Boundary, Control Kernel, Deliberation, Execution, Verification, State and Knowledge  
  
At the far side, add a visually separated secondary loop labeled "Consolidation Run", fed by Archive but detached from the main runtime loop.  
Elegant, disciplined, executive-grade, easy to understand in one glance

### 这张图的核心卖点

主循环 10 步、顺序不可更改，而且文档特别强调：  
**Boundary Check 先于 Plan Formation**，**Verification 先于 Commit**，**学习不进入主循环**

## 4）双环图：实时主链 vs 后台演进回路

这张图很适合体现“优雅感”，因为它把最容易混掉的两件事拆清楚了。

### 提示词

elegant dual-loop architecture infographic, premium AI system poster, white background, clean thin lines, strong conceptual clarity, minimalistic enterprise style  
  
Illustrate a dual-loop architecture for a situational awareness AI-native system.  
  
Left side: Real-time Task Run loop  
- event-driven  
- latency-sensitive  
- auditable  
- operator-interruptible  
- governed by boundary and commit authority  
  
Right side: Background Consolidation Run loop  
- periodic scheduler trigger  
- reads archived task runs, evidence, traces, memory  
- distills memory  
- proposes new skills and workflows  
- suggests policy optimization  
- cannot write committed state  
- cannot directly promote production capability  
- requires operator review before promotion  
  
Use a strong visual separator between the two loops.  
Connect them only through Archive / Trace Pool and Proposal Queue.  
The central message should be:  
"Learning improves the system, but never contaminates the real-time control path."  
  
This must feel like a refined systems-thinking diagram, not a casual workflow chart.

### 为什么这张会很精彩

文档对 Consolidation Run 的定义非常清楚：  
它是**独立后台 Run**，只读归档/Trace/Memory，只能写 Memory 和能力提议队列，**禁止写 Committed State**，而且产出必须经过 **Operator Review** 才能升级

## 5）状态真相链图：从观测到已提交真相

这张图特别适合帮助非技术用户理解“为什么这系统可信”。

### 提示词

truth-state pipeline diagram, elegant AI reasoning architecture infographic, clean vector poster, dark background with restrained glow, highly readable labels, minimalistic but sophisticated  
  
Create a visual diagram for the single truth chain of a situational awareness AI system.  
Represent the progression of system truth in five distinct layers:  
  
1. Observed Facts  
2. Evidence Graph  
3. Claims / Hypotheses / Projections  
4. Working State  
5. Committed State  
  
Show that:  
- facts are different from judgments  
- judgments are different from committed truth  
- working drafts are not the same as final state  
- projections are future-oriented claims based on current evidence  
- only the Control Kernel can promote results into Committed State through Commit Gate  
  
Visually use a progressive pipeline or layered stack.  
Add side links from Verification, Review Record, and Operator oversight.  
The final Committed State should look stable, hardened, and governance-approved.  
Do not make this a database schema.  
Make it conceptual, rigorous, and visually powerful.

### 这张图对应文档重点

第 2 章把 Runtime State 定义成五层状态链：  
**Observed Facts → Evidence Graph → Claims / Hypotheses / Projections → Working State → Committed State**，并明确只有统一提交链才能写入最终真相

## 6）控制内核拆解图：展示唯一治理权威但不做成巨型上帝类

这张图对介绍架构“优雅而不臃肿”很有帮助。

### 提示词

control kernel decomposition diagram, premium technical architecture infographic, clean white background, strong central module with five disciplined submodules, enterprise platform design style  
  
Draw a decomposition diagram for the Control Kernel of a situational awareness AI-native system.  
The Control Kernel must appear as the single governance authority, but internally decomposed into five disciplined responsibility groups:  
  
1. Ingress and Run Admission  
2. Lifecycle and Scheduling  
3. Policy and Review Coordination  
4. Commit and Audit Authority  
5. Resilience and Degradation Control  
  
Visually emphasize:  
- one governance authority  
- but not one giant god-class  
- no business reasoning inside the kernel  
- no direct domain intelligence inside the kernel  
- governance, lifecycle, arbitration, and commit are the core  
  
Connect the kernel outward to:  
Runtime Boundary, Deliberation, Execution, State and Knowledge, Operator Surface  
  
The diagram should feel precise, restrained, and architecturally mature.  
Avoid flashy AI-brain imagery.  
Avoid multiple competing control centers.

### 这张图能替你讲清一个关键点

文档说得非常明确：  
**Control Kernel 是唯一治理权威，但不等于一个巨型总控类**；首版就应该拆成五个子职责组，而且不能让 Deliberation 直接写 Committed State

## 7）四个正交维度图：帮助用户理解“不是多 lane 拼盘”

这张图适合解释系统为什么没有被场景撕裂。

### 提示词

conceptual matrix infographic, enterprise architecture explanation poster, clean vector style, white background, elegant information design  
  
Visualize the four orthogonal runtime dimensions of a situational awareness AI-native control system.  
Use a 2 x 2 or four-axis conceptual composition to show that runtime behavior is described by combinations of dimensions, not by adding more agents.  
  
The four dimensions are:  
- Trigger dimension: user, event, scheduled, callback, strategy, memory  
- Cognitive depth dimension: Fast Run, Deliberative Run, Mission Continuation  
- Governance dimension: Observe, Semi-Auto, Approval-Gated, Human-Controlled  
- Execution carrier dimension: Direct Action, Workflow, Skill, Worker  
  
The image should communicate:  
- these are orthogonal dimensions  
- they combine per run  
- new requirements should map into combinations of existing dimensions  
- the architecture should not grow a new layer for every scenario  
  
This should look like a clear conceptual architecture teaching graphic, not a mathematical chart.

### 这张图的价值

第 2 章明确要求：  
运行模式必须由 **四个正交维度** 组合表达，不能因为新需求再长出“第五维度”或新的业务主脑

## 8）门控治理图：五个 Gate 如何保护系统

如果你想强调“可控、可审计、可接管”，这张很适合。

### 提示词

governance gate architecture diagram, premium risk-control system poster, dark navy enterprise style, clean vector blocks, authoritative and precise visual language  
  
Create a governance diagram centered on five runtime gates in a situational awareness AI-native system:  
  
- Admission Gate  
- Execution Gate  
- Asset Control Gate  
- Commit Gate  
- Override Gate  
  
Show where each gate intercepts the system lifecycle:  
- admission before run creation  
- execution before action step  
- asset control before lease and dispatch  
- commit before promoted truth  
- override during operator takeover, stop, rollback, resume  
  
Emphasize deterministic enforcement surfaces, policy control, approval suspension, risk constraints, and auditable review records.  
The image should convey that critical actions must hit explicit gates.  
The system should feel safe, disciplined, and governable.  
Avoid artistic abstraction; prefer clean enforcement architecture.

### 这张图的文档依据

第 2 章对 Runtime Boundary 的要求很硬：  
**所有关键动作必须命中明确 gate**，至少包括 Admission / Execution / Asset / Commit / Override 五个关卡

## 一个很实用的投喂技巧

生图模型画架构图时，最容易翻车的是“字多、字小、概念太密”。  
所以你可以分两种版本去出图：

### 版本 A：演示型

只保留大标签：

- Control Kernel
- Runtime Boundary
- Deliberation
- Execution
- State & Knowledge
- Operator Surface
- Consolidation Run

### 版本 B：半注释型

在大标签下面再保留 2 到 4 个关键词。

这样比一开始就塞满所有术语更容易出漂亮图。
