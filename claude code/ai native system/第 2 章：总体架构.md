## Living Mission Agent OS：面向态势任务的 AI-native 智能操作系统

---

## 0. 文档定位

本文档定义态势感知 AI Agent 系统的总体架构。

本章的目标不是把系统拆成尽可能多的模块，而是回答一个更本质的问题：

> **如何构建一个真正 AI-native、响应敏捷、可持续进化、可扩展能力生态、并能长期运行在真实态势任务中的智能系统？**

本文档只定义系统的架构主干、核心抽象和运行机制，不展开字段级 schema、接口签名、算法细节和具体 UI 交互。

后续章节应在本文档定义的骨架内展开：运行时对象、能力系统、任务工作空间、态势模型、经验引擎、执行载体、评估体系和工程实现。

---

## 1. 架构转向：从受控运行时到智能操作系统

### 1.1 本系统不是什么

本系统不是一个态势问答聊天机器人。

它不只是接收用户一句话、调用模型、返回一段回答。

本系统也不是多个业务 Agent 的拼盘。

它不应该因为存在“雷达研判”“周界巡检”“目标跟踪”“风险预警”等场景，就拆出多个彼此独立的常驻 Agent。

本系统也不是一个传统流程编排平台。

它不应该把智能任务固化成一组僵硬流程图，也不应该让每个任务都经过同样重的推理链路。

本系统更不是一个静态态势看板。

它不能只是展示告警、地图和资产状态，而必须能持续理解态势、发现证据缺口、分配注意力、调用能力、提出判断、执行行动、沉淀经验。

### 1.2 本系统是什么

本系统是一个 **Living Mission Agent OS**。

它是一个面向态势任务的智能操作系统：

- 通过多通道任务网关接入真实世界；
    
- 通过活态任务工作空间维持长期上下文；
    
- 通过注意力循环在事件流中分配认知资源；
    
- 通过 Agent Harness 完成理解、计划、行动、验证与重试；
    
- 通过能力系统按需检索和装配技能、工具、资产适配器、工作单元和剧本；
    
- 通过态势世界模型维护可查询、可投影、可推演的连续理解；
    
- 通过经验引擎把每次运行转化为可复用的记忆、技能、剧本、策略建议和评估资产。
    

### 1.3 一句话定义

> **本系统是一个面向态势任务的 Living Mission Agent OS：它通过 Mission Gateway 接入操作员、传感器、地图、资产和外部系统；通过 Living Mission Workspace 维持任务上下文；通过 Attention Loop 在事件流中分配认知资源；通过 Agent Harness 调用可检索的能力生态完成理解、计划与行动；通过 Situation Model 维护可推演的态势世界模型；并通过 Experience Engine 将每次运行转化为可复用的记忆、技能、剧本和评估资产。**

## 2. 设计哲学

### 2.1 工作空间优先，而不是会话优先

传统聊天式 Agent 以对话会话为中心。

本系统以 **Living Mission Workspace** 为中心。

一次对话、一次告警、一次工具调用、一次资产回调，都只是任务工作空间中的事件。真正长期存在的是任务目标、态势模型、证据包、运行历史、操作员意图、可用能力和经验记忆。

因此，本系统的核心不是“回答用户问题”，而是“维护并推进一个长期任务空间”。

### 2.2 注意力优先，而不是流程优先

态势感知系统面对的是连续事件流。

如果每个事件都进入完整推理流程，系统会变慢、变重，也会表现得不够聪明。

真正智能的系统必须先判断：

- 什么值得关注？
    
- 什么只是背景噪声？
    
- 什么只需要更新状态？
    
- 什么需要快速反应？
    
- 什么需要深度审议？
    
- 什么需要立刻提醒操作员？
    

因此，本系统将 **Attention Loop** 作为核心架构机制，而不是把所有事件都塞进一个固定主循环。

### 2.3 能力检索优先，而不是工具罗列优先

能力越多，Agent 不一定越聪明。

如果所有技能、工具、资产、剧本、历史经验都平铺给模型，系统会变得迟钝、混乱、不可控。

真正 AI-native 的能力系统应该支持按任务、区域、资产、风险、证据缺口和历史经验动态检索能力。

Agent 不直接面对全部能力，而是面对经过检索、过滤、排序后的候选能力集。

### 2.4 态势模型优先，而不是数据库状态优先

系统状态不是一堆表。

态势智能需要一个可查询、可投影、可推演的 **Situation Model**。

它不仅记录“发生了什么”，还要表达：

- 当前有哪些对象；
    
- 对象之间有什么关系；
    
- 哪些区域存在约束；
    
- 哪些证据支持当前判断；
    
- 哪些判断仍然不确定；
    
- 哪些未来状态值得预警；
    
- 还缺哪些关键证据。
    

### 2.5 经验复利优先，而不是简单归档优先

Trace 的价值不只是审计。

每一次运行、失败、审批、纠正、误报、漏报、成功处置，都是系统能力增长的原材料。

本系统通过 **Experience Engine** 将运行经验转化为可复用资产：任务记忆、目标画像、区域经验、技能候选、剧本候选、策略建议、评估样例和回归测试集。

智能不是一次性设计出来的，而是通过持续使用产生复利。

## 5. Mission Gateway：任务网关

### 5.1 定义

Mission Gateway 是系统接入真实世界的统一入口。

它不是普通 API Gateway，而是 Agent 与操作员、传感器、地图、资产、外部系统和多种任务表面的连接层。

### 5.2 设计目标

Mission Gateway 的目标是让 Agent 出现在真实工作流中，而不是被关在一个聊天框里。

操作员可以从多种入口与系统交互：

- 态势大屏；
    
- 地图组件；
    
- 告警卡片；
    
- 审批卡片；
    
- 移动端；
    
- 语音通道；
    
- 任务看板；
    
- 外部业务系统。
    

### 5.3 输入类型

Mission Gateway 接收：

- 用户请求；
    
- 雷达告警；
    
- 视频异常；
    
- GIS 事件；
    
- 数据库变更；
    
- 定时触发；
    
- 外部系统 webhook；
    
- 资产执行回执；
    
- 操作员审批结果；
    
- 移动端或语音输入；
    

### 5.4 输出类型

Mission Gateway 输出：

- UI 更新；
    
- 告警通知；
    
- 审批请求；
    
- 操作员提示；
    
- 外部 API 调用；
    
- 资产控制指令；
    
- 报表和摘要；
    
- 任务状态广播。
    

### 5.5 核心原则

Mission Gateway 遵循三个原则：

1. **Channel-native**  
    Agent 应进入操作员已有工作渠道，而不是强迫操作员进入新入口。
    
2. **Event-native**  
    所有输入都标准化为事件，进入任务工作空间。
    
3. **Surface-aware**  
    同一判断在不同任务表面有不同呈现方式：地图需要空间标注，告警卡需要优先级，审批卡需要风险与后果。

## 6. Living Mission Workspace：活态任务工作空间

### 6.1 定义

Living Mission Workspace 是 Agent 的认知工作台。

它不是简单上下文缓存，也不是一次 Run 的状态对象，而是围绕长期任务持续存在的活态工作空间。

### 6.2 Workspace 承载什么

Living Mission Workspace 维护：

- 当前任务目标；
    
- 当前态势快照；
    
- 活跃假设；
    
- 证据包；
    
- 证据缺口；
    
- 关注区域；
    
- 关注目标；
    
- 可用资产；
    
- 可用能力；
    
- 活跃 Run；
    
- 任务历史；
    
- 操作员最近意图；
    
- 当前注意力焦点；
    
- 推荐剧本；
    
- 推荐技能；
    
- 待审批动作；
    
- 待验证判断。
    

### 6.3 Workspace 与 Run 的关系

Run 是一次执行。

Workspace 是长期任务空间。

```text
Mission Workspace
  ├── Run A：一次告警研判
  ├── Run B：一次资产调度建议
  ├── Run C：一次操作员查询
  ├── Run D：一次深度审议
  └── Trace Timeline
```

系统的中心不是 Run，而是 Workspace。

Run 可以结束，Workspace 持续存在。

### 6.4 Workspace 与 Context 的关系

Workspace 不是直接塞给模型的 prompt。

模型可见上下文来自 Workspace 的投影。

```text
Living Mission Workspace
        │
        ▼
Context Projection
        │
        ▼
Model Context
```

每种任务、路径和能力调用都有不同的上下文投影规则。

### 6.5 Workspace 的智能价值

Living Mission Workspace 让系统具备连续性。

它让 Agent 不是在处理孤立事件，而是在持续回答：

- 当前任务正在朝什么目标推进？
    
- 新事件对当前态势有什么影响？
    
- 当前最重要的证据缺口是什么？
    
- 哪些假设需要验证？
    
- 哪些资产可以用于补证？
    
- 操作员最近更关心什么？
    
- 历史上类似情况如何处理？
    

这正是系统产生“智能感”的基础。

## 7. Attention Loop：注意力循环

### 7.1 定义

Attention Loop 是系统在事件流中分配认知资源的机制。

它决定一个事件是被忽略、合并、仅更新状态、触发快速响应、进入技能研判、升级深度审议，还是立即提醒操作员。

### 7.2 为什么需要 Attention Loop

态势感知系统面对的是高频、多源、异构事件流。

如果每个事件都进入完整 Agent 推理流程，会导致：

- 响应慢；
    
- 成本高；
    
- 上下文膨胀；
    
- 操作员被噪声淹没；
    
- 系统无法表现出重点感。
    

Attention Loop 的目标是让系统像一个经验丰富的值班员：知道什么该立刻看，什么可以合并，什么需要补证，什么值得升级。

### 7.3 注意力输入

Attention Loop 读取：

- 新事件；
    
- 当前态势模型；
    
- Workspace 当前焦点；
    
- 活跃假设；
    
- 证据缺口；
    
- 任务优先级；
    
- 区域风险；
    
- 资产状态；
    
- 历史相似案例；
    
- 操作员最近意图；
    
- 当前系统负载。
    

### 7.4 注意力输出

Attention Loop 输出路径决策：

```text
ignore              忽略
merge               合并到已有事件
workspace_update    只更新任务工作空间
hot_response        进入 Hot Path
warm_reasoning      进入 Warm Path
deep_reasoning      进入 Deep Path
operator_nudge      提醒操作员
```

### 7.5 注意力循环流程

```text
Event
  → Normalize
  → Compare with Situation Model
  → Compute Workspace Delta
  → Score Importance / Novelty / Risk / Urgency
  → Select Path
  → Update Workspace
  → Emit Trace
```

### 7.6 注意力评分维度

Attention Score 至少考虑：

|维度|含义|
|---|---|
|Novelty|是否出现新对象、新模式、新区域变化|
|Risk|是否涉及高风险区域、资产或动作|
|Urgency|是否需要立即处理|
|Uncertainty|是否显著增加或降低不确定性|
|Mission Relevance|是否与当前任务目标相关|
|Evidence Value|是否能填补关键证据缺口|
|Operator Interest|是否匹配操作员当前关注点|
|Historical Pattern|是否命中历史高价值模式|

### 7.7 Attention Loop 的架构地位

Attention Loop 不等于规则引擎。

它可以由确定性规则、轻量模型、向量检索、异常检测和历史模式共同组成。

V 1 可以使用简单规则和轻量评分实现，但架构上必须承认：注意力分配是智能系统的核心能力。

## 8. 多事件并发处理与 Workspace 路由

### 8.1 问题定义

真实系统不会按“一个事件处理完再处理下一个事件”的方式运行。

用户输入、告警事件、资产回调、定时触发、历史研判任务回调、外部系统更新，可能在同一时间窗口内同时到达。

因此，系统必须先回答三个问题：

1. 这些事件是否属于同一个任务上下文？
    
2. 它们应该进入同一个 Workspace，还是多个 Workspace？
    
3. 它们应该触发一个 Run、多个 Run，还是只更新态势模型？
    

本系统采用 **Event Stream → Correlation → Workspace Routing → Attention Scheduling → Run Dispatch** 的处理方式。

### 8.2 总体流程

```text
Concurrent Events
  → Event Envelope
  → Event Correlation
  → Workspace Routing
  → Workspace Delta Batch
  → Attention Scheduling
  → Run Dispatch / Workspace Update
  → Situation Model Update
```

多个事件不会直接并发打进模型上下文，也不会无脑创建多个独立 Agent。

它们首先被标准化、关联、分流、合并，然后再决定是否触发 Hot / Warm / Deep 路径。

### 8.3 Event Envelope：事件信封

所有输入进入系统后，先被包装为标准 Event Envelope。

Event Envelope 至少包含：

```text
event_id
source
source_type
occurred_at
received_at
spatial_scope
target_refs
asset_refs
mission_refs
workspace_refs
parent_run_id
correlation_keys
payload
priority_hint
```

其中 `correlation_keys` 是并发事件处理的核心。

它可以来自：

- 空间范围；
    
- 目标 ID；
    
- 资产 ID；
    
- Mission ID；
    
- Workspace ID；
    
- 用户当前关注对象；
    
- 父 Run ID；
    
- 回调任务 ID；
    
- 时间窗口；
    
- 事件类型；
    
- 证据链引用。
    

### 8.4 Workspace Routing：一个 Workspace 还是多个 Workspace

系统不以事件来源决定 Workspace，而以任务语义决定 Workspace。

#### 进入同一个 Workspace 的情况

多个事件满足以下条件之一时，进入同一个 Living Mission Workspace：

- 指向同一个 Mission；
    
- 指向同一个目标或目标群；
    
- 指向同一空间区域和同一时间窗口；
    
- 是同一个 Run 的回调或后续证据；
    
- 用户输入正在询问当前活跃态势；
    
- 告警事件能补充当前 Workspace 的证据缺口；
    
- 资产回调是当前任务计划的一部分。
    

例如：

```text
用户问：“刚才 3 号区域怎么回事？”
同时雷达告警进入 3 号区域。
同时前一个轨迹分析 Run 返回结果。

→ 三个事件应进入同一个 Workspace。
→ 系统合并为一次 Workspace Delta。
→ Attention Loop 决定是否更新当前研判或升级为深度分析。
```

#### 进入多个 Workspace 的情况

多个事件满足以下条件时，进入不同 Workspace：

- 任务目标不同；
    
- 空间区域无明显关联；
    
- 关注对象不同；
    
- Mission 生命周期不同；
    
- 事件之间不存在有效证据关系；
    
- 一个事件属于操作员查询，另一个事件属于独立自动监测任务。
    

例如：

```text
用户查询 A 区域历史态势。
同时 B 区域出现实时告警。
同时无人机巡逻任务返回 C 区域结果。

→ 三个事件可能进入三个 Workspace。
→ 系统分别更新各自 Workspace。
→ Attention Scheduler 根据优先级调度处理。
```

#### 跨 Workspace 关联的情况

部分事件可能同时影响多个 Workspace。

例如一个目标从 A 区域移动到 B 区域，可能同时影响 A 区域监控任务、B 区域预警任务和目标跟踪任务。

此时系统不复制完整推理链，而是采用引用机制：

```text
Shared Event / Shared Evidence
  → linked to Workspace A
  → linked to Workspace B
  → linked to Workspace C
```

事件和证据可以被多个 Workspace 引用，但每个 Workspace 只维护与自身任务相关的投影。

### 8.5 Workspace Delta Batch：先合并变化，再触发推理

并发事件进入同一 Workspace 时，不应立即逐条触发模型推理。

系统先形成 Workspace Delta Batch：

```text
Workspace Delta Batch
  - new_events
  - changed_objects
  - new_evidence
  - returned_results
  - operator_intents
  - asset_updates
  - affected_hypotheses
  - affected_predictions
  - missing_evidence_updates
```

Attention Loop 面对的是 Delta Batch，而不是孤立事件。

这样可以避免：

- 同一时间窗口内重复推理；
    
- 多个 Run 对同一个态势做出冲突判断；
    
- 用户输入和系统告警互相覆盖；
    
- 回调结果晚到后无法合并到当前任务。
    

### 8.6 Attention Scheduling：注意力调度

当多个 Workspace 同时产生 Delta Batch 时，系统通过 Attention Scheduler 统一调度。

调度依据包括：

- 风险等级；
    
- 时间紧迫度；
    
- Mission 优先级；
    
- 操作员当前关注；
    
- 是否存在待审批动作；
    
- 是否存在关键证据回调；
    
- 是否影响已确认态势；
    
- 系统当前负载；
    
- 是否已有活跃 Run 正在处理相同问题。
    

Attention Scheduler 输出：

```text
update_only     只更新 Workspace / Situation Model
merge_into_run  合并到已有 Run
start_hot_path  启动 Hot Path
start_warm_path 启动 Warm Path
start_deep_path 启动 Deep Path
defer           延迟处理
interrupt       打断或重排当前处理
```

### 8.7 Run Dispatch：何时复用 Run，何时创建新 Run

Run 是处理动作，不是事件容器。

同一个 Workspace 内可以有多个 Run，但并不是每个事件都创建 Run。

#### 复用已有 Run

当新事件属于当前 Run 的上下文补充时，应合并进入已有 Run：

- 工具调用回调；
    
- Worker 返回结果；
    
- 资产执行回执；
    
- 与当前计划相关的新证据；
    
- 操作员对当前问题的补充输入。
    

#### 创建新 Run

当事件代表新的任务意图或新的问题焦点时，创建新 Run：

- 用户发起新的问题；
    
- 新区域出现独立告警；
    
- 当前 Workspace 需要并行补证；
    
- 当前 Run 被阻塞但新事件需要快速处理；
    
- 注意力调度认为需要独立 Deep Path。
    

#### 只更新 Workspace

当事件只是状态变化或低价值补充时，不创建 Run：

- 普通资产心跳；
    
- 重复告警；
    
- 低置信度噪声；
    
- 与当前任务无关的背景变化；
    
- 已有证据的轻微更新。
    

### 8.8 并发控制：Workspace 内串行化，Workspace 间并行化

系统采用以下并发原则：

```text
Workspace 内：关键状态更新串行化
Workspace 间：可并行处理
共享对象：通过引用、版本和冲突检测协调
```

这意味着：

- 同一个 Workspace 的 Situation Model 更新应按版本顺序提交；
    
- 同一个假设的 Assessment 更新不能被多个 Run 同时覆盖；
    
- 不同 Workspace 的 Hot / Warm Path 可以并行；
    
- Deep Path 可以异步运行，并在结果返回时生成新的回调事件；
    
- 共享证据不复制，只引用；
    
- 共享资产需要通过租约或调度机制协调。
    

### 8.9 为什么这更支持智能

多事件并发处理不是单纯工程问题，而是智能系统的核心能力。

一个真正有智能感的系统应该能做到：

- 用户刚提出问题时，自动合并同区域新告警；
    
- 深度分析还没完成时，先给出 Hot Path 的即时态势更新；
    
- Worker 返回结果后，自动补进当前假设，而不是另开一个孤立结论；
    
- 多个告警短时间集中出现时，识别为一个态势变化，而不是生成多条碎片化回答；
    
- 不同任务并发推进时，保持各自 Workspace 清晰，不互相污染。
    

因此，系统不是“一个事件一个 Agent 反应”，而是“多个事件共同更新一个或多个 Living Workspace，再由 Attention Loop 决定认知资源如何分配”。

## 9. 三条运行路径：Hot / Warm / Deep

### 8.1 设计目标

系统响应速度不能只依赖模型速度。

真正的响应能力来自路径分层：简单事件走轻路径，复杂问题走深路径，高风险高价值任务才进入重推理。

### 8.2 Hot Path：即时路径

Hot Path 面向毫秒到秒级响应。

适用场景：

- 告警展示；
    
- 态势快照更新；
    
- 已知规则命中；
    
- UI 状态刷新；
    
- 低复杂度通知；
    
- 事件合并；
    
- 操作员正在查看区域的轻量提示。
    

典型流程：

```text
Event → Attention Loop → Workspace Delta → Situation Update → UI / Alert
```

Hot Path 不做开放式推理。

### 8.3 Warm Path：快速研判路径

Warm Path 面向秒级到十几秒响应。

适用场景：

- 简单证据合并；
    
- 风险初判；
    
- 调用轻量技能；
    
- 查询历史相似案例；
    
- 生成短摘要；
    
- 给操作员一个初步判断。
    

典型流程：

```text
Event
  → Attention Loop
  → Capability Retrieval
  → Skill / Lightweight Model
  → Assessment Candidate
  → Workspace Update
```

Warm Path 以技能和检索为主，不默认进入复杂规划。

### 8.4 Deep Path：深度审议路径

Deep Path 面向复杂态势、关键任务和高价值判断。

适用场景：

- 多源证据冲突；
    
- 高风险动作前分析；
    
- 多资产协同；
    
- 长周期任务重规划；
    
- 需要 Worker 并行调查；
    
- 需要生成结构化行动计划；
    
- 需要预测未来态势演化。
    

典型流程：

```text
Event / Mission Trigger
  → Attention Loop
  → Living Workspace Load
  → Capability Retrieval
  → Context Projection
  → Plan
  → Worker / Skill / Tool
  → Evidence / Assessment
  → Verification
  → Situation Model Update
  → Experience Capture
```

### 8.5 路径升级与降级

路径可以升级：

```text
Hot → Warm → Deep
```

例如，一个普通告警先更新 UI；如果短时间内同区域多次出现，则升级为 Warm Path；如果证据冲突或风险升高，则进入 Deep Path。

路径也可以提前终止。

如果 Warm Path 已经给出足够可靠的低风险判断，不必进入 Deep Path。

## 10. Agent Harness：智能运行底座

### 9.1 定义

Agent Harness 是系统执行智能任务的运行底座。

它不是一个万能大脑，也不是业务场景 Agent，而是支持上下文、计划、执行、验证、重试、沙箱和追踪的通用智能运行基础设施。

### 9.2 核心组件

Agent Harness 包含：

```text
Context Builder
Planner
Capability Retriever
Tool / Skill Runner
Worker Manager
Verifier
Retry / Recovery Manager
Sandbox
Trace Emitter
```

### 9.3 Context Builder

Context Builder 从 Living Workspace、Situation Model、Memory 和 Capability System 中生成任务所需的上下文投影。

它必须遵守：

- 不全量注入 Workspace；
    
- 不全量注入长期记忆；
    
- 不把无关资产、无关规则、无关历史塞进上下文；
    
- 根据 Hot / Warm / Deep 路径选择上下文预算；
    
- 根据任务目标选择证据、假设和能力说明。
    

### 9.4 Planner

Planner 负责生成结构化计划。

计划可以是：

- 调查计划；
    
- 补证计划；
    
- 风险分析计划；
    
- 资产使用建议；
    
- 多 Worker 分析计划；
    
- 操作员交互计划；
    
- 报告生成计划。
    

Planner 面对的是检索后的候选能力，而不是全部系统能力。

### 9.5 Worker Manager

Worker Manager 管理隔离子任务。

Worker 适合：

- 并行分析不同区域；
    
- 对不同假设分别补证；
    
- 分析历史相似案例；
    
- 生成目标画像；
    
- 对复杂数据做离线计算；
    
- 让一个子任务在受限上下文中完成专项分析。
    

Worker 之间不共享完整上下文，只通过引用和结果交接。

### 9.6 Verifier

Verifier 负责判断结果是否可靠。

它不仅用于安全审查，也用于提升智能质量：

- 判断证据是否足够；
    
- 判断是否存在反证；
    
- 判断是否遗漏关键变量；
    
- 判断计划是否可执行；
    
- 判断预测是否有依据；
    
- 判断报告是否与证据一致。
    

### 9.7 Sandbox

Sandbox 为实验性分析、代码执行、仿真、候选技能测试和非生产能力评估提供隔离环境。

Sandbox 的意义不只是安全，也是让系统可以大胆探索而不污染真实态势。

## 11. Capability System：能力系统

### 10.1 定义

Capability System 是系统的能力生态。

它让系统可以通过新增、组合和检索能力来扩展，而不是通过增加新的常驻 Agent 来扩张。

### 10.2 能力类型

系统支持八类能力：

|类型|作用|
|---|---|
|Mission Profile|定义任务角色、风格、偏好和领域边界|
|Skill|可复用专业能力|
|Tool|原子操作能力|
|Asset Adapter|真实资产或外部系统适配器|
|Worker|隔离子任务能力|
|Hook|生命周期扩展点|
|Playbook|稳定任务剧本|
|Plugin|能力打包与分发单元|

### 10.3 Mission Profile

Mission Profile 定义 Agent 在某类任务中的角色。

例如：

```text
周界异常研判助手
  - 优先补证
  - 避免过早升级风险等级
  - 输出证据链
  - 对资产动作只给建议

无人机巡逻副驾
  - 优先任务连续性
  - 关注航线安全
  - 提醒电量和禁飞区
  - 推荐下一步巡逻动作
```

Mission Profile 让 Agent 的行为在长期任务中保持一致。

### 10.4 Skill

Skill 是输入输出清晰、可复用、可审计的专业能力。

示例：

- 目标轨迹分析；
    
- 多源告警聚合；
    
- 风险等级初判；
    
- 历史相似案例检索；
    
- 巡逻路线建议；
    
- 态势摘要生成；
    
- 报告生成。
    

### 10.5 Tool

Tool 是原子操作能力。

示例：

- 查询轨迹；
    
- 查询地图图层；
    
- 查询资产状态；
    
- 发送通知；
    
- 创建任务卡；
    
- 读取历史记录。
    

### 10.6 Asset Adapter

Asset Adapter 连接真实资产或外部系统。

示例：

- 无人机任务系统；
    
- 相机云台；
    
- 雷达系统；
    
- GIS 平台；
    
- 工单系统；
    
- 通知系统。
    

### 10.7 Hook

Hook 是生命周期扩展点。

系统可以在关键事件上触发 Hook：

- EventReceived；
    
- WorkspaceUpdated；
    
- AttentionRaised；
    
- RunStarted；
    
- ContextProjected；
    
- PlanProposed；
    
- BeforeAction；
    
- AfterAction；
    
- EvidenceProduced；
    
- AssessmentUpdated；
    
- VerificationCompleted；
    
- CommitAccepted；
    
- RunCompleted；
    
- ExperienceCaptured。
    

Hook 让能力生态可以在不修改核心代码的情况下扩展系统行为。

### 10.8 Playbook

Playbook 是稳定任务剧本。

当某类任务被反复执行、路径稳定、效果可评估时，可以沉淀为 Playbook。

示例：

- 多次周界告警处置剧本；
    
- 夜间低空目标研判剧本；
    
- 巡逻任务异常中断恢复剧本；
    
- 目标进入敏感区域预警剧本。
    

### 10.9 Plugin

Plugin 是能力分发单元。

一个 Plugin 可以包含：

- Mission Profile；
    
- Skills；
    
- Tools；
    
- Asset Adapters；
    
- Hooks；
    
- Playbooks；
    
- Evaluation Cases；
    
- Documentation。
    

Plugin 让系统具备生态化扩展能力。

## 12. Capability Retrieval：能力检索与装配

### 11.1 为什么需要能力检索

能力系统一旦扩展，能力数量会快速增长。

如果 Planner 面对全部能力，系统会出现：

- 选择困难；
    
- 上下文膨胀；
    
- 工具误用；
    
- 响应变慢；
    
- 任务计划不稳定。
    

因此，能力必须先被检索和排序，再进入 Agent Harness。

### 11.2 检索输入

Capability Retrieval 读取：

- 任务目标；
    
- 当前态势；
    
- 事件类型；
    
- 区域；
    
- 资产类型；
    
- 证据缺口；
    
- 风险等级；
    
- 历史相似案例；
    
- 操作员偏好；
    
- 当前路径类型：Hot / Warm / Deep。
    

### 11.3 检索输出

输出为候选能力集：

```text
Candidate Capabilities
  - mission_profile
  - playbooks[]
  - skills[]
  - tools[]
  - asset_adapters[]
  - workers[]
  - hooks[]
  - verification_strategies[]
```

### 11.4 能力选择原则

能力选择遵守：

- 优先选择任务相关能力；
    
- 优先选择历史效果好的能力；
    
- 优先选择上下文成本低的能力；
    
- 优先选择可解释能力；
    
- 对复杂任务允许组合多个能力；
    
- 对快速路径限制能力数量；
    
- 对深度路径允许 Worker 和 Playbook 参与。
    

### 11.5 能力检索的智能价值

Capability Retrieval 是系统变聪明的关键。

一个成熟系统不应该靠 prompt 告诉模型“你有哪些工具”，而应该能够根据当前任务主动选出最合适的能力组合。

## 13. Situation Model：态势世界模型

### 12.1 定义

Situation Model 是系统对当前世界的结构化理解。

它不是简单数据库状态，也不是模型上下文，而是可以被查询、投影、推演和验证的世界模型。

### 12.2 Situation Model 包含什么

Situation Model 包含：

- 地图图层；
    
- 区域边界；
    
- 关键对象；
    
- 对象轨迹；
    
- 对象关系；
    
- 资产状态；
    
- 任务状态；
    
- 观测事实；
    
- 证据关系；
    
- 判断与假设；
    
- 风险评估；
    
- 未来预测；
    
- 不确定性；
    
- 证据缺口；
    
- 已确认态势。
    

### 12.3 Situation Memory 与 Situation Model

Situation Memory 是存储层。

Situation Model 是运行时联合视图。

```text
Situation Memory
  - observations
  - evidence
  - assessments
  - predictions
  - committed states
  - historical episodes

Situation Model
  - current world view
  - active hypotheses
  - uncertainty
  - missing evidence
  - projected future
```

### 12.4 四类核心视图

Situation Model 至少包含四类视图：

```text
Observed → Evidence → Assessment / Prediction → Committed
```

#### Observed

记录原始观测和标准化事实。

#### Evidence

记录证据项、来源、置信度、支持关系和反驳关系。

#### Assessment / Prediction

记录当前判断、候选假设、风险评估和未来预测。

#### Committed

记录已经被系统确认并可被外部依赖的态势状态。

### 12.5 不确定性与证据缺口

真正智能的系统不只输出结论，还要知道自己不知道什么。

Situation Model 必须显式表达：

- 哪些判断置信度低；
    
- 哪些证据互相冲突；
    
- 哪些关键证据缺失；
    
- 哪些预测需要继续观测；
    
- 哪些区域或对象需要补充感知。
    

### 12.6 Situation Model 的智能价值

Situation Model 让系统从“事件处理器”升级为“态势理解者”。

它支持：

- 快速态势查询；
    
- 复杂上下文投影；
    
- 证据链解释；
    
- 风险预测；
    
- 任务重规划；
    
- 能力检索；
    
- 经验沉淀；
    
- 操作员可视化。

## 14. Experience Engine：经验引擎

### 13.1 定义

Experience Engine 是系统的智能复利引擎。

它把 Trace、运行结果、失败案例、操作员修正、任务后果和评估结果转化为可复用的记忆、技能、剧本、策略建议和评估资产。

### 13.2 输入

Experience Engine 读取：

- Trace；
    
- Run 结果；
    
- Mission 结果；
    
- 操作员审批和接管记录；
    
- 验证结果；
    
- 成功处置案例；
    
- 失败案例；
    
- 误报和漏报；
    
- 技能调用效果；
    
- 历史相似任务；
    
- 任务后果反馈。
    

### 13.3 输出

Experience Engine 产出：

- Episode：情景记忆；
    
- Pattern：事件模式、风险模式、误报模式；
    
- Target Profile：目标画像；
    
- Region Knowledge：区域经验；
    
- Skill Candidate：候选技能；
    
- Playbook Candidate：候选剧本；
    
- Policy Suggestion：策略建议；
    
- Evaluation Case：评估样例；
    
- Regression Set：回归测试集；
    
- Capability Routing Signal：能力路由信号。
    

### 13.4 三类智能复利

#### 13.4.1 认知复利

系统越来越理解：

- 某区域常见异常；
    
- 某目标类型常见行为；
    
- 某类告警常见误报原因；
    
- 某些证据对判断最关键；
    
- 某些场景需要提前关注。
    

#### 13.4.2 能力复利

系统越来越会把重复成功经验沉淀为能力：

- 重复分析步骤 → Skill；
    
- 稳定处置路径 → Playbook；
    
- 高价值专项分析 → Worker Template；
    
- 常见判断逻辑 → Assessment Pattern；
    
- 常见补证路径 → Investigation Pattern。
    

#### 13.4.3 交互复利

系统越来越懂操作员和组织：

- 哪些操作员偏好先看证据；
    
- 哪些班组偏好保守判断；
    
- 哪类动作通常需要人工确认；
    
- 哪些摘要形式更容易被采纳；
    
- 哪些提醒经常被忽略。
    

### 13.5 经验转化流程

```text
Trace
  → Episode Extraction
  → Pattern Mining
  → Candidate Capability
  → Offline Evaluation
  → Simulation / Shadow Run
  → Human Review
  → Promotion
  → Capability Registry / Memory / Evaluation Set
```

### 13.6 Experience Engine 与实时路径的关系

Experience Engine 不阻塞实时任务。

它在后台运行，从归档数据中生成候选资产。

实时路径只消费已经被验证、评估、复核并发布的能力资产。

## 15. Mission Event Lifecycle：任务事件生命周期

### 14.1 为什么不用固定主循环

固定主循环容易让系统变成流程引擎。

态势任务更适合事件化生命周期：每个关键动作都发布事件，Attention Loop、Capability System、Trace、Experience Engine 和 Mission Surfaces 都可以订阅这些事件。

### 14.2 核心生命周期事件

系统至少定义以下生命周期事件：

```text
EventReceived
EventNormalized
WorkspaceLoaded
WorkspaceUpdated
AttentionScored
PathSelected
RunStarted
ContextProjected
CapabilitiesRetrieved
PlanProposed
ActionRequested
ActionExecuted
EvidenceProduced
AssessmentUpdated
VerificationCompleted
CommitProposed
CommitAccepted
OperatorInterrupted
RunSuspended
RunResumed
RunCompleted
ExperienceCaptured
```

### 14.3 生命周期事件的价值

事件化生命周期带来四个价值：

1. **更快**  
    轻量事件可以只触发轻量处理，不必进入完整推理链。
    
2. **更可扩展**  
    Hook、Skill、Playbook、Experience Engine 都可以订阅生命周期事件。
    
3. **更可观测**  
    Trace 可以自然形成时间线。
    
4. **更智能**  
    Attention Loop 和 Experience Engine 可以从事件流中学习模式。

## 16. Mission Surfaces：任务表面

### 15.1 定义

Mission Surfaces 是 Agent 与用户交互的所有表面。

系统不应把 Agent 限制在聊天框中。

### 15.2 表面类型

Mission Surfaces 包括：

- 地图态势层；
    
- 告警卡片；
    
- 审批卡片；
    
- 任务时间线；
    
- 证据面板；
    
- 风险解释面板；
    
- 移动端提醒；
    
- 语音播报；
    
- 报告视图；
    
- 复盘视图；
    
- Live Canvas。
    

### 15.3 表面适配原则

同一态势判断需要根据表面适配表达：

|表面|表达重点|
|---|---|
|地图|空间位置、范围、轨迹、风险区域|
|告警卡|优先级、风险、建议动作|
|审批卡|后果、证据、可选操作|
|任务时间线|事件顺序、决策链、责任主体|
|复盘视图|证据链、判断演化、经验提取|

Mission Surfaces 让 Agent 的智能嵌入真实工作流，而不是停留在对话中。

## 17. Control Spine：控制脊柱

### 16.1 定位

Control Spine 是横向贯穿机制，不是架构主叙事中心。

它确保系统在拥有强智能和高扩展性的同时，仍然保持可控、可审计、可接管、可恢复。

### 16.2 控制能力

Control Spine 包含：

- 权限；
    
- 资产租约；
    
- 审批；
    
- 人工接管；
    
- 验证；
    
- 提交；
    
- 审计；
    
- 回滚；
    
- 幂等；
    
- 预算与背压。
    

### 16.3 三个关键原则

1. Agent 可以生成候选判断，但不能直接把判断变成已确认态势。
    
2. Agent 可以调用能力，但关键副作用动作必须经过相应控制点。
    
3. Agent 可以学习经验，但新能力进入生产必须经过评估和发布流程。
    

Control Spine 不应该压过 AI-native 主线，但必须作为系统长期可靠运行的骨架约束存在。

## 18. V 1 收缩策略

### 17.1 V 1 目标

V 1 不追求完整 Mission Agent OS。

V 1 的目标是证明以下最小闭环成立：

> 系统可以从多源事件进入，更新 Living Mission Workspace，经 Attention Loop 判断路径，调用合适 Skill 进行快速研判，更新 Situation Model，生成 Trace，并把运行经验沉淀为后续可复用资产。

### 17.2 V 1 必须实现

|能力|V 1 范围|
|---|---|
|Mission Gateway|接用户请求、模拟告警、资产回调|
|Living Workspace|任务目标、当前态势、活跃 Run、证据包、可用能力|
|Attention Loop|简单重要性评分、路径选择|
|Hot Path|事件入库、态势更新、UI 推送|
|Warm Path|调用 2-3 个核心 Skill 完成快速研判|
|Deep Path|最小深度审议，不做复杂 Worker 并发|
|Capability System|Skill / Tool / Asset Adapter 最小集合|
|Capability Retrieval|基于任务类型和事件类型选择能力|
|Situation Model|Observed / Evidence / Assessment / Committed 窄版|
|Trace Core|生命周期事件、能力调用、判断更新|
|Experience Engine|Episode extraction 最小版|
|Mission Surfaces|地图、告警卡、审批卡、任务时间线最小版|

### 17.3 V 1 不做

|能力|后置原因|
|---|---|
|完整 Plugin 生态|需要稳定能力 manifest|
|自动技能生成|依赖足够 Trace 和评估样例|
|复杂 Worker 并发|需要成熟 Workspace 与上下文隔离|
|完整 Playbook 引擎|需要沉淀稳定任务路径|
|复杂 Experience Engine|需要历史数据积累|
|多通道全覆盖|V 1 只接核心操作台和告警入口|
|复杂预测模型|V 1 先做简单风险评估和证据缺口|

### 17.4 V 1 成功标准

V 1 成功不以“模型多聪明”为唯一标准。

V 1 应证明：

1. 系统能维护一个活态任务工作空间；
    
2. 系统能区分哪些事件需要注意；
    
3. 系统能走 Hot / Warm / Deep 不同路径；
    
4. 系统能按需选择能力，而不是暴露全部工具；
    
5. 系统能形成结构化态势模型；
    
6. 系统能从运行中生成可复用经验；
    
7. 操作员能在地图、告警和时间线中感知 Agent 的智能行为。

## 19. 架构复杂度控制原则

### 18.1 不按业务场景创建常驻 Agent

新增业务场景优先通过 Mission Profile、Skill、Playbook 和 Capability Retrieval 扩展。

### 18.2 不让所有事件走深度推理

Attention Loop 必须先判断路径。

Hot Path 和 Warm Path 是系统反应速度的基础。

### 18.3 不让能力系统变成工具垃圾场

新增能力必须有 manifest、适用场景、输入输出、评估方式和调用边界。

### 18.4 不让 Workspace 变成上下文大杂烩

Workspace 是结构化任务空间。

模型上下文必须由 Context Projection 生成。

### 18.5 不让 Situation Model 退化为状态表

Situation Model 必须表达对象、关系、证据、判断、预测、不确定性和证据缺口。

### 18.6 不让 Experience Engine 变成归档任务

Experience Engine 必须产出可复用资产，而不只是保存历史。

### 18.7 不让 Control Spine 抢占主叙事

治理能力必须存在，但核心文档的主线应是智能形成过程：感知、注意、理解、行动、沉淀、进化。

## 20. 架构优雅性测试

本架构是否足够优雅，可以用以下问题检验：

1. **系统是否有一个清晰中心？**  
    有。中心是 Living Mission Workspace，而不是会话或单次 Run。
    
2. **系统如何避免变慢？**  
    通过 Attention Loop 和 Hot / Warm / Deep 三路径分流。
    
3. **系统如何扩展能力？**  
    通过 Capability System 和 Capability Retrieval，而不是新增常驻 Agent。
    
4. **系统如何形成真正理解？**  
    通过 Situation Model 表达对象、关系、证据、判断、预测和不确定性。
    
5. **系统如何越用越强？**  
    通过 Experience Engine 将 Trace 转化为记忆、技能、剧本、评估资产和能力路由信号。
    
6. **系统如何进入真实工作流？**  
    通过 Mission Gateway 和 Mission Surfaces 接入地图、告警、移动端和外部系统。
    
7. **系统如何保持简洁？**  
    通过七个架构原语组织系统，不为每个业务场景发明新主脑。
    

如果这些问题需要额外口头解释，说明文档仍然不够清晰。

## 21. 架构结论

本系统的最终目标不是构建一个更复杂的 Agent 平台，而是构建一个真正 AI-native 的态势任务智能操作系统。

它的主线不是“输入 → 权限 → 执行 → 提交”，而是：

```text
感知世界
  → 维持任务工作空间
  → 分配注意力
  → 调用能力理解与行动
  → 更新态势世界模型
  → 从经验中持续进化
```

最终架构可以概括为：

```text
七个架构原语：
Mission Gateway
Living Mission Workspace
Attention Loop
Agent Harness
Capability System
Situation Model
Experience Engine

三条运行路径：
Hot Path
Warm Path
Deep Path

一个横向脊柱：
Control Spine

一个核心扩展机制：
Capability Retrieval

一个智能复利机制：
Trace → Episode → Pattern → Capability
```

这套架构保留了原设计中必要的可控性和可追溯性，但主叙事不再是治理，而是智能。

它让系统具备五个关键特征：

1. **反应快**：通过 Attention Loop 和路径分层避免所有事件走重推理。
    
2. **上下文稳**：通过 Living Workspace 维护长期任务上下文。
    
3. **能力强**：通过 Capability System 和 Capability Retrieval 动态装配能力。
    
4. **理解深**：通过 Situation Model 建立连续态势世界模型。
    
5. **会进化**：通过 Experience Engine 将运行经验转化为智能复利。
    

因此，本架构推荐作为态势感知 AI Agent 系统的新总体架构主干。
